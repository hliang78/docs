## 1. 总结论

### 1.1 DoWhy 不是当前主工具

DoWhy 适合回答“某类变更是否会稳定影响某类结果”。

它适合做：

- 某类发布是否提高错误率
- 某类策略是否增加延迟
- 某类配置是否更容易诱发某类告警

它不适合直接做：

- 单次事故的唯一根因定位
- A1 单源探测下的拓扑断点定位
- 从当前告警直接反推责任点

当前主线里，**DoWhy 只放在后续历史验证位置，不放在实时根因主路径里**。

### 1.2 告警不是根因，只是入口

告警本质上是：

- 症状
- 时间锚点
- 对象锚点

因此告警只能告诉我们“哪里出现了异常”，不能直接回答“是谁导致了异常”。

### 1.3 当前根因定位主线

当前主线应固定为：

1. 先定义独立于 OneOps 的根因定位机制输入输出
2. 再确定 OneOps 如何把拓扑、告警、日志映射到这份输入
3. 用拓扑和探测结果生成候选故障边/点
4. 用日志证据收缩和校正候选
5. 输出候选根因及其证据

如果未来接入变更和操作记录，再把它们接到这条主线后半段做关联；在此之前，不把它们放进当前主判定。

## 2. 告警、拓扑、日志三者分工

### 2.1 告警

告警负责：

- 给出异常开始时间
- 给出异常对象集合
- 触发排障流程

告警不负责直接给出根因。

### 2.2 拓扑与探测

拓扑与探测负责：

- 定义异常传播范围
- 给出哪些边或点可以解释当前不可达现象
- 区分更像根因的对象与连带症状

这是当前主判定层。

### 2.3 日志

日志负责：

- 校对时间顺序
- 补充节点自身异常证据
- 区分“自身故障”与“上游影响”
- 在多个拓扑候选之间做收缩

日志是当前主线里的**证据层**，不是独立根因引擎。

## 3. 拓扑场景：A1 探测、多节点连通性同时告警

### 3.1 问题定义

- 平台位于 **A1**
- A1 对图中多个节点做连通性探测
- 故障时，多个节点可能同时出现连通性告警
- 目标是识别更接近故障发源位置的边或点

### 3.2 当前代码现状

当前仓库里，拓扑能力还没有完全沉到 `app/topology`。

已确认的事实是：

- `phy_node_map` 中已经包含拓扑生成相关逻辑
- `app/topology` 当前更多是在已有拓扑结果之上做组装、补全和输出

因此，当前阶段不能把“拓扑已经成熟可直接消费”作为前提。**拓扑能力建设本身要和 RCA 一起纳入主线推进。**

### 3.3 主判定前提

在做定位前，必须先明确：

- 探测是单向还是双向
- 探测结果是节点级成功/失败，还是链路级失败
- 是否保留成功样本
- 当前拓扑是否可信

如果这些前提不明确，后续结果只能算候选解释，不能算严格定位。

### 3.4 LCA 的位置

若拓扑可视为一棵树，并同时满足：

- 单故障
- 全量探测
- 成功和失败样本都可用

则可以使用树上的祖先关系帮助缩小候选范围。

但结论要收紧：

- **LCA 只能作为辅助，不直接等价于根因**

更稳妥的主判定是：

1. 收集失败节点集合
2. 收集成功节点集合
3. 在树上找“上游仍可达、下游整体失败”的边界
4. 将该边界对应的边或点作为候选根因

### 3.5 什么时候不能直接用 LCA

以下情况不应把 LCA 当作主判定：

- 图不是树
- 存在多路径或冗余
- 只有失败样本，没有成功样本
- 部分节点未被探测
- 图上的语义不只是“通/断”

这时主方法应退回到：**基于拓扑和观测一致性的候选生成与过滤**，而不是继续套 LCA。

这里还要补一条现实约束：

- **真实网络里，多路径通常是常态，不是例外**

因此当前主线不能把“树形拓扑、唯一路径”当默认前提。

### 3.6 当前结论

在 A1 单源连通性场景下：

- 主工具是拓扑 + 探测结果
- `LCA` 只在树形、单故障、观测完整时作为辅助
- 输出应是候选根因，而不是先验假设唯一答案
- 多路径场景下，宁可不输出桥接候选，也不伪造唯一桥接根因

同时，当前还要补一条工程结论：

- 在 RCA 之前，必须先把“谁负责生成可消费拓扑”这件事收敛清楚
- 当前更合理的方向是：`phy_node_map` 作为拓扑原始来源，`app/topology` 作为拓扑能力主承载
- 不建议在主线未闭合前，再新建一个独立顶级模块承接这部分工作

### 3.7 当前职责切分

#### `phy_node_map`

应负责：

- 从设备、接口、线缆、IPAM 等数据生成原始 L2 关系
- 提供原始邻接与连接素材

不应继续长期负责：

- 直接产出最终可消费拓扑
- 定义最终节点/边语义
- 承载 RCA 判定逻辑

当前代码中的 `ToTopology()` 可作为过渡能力保留，但不应被视为长期正式边界。

#### `app/topology`

应负责：

- 作为唯一的可消费拓扑出口
- 接收并归一化 `phy_node_map` 的原始结果
- 统一输出节点、边、状态、快照、补全结果

因此，长期方向应是由 `app/topology` 接管统一拓扑输出语义，而不是继续由 `phy_node_map` 直接决定最终拓扑结构。

不应负责：

- 直接承载完整 RCA
- 把告警、日志、变更分析揉进拓扑域本身

#### RCA 层

应负责：

- 先消费平台无关的统一图输入
- 结合告警、探测、日志生成候选根因
- 输出候选、证据和前提

不应负责：

- 自己重新定义拓扑模型
- 直接绑定 OneOps 具体表结构或服务实现

### 3.8 当前收口点

当前最需要先收口的一点是：

- `phy_node_map` 中的 `ToTopology()` 只能视为过渡能力
- `app/topology` 才应成为长期稳定的拓扑输出边界

如果这条边界不先收敛，后续 RCA 将缺少稳定输入，主线无法闭合。

## 4. 当前主线输入与输出

### 4.1 最小输入

当前主线最少需要：

- 图快照
- 异常对象集合
- 观测时间
- 可选日志证据

若当前没有变更或操作记录，不影响先闭合这条主线。

当这套机制接入 OneOps 时，再额外回答：

- 拓扑由谁生成
- 拓扑以什么结构输出
- 拓扑快照是否可稳定复用

### 4.2 主判定路径

当前建议固定为：

1. 先形成可消费的拓扑结果
2. 聚合同一时间窗内的连通性告警
3. 绑定对应时间窗的拓扑快照和探测结果
4. 生成可解释当前现象的候选边/点
5. 用日志证据校正和收缩候选
6. 输出结果

### 4.3 最小输出

当前输出最少包含：

- 候选根因对象
- 每个候选解释了哪些异常
- 每个候选依赖了哪些证据
- 当前结论依赖的关键前提

若当前未接入变更或操作记录，输出里只需要明确说明“当前未做变更/操作关联”，不展开更多设计。

## 5. 主线任务拆分

### 5.1 拓扑主线

拓扑主线当前只需要闭合三件事：

- 谁负责生成统一拓扑
- 统一拓扑如何映射到平台无关图输入
- 拓扑快照是否可稳定复用

当前方向已经明确：

- `phy_node_map` 负责原始关系来源
- `app/topology` 负责统一拓扑输出
- `ToTopology()` 只作为过渡能力存在
- `app/topology` 通过 `IPhyNodeMap` 消费原始 L2 关系，不绑定 `phy_node_map` 的具体实现类
- `Topology()` 主路径的必需输入只有原始 L2 关系与坐标读取；快照补全和手工边合并只作为附加补全层
- OneOps 统一拓扑输出只作为平台适配层输入，不等于根因定位机制本体

在这三件事未闭合前，不进入更细的拓扑实现设计。

### 5.2 RCA 主线

RCA 主线当前只需要闭合四件事：

- 机制本体以什么平台无关契约作为输入
- OneOps 如何把告警映射成异常对象和时间窗
- OneOps 如何把日志映射成证据输入
- 最终结果最少输出什么

当前方向已经明确：

- 根因定位机制本体优先独立于 OneOps
- OneOps 当前实现侧暂以 `app/topology` 统一输出作为图输入来源
- 告警是入口，不是根因
- 日志是证据层，不是独立引擎
- DoWhy 不进入当前实时主路径

当前最小代码边界也应保持一致：

- 根因定位机制应有一份独立于 OneOps 的最小输入输出
- OneOps 侧只负责把平台数据映射到这份最小输入输出
- 当前 `platform` 中的 RCA 服务只应视为 OneOps 适配入口
- 适配入口不直接依赖 `phy_node_map`
- 平台无关契约以 [RCA_PLATFORM_AGNOSTIC_MINIMAL_CONTRACT.md](/home/jacky/project/OneOPS-ALL/docs/RCA_PLATFORM_AGNOSTIC_MINIMAL_CONTRACT.md) 为准

当前对统一拓扑输出的最小消费契约也已经明确：

- RCA 只长期依赖 `node.device_code`
- RCA 只长期依赖 `edge.a_device_code`、`edge.b_device_code`、`edge.a_interface_code`、`edge.b_interface_code`
- `coordinates` 不进入 RCA 主判定
- `device_name`、`device_type`、`status`、`bandwidth` 当前不作为 RCA 主路径必需字段
- 若统一拓扑输出不满足这份最小契约，RCA 直接报错，不继续生成候选结果

当前代码主线已推进到：

- 已有一条可运行的 OneOps 适配主路径
- 已提供 `platform/root-cause/analyze-alerts` API 入口
- 已提供 `alert/alarm/root-cause/analyze` 告警侧桥接入口
- 告警侧桥接入口可从告警本身补齐租户与观测时间
- 若统一拓扑契约不满足，入口层直接透传错误，不再返回泛化失败消息
- 当前候选生成逻辑已下沉到平台无关的 `OneOps/pkg/rca`
- 当前候选生成逻辑只依赖最小图契约，不依赖 OneOps 扩展字段
- 基于统一拓扑输出节点候选
- 基于统一拓扑输出直接故障边候选
- 针对任意两台失败设备，若其间存在唯一最短路径，则补充桥接边/点候选
- 针对任意两台失败设备，若其间存在多条最短路径且存在公共桥接边/点，则输出这部分确定性候选
- OneOps 适配层当前只保留日志证据挂载和结果字段映射
- 两台失败设备之间若存在多条最短路径且没有公共桥接边/点，当次不强行判定桥接候选

在这四件事未闭合前，不进入更细的算法或接口设计。

### 5.3 当前阶段优先级

当前阶段的第一优先级不是 RCA 实现，而是**拓扑主线收敛**。

原因只有一条：

- 如果 OneOps 侧图输入边界未收敛，机制映射层就没有稳定输入，主线无法闭合

因此当前阶段顺序应固定为：

1. 先定义平台无关的 RCA 最小输入输出
2. 再收敛 `phy_node_map -> app/topology` 的职责边界
3. 再确认 `app/topology` 的统一输出可稳定映射到 RCA 输入
4. 然后再收敛 OneOps 适配入口

在这四个阶段完成前，不继续展开更细的 RCA 技术设计。

### 5.4 当前接口的最小边界调整建议

围绕当前接口：

- `NewL2NodeMapFromTenantCode(code, ctx) (*L2NodeMap, error)`
- `L2NodeMap.ToTopology()`

当前建议是：

#### `NewL2NodeMapFromTenantCode`

暂时保留。

原因：

- 它表达的是“按租户生成原始 L2 关系”的能力
- 这仍然属于 `phy_node_map` 的合理职责

因此，当前不需要先改掉这个入口。

#### `ToTopology()`

不建议继续作为长期正式边界。

原因：

- 它让 `phy_node_map` 直接决定最终可消费拓扑结构
- 这会削弱 `app/topology` 作为统一拓扑出口的职责

因此，当前方向应是：

1. `phy_node_map` 继续产出 `L2NodeMap`
2. `app/topology` 负责把 `L2NodeMap` 转成统一拓扑结果
3. `ToTopology()` 只作为过渡能力保留，不再作为长期主边界继续扩展

当前阶段不要求立刻删除 `ToTopology()`，但应停止继续把新的拓扑语义堆到这个方法上。

### 5.5 最小改造顺序

当前建议的最小改造顺序只分三步：

1. 保持 `NewL2NodeMapFromTenantCode(...)` 不动，继续作为 `phy_node_map` 的原始关系入口
2. 停止继续扩展 `L2NodeMap.ToTopology()` 的职责，把新增拓扑语义收敛到 `app/topology`
3. 由 `app/topology` 逐步接管统一拓扑输出，随后 RCA 只消费这里的结果

当前阶段先做到这三步即可，不继续展开更细的结构、接口或算法改造。

## 6. 未来接入变更/操作事件的位置

- 在拓扑候选已经生成之后，作为附加证据参与关联

因此当前结论保持不变：

- **主线先用拓扑 + 告警 + 探测 + 日志闭合**
- **DoWhy 和变更分析放在主线闭合之后**

### 7.5 已核对一致的场景理解

当前后续动画与文档，必须以这组已核对一致的理解为准，不再沿用“同质节点 + 最短路径 + 直接看边状态”的简化口径。

当前统一理解只有这些：

1. 当前问题不是纯图论问题，而是“监控源点出发的分层定位问题”
2. 拓扑中的节点至少包含三类：网络设备、服务器、应用
3. 拓扑中的边至少包含三类关系：网络连接关系、部署承载关系、应用依赖关系
4. 应用与应用之间也允许存在依赖关系
5. 监控是从某个监控器部署点触发，不是从整张图无源观测
6. 边本身不能被直接监控
7. 边状态只能通过两类信息推断：
   - 路径上的节点接口状态
   - 该边是否命中监控器到目标对象的实际监控路径
8. 这里的“节点 up/down”不是单一网络语义，而是节点类型相关语义：
   - 网络设备看网络侧可达与接口状态
   - 服务器看主机侧可达与主机状态
   - 应用看应用侧存活与服务状态
9. 路径不是默认最短路径
10. 在复杂网络中，路径应指实际报文转发路径，需要通过二层、三层转发表采集与推导获得
11. 根因定位顺序应固定为分层排除，不应直接跳到“当前应用自身故障”
12. 当前应用异常时，判定顺序固定为：
   - 先看监控源到承载位置的实际转发路径是否阻断
   - 再看承载服务器是否异常
   - 再看其上游依赖应用是否异常
   - 上述都排除后，才收敛到当前应用自身异常

当前还应固定四个判定例子：

1. 网络路径阻断
   - 监控源到服务器 S 的实际转发路径命中边 `E`
   - 路径前段节点均 up
   - 边 `E` 关联接口异常
   - 应用 A 未被成功观测
   - 此时应先收敛到“实际路径上的边 `E` 阻断”
   - 不继续把问题收敛到服务器 S 或应用 A 自身

2. 承载服务器异常
   - 监控源到服务器 S 的实际转发路径完整且可达
   - 路径边均可推断为 up
   - 服务器 S down
   - 应用 A 部署在服务器 S 上
   - 此时应收敛到“服务器 S 异常”
   - 不直接跳到“应用 A 自身 down”

3. 上游应用依赖异常
   - 监控源到服务器 S 的实际转发路径完整且可达
   - 服务器 S up
   - 应用 A 部署在服务器 S 上
   - 应用 A 依赖应用 B
   - 应用 B down
   - 此时应收敛到“上游依赖应用 B 异常”
   - 应用 A 只标记为受影响对象

4. 目标应用自身异常
   - 监控源到服务器 S 的实际转发路径完整且可达
   - 服务器 S up
   - 应用 A 所有上游依赖应用均 up
   - 应用 A 仍 down
   - 只有在此前三层都排除后，才允许收敛到“应用 A 自身异常”

当前还应固定两条判定约束：

- 若应用 A down，且它只依赖服务器 S，服务器 S up，则可以继续收敛到“应用 A 自身 down”
- 若应用 A down，且它除了依赖服务器 S，还依赖应用 B，则在服务器 S up 的前提下，仍需先检查应用 B；只有当应用 B 也被排除后，才能把根因收敛到应用 A 自身

当前 demo 与上述例子的对应关系固定为：

- `network_path_blocked`：对应“网络路径阻断”
- `hosting_server_fault`：对应“承载服务器异常”
- `upstream_application_dependency`：对应“上游应用依赖异常”
- `target_application_self_fault`：对应“目标应用自身异常”

当前未来需求只保留一条，不提前展开设计：

- 后续可以扩展为分布式监控，即从多个监控源点同时发起观测，以降低多路径环境下单点视角的不确定性

但在当前阶段，这只作为后续边界保留，不进入当前主线实现。

当前展示层可以额外保留一个“后续方向演示场景”：

- `distributed_monitors_multi_path`

但必须同时满足以下限制：

- 该场景只用于帮助理解“多监控源为何有价值”
- 该场景不代表当前 RCA 主线已经支持多源联合判定
- 该场景不进入当前主线验收闭合条件
- 该场景不反推当前机制边界扩展

### 7.10 根因定位能力的下一步升级方向

当前若回到根因定位机制本体，下一步不应继续扩展“失败节点 + 最短路径桥接”这一条旧主线。

原因只有一条：

- 当前 `pkg/rca` 的最小契约仍然是：
  - 无类型节点
  - 无类型边
  - 无监控源
  - 无目标对象
  - 无实际转发路径输入
  - 无节点观测状态输入
- 这套契约可以继续支撑“图上失败节点与桥接候选”分析
- 但已经不足以表达当前已核对一致的真实场景

因此，下一步升级方向应固定为：

- 从“失败节点图分析”升级到“单监控源、单目标、实际路径驱动、分层排除”的新契约

这里强调“单监控源、单目标”，是为了继续满足当前主线约束：

- 不泛化
- 不衍生
- 不一次扩太多能力面
- 先把一条主线闭合，再谈多监控源、多目标、多时间窗

#### 7.10.1 当前能力缺口

当前机制与目标场景之间的缺口只保留五个：

1. 缺少监控源
   - 现有输入只知道哪些节点失败
   - 但真实场景首先需要知道“从哪里发起观测”

2. 缺少目标对象
   - 现有输入没有“当前到底在定位哪个应用/服务器/对象”
   - 但分层排除必须围绕单个目标收敛

3. 缺少节点与边的语义类型
   - 当前节点没有区分网络设备、服务器、应用
   - 当前边没有区分转发、承载、依赖
   - 因此现有机制无法知道下一层应该排除谁

4. 缺少实际路径与观测事实
   - 当前图里只有结构
   - 没有“实际命中的路径”
   - 没有“哪些节点 up/down”
   - 没有“哪些边只是根据接口与路径被推断”

5. 缺少分层收敛规则
   - 现有机制只会产出候选
   - 不具备“先路径、再服务器、再依赖、最后目标自身”的固定收敛顺序

#### 7.10.2 下一步只推进这一层

下一步只应推进下面这一层能力，不再更多：

- 在 `pkg/rca` 内新增一条“分层观测分析”主线

它的边界固定为：

- 只支持单监控源
- 只支持单目标对象
- 只支持单次观测时间
- 只支持当前四层收敛顺序：
  - 路径层
  - 服务器层
  - 应用依赖层
  - 目标应用自身
- 只输出确定性结果
- 不做排序
- 不做置信度
- 不做历史时间窗合并
- 不做多监控源联合判定

这里不建议直接替换现有 `AnalyzeConnectivity`。

更稳妥的做法应固定为：

- 保留现有 `AnalyzeConnectivity` 作为旧基线
- 在 `pkg/rca` 内新增一个并行入口，例如：
  - `AnalyzeLayeredObservation`
  - 或 `AnalyzeReachabilityRCA`

这样做的理由只有三个：

1. 不破坏当前已通过的夹具基线
2. 不迫使 OneOps 适配层一次性迁移
3. 可以先把新主线独立验收，再决定是否替换旧入口

#### 7.10.3 新契约建议只保留最小字段

下一步的新契约不应做成大而全模型，只建议保留这些最小字段：

1. `monitor_id`
   - 当前监控源

2. `target_id`
   - 当前目标对象

3. `nodes`
   - `node_id`
   - `node_kind`
   - 只允许：
     - `network`
     - `server`
     - `application`

4. `edges`
   - `a_node_id`
   - `b_node_id`
   - `relation_kind`
   - 只允许：
     - `forwarding`
     - `hosting`
     - `dependency`

5. `observations`
   - 当前节点观测结果
   - 只保留：
     - `node_id`
     - `status`
   - `status` 只允许：
     - `up`
     - `down`

6. `active_path_edges`
   - 当前监控源到目标对象实际命中的路径边

7. `blocked_path_edge_keys`
   - 当前实际路径中已明确判定阻断的路径边

8. `observed_at`
   - 单次观测时间

9. `log_evidence_node_ids`
   - 当前时间窗内有日志证据的节点

当前不建议在第一步引入以下字段：

- 权重
- 分数
- 概率
- 多时间片
- 多监控源
- 变更事件
- 历史统计特征

#### 7.10.4 新分析器的固定判定顺序

新分析器的规则应固定，不做自由推理：

1. 先判定路径层
   - 若 `active_path_edges` 上存在明确阻断边，直接收敛到路径边
   - 若路径上的节点接口与观测足以推断某条路径边阻断，也直接收敛到该边

2. 再判定服务器层
   - 若路径已排除，且目标应用的承载服务器 `down`，收敛到服务器

3. 再判定应用依赖层
   - 若路径与服务器已排除，且存在上游依赖应用 `down`，收敛到该依赖应用

4. 最后判定目标应用自身
   - 只有前三层都排除后，且目标应用 `down`，才收敛到目标应用自身

5. 若输入不足
   - 明确报错或明确返回“当前无法收敛”
   - 不伪造根因

#### 7.10.5 OneOps 侧下一步只做适配，不改机制边界

若上述新分析器启动实现，OneOps 侧的职责仍应固定为适配：

1. 从拓扑中映射：
   - 节点类型
   - 边关系类型

2. 从监控数据中映射：
   - 监控源
   - 目标对象
   - 节点观测状态
   - 实际命中的路径边

3. 从日志中映射：
   - `log_evidence_node_ids`

OneOps 不应承担以下职责：

- 在平台层写分层收敛规则
- 在平台层拼凑排序逻辑
- 在平台层补智能推理

规则仍应在 `pkg/rca` 内闭合。

#### 7.10.8 第一轮升级只做前两层

虽然目标口径已经固定为四层收敛，但第一轮实现不应一次做完四层。

第一轮只建议覆盖下面两类判定：

1. 网络路径阻断
2. 承载服务器异常

不在第一轮进入：

1. 上游应用依赖异常
2. 目标应用自身异常
3. 多监控源联合判定

这样切分的原因只有三个：

1. 这是最短闭环
   - 只要引入监控源、目标对象、节点/边类型、节点观测状态、实际路径边
   - 就已经足够把“路径阻断”和“服务器异常”两类场景讲闭合

2. 这是最少语义扩展
   - 还不需要引入应用到应用依赖图的完整处理
   - 还不需要讨论“依赖链继续下钻到哪里为止”

3. 这是最容易做真伪分界
   - 路径层和服务器层的输入边界更清楚
   - 更容易快速发现真实数据映射是否成立

因此第一轮新契约即使落地，也只需要保证：

- 能表达 `forwarding`
- 能表达 `hosting`
- 能表达 `network` / `server` / `application`
- 能表达节点 `up/down`
- 能表达 `active_path_edges`

但第一轮分析器只真正消费下面这些规则：

1. 若实际路径存在明确阻断边，收敛到路径边
2. 若路径已排除，且承载服务器 `down`，收敛到服务器
3. 否则返回“当前阶段无法继续收敛”

这里的“无法继续收敛”是刻意保留，不是缺陷。

因为在第一轮里：

- 依赖层尚未实现

- `not_supported_yet`

而不是伪造“应用自身异常”。

#### 7.10.14 `layered_input` 真实生产方判断

当前如果继续推进，最关键的问题不是再改 RCA，而是判断 OneOps 里谁负责稳定产出 `layered_input`。

围绕第一轮最小闭环，当前只看这 5 个关键字段：

1. `monitor_node_id`
2. `target_node_id`
3. `active_path_edges`
4. `blocked_path_edges`
5. `node_observations`

当前代码分析后，应固定为下面这个结论。

##### 已有较明确宿主的字段

1. `monitor_node_id`
   - 当前最接近的宿主是 `maintenance.monitor_probe`
   - 其中 `monitor_probe.host_code` 已经表达“探针部署在哪台主机/服务器上”
   - 若该主机已进入统一拓扑节点集合，则可直接作为监控源点的第一落点

2. `target_node_id`
   - 对服务器目标，当前可直接使用设备编码
   - 对应用目标，当前最接近的宿主是 `application/v2`
   - `application_v2.code` 可作为应用节点标识

3. 应用到服务器的 `hosting` 关系
   - 当前最接近的宿主也是 `application/v2`
   - 现有代码里已经有：
     - `application_v2.device_code`
     - `runs_on` 关系同步
   - 这意味着第一轮所需的 `application -> server` 承载关系，在现有代码里已有现实锚点

##### 已有部分基础，但还不是最终生产方的字段

1. 监控目标归属与分发
   - `monitoring_v2_target_resolver_registry.go` 已能把 `application` / `device` 目标解析出来
   - 它还能反查：
     - 应用所属 `device_code`
     - 对应地址与端口
     - 对应 agent
   - 这说明它适合作为“目标归一化入口”的上游能力
   - 但它当前还不是 RCA 观测事实生产方

2. 运行态任务快照
   - `monitoring_v2_agent_task_snapshot_store.go`
   - `monitoring_v2_task_projection_snapshot_sync.go`
   - `monitoring_v2_wizard_runtime_snapshot_sync.go`
   - 这些能力已经能采集 agent 上实际运行任务、配置、插件、pipeline 签名
   - 它们适合作为“当前监控任务确实已下发且在运行”的运行态依据
   - 但它们当前仍不能直接产出“实际网络转发路径”或“节点 down 结论”

##### 当前仍然缺失稳定生产方的字段

1. `active_path_edges`
   - 当前代码里还没有稳定模块能够产出“监控源到目标对象的实际转发路径边集合”
   - 现有 `app/topology` 只给结构拓扑
   - 现有 `monitoring_v2` 只给任务与运行态快照
   - 两者之间还缺一层“路径求值/路径观测归一化”

2. `blocked_path_edges`
   - 当前代码里也没有稳定模块能够明确给出“哪条实际路径边已阻断”
   - 这要求同时结合：
     - 实际路径
     - 接口观测/链路观测
     - 判定规则
   - 当前这些能力还没有在 OneOps 中闭合成一个统一生产方

3. `node_observations`
   - 当前虽然平台上有大量监控任务与应用监控任务能力
   - 但还没有看到一个已统一沉淀的“RCA 节点 up/down 事实表述层”
   - 也就是说：
     - 监控任务存在
     - 不等于 RCA 所需节点事实已经统一归一化

##### 因此当前应收敛出的唯一主线结论

当前 OneOps 若进入下一步，正确顺序不应是“直接接 RCA”，而应是：

1. 先确定 `layered_input` 的生产方模块
2. 该模块先只负责第一轮 5 个关键字段
3. 其中优先闭合：
   - `monitor_node_id`
   - `target_node_id`
   - `hosting`
4. 然后再补：
   - `node_observations`
   - `active_path_edges`
   - `blocked_path_edges`

在这个顺序里，当前最值得明确的一条实现建议是：

- 不要把 `layered_input` 生产逻辑塞进 `pkg/rca`
- 也不要让 `monitoring_root_cause_service.go` 自己推导实际路径

###### 1. `node_observations`

最小契约只保留：

- `node_id`
- `status`

- `node_id` 必须是 RCA 当前节点集合中的节点标识
- `status` 第一轮只允许：
  - `up`
  - `down`

当前第一轮不允许继续扩成：

- `unknown`
- 分数
- 百分比
- 多时间片
- 多维度健康子项

当前只要求表达一件事：

- 在本次 `observed_at` 上，这个节点是否被归一化为 `up/down`

###### 2. `active_path_edges`

最小契约只保留：

- `a_node_id`
- `b_node_id`

- ECMP 权重
- 流量比例
- 多条候选路径合集
- 推导置信度

也就是说，第一轮里它只是：

###### 3. `blocked_path_edges`

- `a_node_id`
- `b_node_id`

- `active_path_edges` 表示“命中过”
- `blocked_path_edges` 表示“在命中的路径边里，当前已明确判定阻断”

- `blocked_path_edges` 必须是 `active_path_edges` 的子集

- 这条命中路径边已经可以直接收敛为阻断候选

1. 时间窗口聚合

3. 置信度模型
4. 历史回放
5. 第二轮应用依赖层语义

   - `pkg/rca` 只应消费平台无关输入
   - 不应知道 OneOps 的探针、任务、应用、投影、快照这些细节

3. 它不是告警入口职责
   - `alert/api` 只适合做告警入口归一化
   - 不适合承载观测事实拼装

4. 它不是探针资产管理职责
   - `maintenance` 适合管探针对象、作业配置、推送
   - 不适合承载跨域 RCA 观测归一化

因此当前最合理的落点应固定为：

1. 放在 `OneOps/app/platform/service`
2. 由 `monitoring_root_cause_service.go` 调用
3. 自己不承载 RCA 判定，只负责生产 `layered_input`

当前不建议的落点也应明确冻结：

1. 不放在 `pkg/rca`
2. 不放在 `app/alert`
3. 不放在 `app/maintenance`
4. 不放在 `app/application/v2`
5. 不新建 `oneops/app/rca` 这类新的顶级业务模块

当前若继续推进，第一轮代码组织建议也只收口到这一层：

1. 在 `app/platform/service` 增加一个新的最小服务边界
2. 例如承载“观测归一化”职责
3. `monitoring_root_cause_service.go` 只调用它拿到 `layered_input`
4. 然后继续复用当前已经存在的分层 RCA 并行入口

当前最小服务边界也可以先固定成一条接口，不再继续展开：

```go
type IMonitoringRootCauseObservationNormalizer interface {
    BuildLayeredInput(ctx context.Context, req platformDTO.MonitoringRootCauseAnalyzeReq) (*platformDTO.MonitoringRootCauseLayeredAnalyzeInput, error)
}
```

这里固定这条接口的目的只有三个：

1. 把“观测归一化”和“RCA 判定”切开
2. 让 `monitoring_root_cause_service.go` 不再继续膨胀
3. 为后续真实数据接入预留唯一主线入口

当前仍不进入：

1. 这个接口的实现类拆分
2. 它内部依赖哪些 reader
3. 它是否落缓存或持久化
4. 它如何调度路径推导

这里仍然不进入：

1. 接口字段细化
2. 数据库存储设计
3. 调度流程设计
4. 第二轮能力设计

#### 7.10.16 监控地图能力判断

当前 OneOps 里已经有：

- 资源拓扑能力
- 监控任务发布与运行态能力

但还缺一层专门表达“多层级监控体系”的稳定载体。

这层能力可以定义为：

- **监控地图**

当前应先把一个结论固定下来：

- **监控地图不是 topo**
- **监控地图也不是 monitoring_v2 task graph**
- **监控地图应作为 RCA 上游观测归一化的承载图**
- **监控源不是静态探针对象本身，而是实际执行 remote 监控任务的 agent**

##### 三种图的职责区别

1. topo
   - 表达资源之间如何连接、承载、依赖
   - 回答“设备与设备如何连起来”“应用部署在哪台服务器上”

2. monitoring_v2 task graph
   - 表达监控任务、插件、processor、output 的运行态组织关系
   - 回答“哪个 agent 上有什么任务、任务由哪些插件组成”

3. 监控地图
   - 表达“从哪个监控源点出发，经由哪些监控任务，对哪些目标形成了什么覆盖，并沉淀了哪些观测事实”
   - 回答“这套监控体系实际上覆盖了谁、覆盖关系是什么、当前 RCA 能消费哪些观测证据”

因此，监控地图的主目标不应定义成“再画一张更漂亮的拓扑图”，而应固定为：

- **把多层级监控体系稳定表达出来**
- **给 RCA 观测归一化层提供稳定载体**

##### 当前建议的最小层级

当前若开始构思监控地图，只建议固定 5 层，先不继续泛化：

1. 监控源层
   - 表达实际执行 remote 监控任务的 agent
   - `monitor_probe` 与任务下发配置只表达“任务下发到哪里”
   - 真正的监控源应收敛为“哪个 agent 发起了这次观测”

2. 监控任务层
   - 表达某个监控源上实际运行的任务投影与 runtime pipeline
   - 当前最接近来源是 `platform/monitoring_v2` 的 target resolver、projection snapshot、runtime snapshot、task graph

3. 监控目标层
   - 表达被覆盖的设备、服务器、应用
   - 当前最接近来源是 `device`、`application/v2`

4. 承载与路径关联层
   - 表达物理拓扑、服务器挂接关系、目标对象承载关系、实际转发路径事实之间的关联
   - 当前代码入口位于 `analysis_l2/service/impl/analysis_l2.go` 的 `AnalysisProcessCommon`
   - `ProcessAnalysisServer.Main` 会把 `L2Task` 分发到这个统一入口
   - 物理拓扑部分当前对应 `enum.L2nodeMapServer -> l2Struct.L2nodeMapServer(...)`
   - 服务器与物理拓扑的挂接部分当前对应：
     - `enum.ArpTableServer -> l2Struct.ArpTableServerV2(...)`
     - `enum.ArpTableServer2 -> l2Struct.ArpTableServer2(...)`
   - 应用到服务器的承载关系当前还未自动闭合，第一阶段先允许手工关联
   - 应用到应用的依赖关系当前也先允许手工关联
   - 其中“实际路径”当前还没有稳定生产方，这一层先只收口为占位语义

5. 观测证据层
   - 表达 RCA 真正要消费的观测事实
   - 当前只先收口：
     - `node_observations`
     - `active_path_edges`
     - `blocked_path_edges`
     - `log_evidence_node_ids`

这 5 层当前只服务一个目标：

- 让 `IMonitoringRootCauseObservationNormalizer` 有一份稳定的上游事实载体

##### 监控地图与 RCA 的关系

当前主线里，监控地图不直接做 RCA 判定。

它当前只负责两件事：

1. 把“谁在监控谁”表达清楚
2. 把“RCA 需要的观测事实从哪里来”表达清楚

因此当前关系应固定为：

1. topo 提供资源结构
2. monitoring_v2 task graph 提供任务运行事实
3. 监控地图把两者与监控源、监控目标、观测事实串起来
4. 观测归一化层从监控地图收敛出 `layered_input`
5. `pkg/rca` 只消费 `layered_input`

##### 当前可直接复用的 OneOps 能力

当前构思监控地图时，不需要另起一套全新基础设施，可直接复用下面这些现有能力：

1. `maintenance.monitor_probe`
   - 提供监控任务下发与部署事实
   - 但它本身不直接等于最终监控源标识

2. `platform/monitoring_v2`
   - 提供目标解析、任务投影、运行态快照、task graph
   - 用于确认“哪个 agent 在执行哪些 remote 监控任务”

3. `app/topology`
   - 提供统一资源拓扑出口
   - 其上游物理拓扑数据应以 `L2NodeMapServer` 周期产物为基础

4. `application/v2`
   - 提供应用对象
   - 当前自动 `runs_on` / `device_code` 承载事实还未完全闭合，第一阶段允许手工补录应用到服务器关联

5. `ArpTableServer`
   - 提供服务器与物理拓扑之间的挂接关系

6. 手工关联能力
   - 第一阶段允许手工补录：
     - 应用到服务器
     - 应用到应用

7. 日志检索侧能力
   - 提供节点日志证据

因此当前“监控地图”的正确构思方向不是再造底层数据源，而是：

- **在平台侧增加一个覆盖关系视图层，把现有事实拼成稳定监控体系表达**

##### 当前已固定的数据来源与最小实现

为了避免后续再次泛化，当前把监控地图的数据来源固定为下面 5 条：

1. 监控源来自监控任务的实际执行者
   - 谁执行了 remote 监控任务，谁就是当前监控源
   - 这里的主标识应是 agent，而不是仅仅是 `monitor_probe` 记录

2. 物理拓扑来自 `L2NodeMapServer` 周期任务
   - 这部分是监控地图的基础数据之一
   - 它负责回答“网络设备之间当前怎样连接”
   - 当前实际处理入口是 `AnalysisProcessCommon -> enum.L2nodeMapServer`

3. 服务器与物理拓扑的关联来自 `ArpTableServer` 周期任务
   - 这部分负责把服务器挂接到物理拓扑上
   - 它负责回答“这台服务器当前接在哪个网络位置上”
   - 当前代码里要区分两个处理分支：
     - `AnalysisProcessCommon -> enum.ArpTableServer -> ArpTableServerV2`
     - `AnalysisProcessCommon -> enum.ArpTableServer2 -> ArpTableServer2`

4. 应用与服务器的关联当前先靠手工关联
   - 当前不等待自动发现闭合后再启动主线
   - 第一阶段允许最小化手工维护 `application -> server`

5. 应用与应用的关联当前也先靠手工关联
   - 第一阶段允许最小化手工维护 `application -> application`
   - 只要求关系可录入、可读取、可进入监控地图

这一段当前只表达一个主线决定：

- **监控地图先把多层级关系串起来，不等待所有自动发现能力成熟后才开始**

##### 当前不应做的事

按照主线约束，监控地图当前明确不进入下面这些内容：

1. 不做动态布局设计
2. 不做复杂交互编排
3. 不做智能推荐或自动推理 UI
4. 不做第二轮应用依赖 RCA 语义
5. 不做多监控源联合判定实现
6. 不做分布式监控选路与调度设计
7. 不等待应用自动建模闭合后再推进当前主线

这些都可以是后续阶段的话题，但当前不进入。

##### 当前应收口的最小功能

如果把监控地图作为下一步主线任务的一部分，当前只建议先收口下面 4 个最小功能：

1. 能看到监控源点与被监控目标的覆盖关系
2. 能看到某个目标对象由哪些监控任务负责观测
3. 能看到物理拓扑、服务器挂接、应用承载、应用依赖这几层关系是否已经补齐
4. 能看到 RCA 一期所需观测事实是否已齐：
   - 节点事实
   - 路径命中边
   - 阻断路径边
   - 日志证据

这 4 个最小功能里，当前最重要的是第 4 条。

原因只有一条：

- 监控地图的第一价值不是展示，而是帮助判断 RCA 上游输入是否闭合

##### 当前主线判断

因此，当前关于“是否需要监控地图”的结论可以直接固定为：

1. 需要
2. 它是 topo 与 RCA 之间缺失的一层
3. 它首先是观测归一化载体，不是独立根因引擎
4. 它当前应放在 `OneOps/app/platform` 主线内推进，而不是新建顶级模块
5. 在主线未闭合前，先只定义最小层级与职责，不进入实现细节

这也是当前最符合主线约束的推进方式：

- 不抽象泛化
- 不冗余兜底
- 不在主线未闭合前做过多细节设计

#### 7.10.17 监控地图最小读模型

在上面的边界已经固定后，当前可以继续收口一件事：

- 监控地图在第一阶段应以什么最小读模型被读取

这里先固定一个原则：

- **第一阶段只定义读模型，不定义写模型，不定义存储模型**

##### 最小对象

第一阶段监控地图只需要 6 类对象：

1. `monitor_source`
   - 表示实际执行 remote 监控任务的 agent
   - 最小字段只保留：
     - `source_id`
     - `agent_code`

2. `monitor_task`
   - 表示某个监控源上实际运行的任务
   - 最小字段只保留：
     - `task_id`
     - `source_id`
     - `task_type`

3. `monitor_target`
   - 表示被覆盖的目标对象
   - 最小字段只保留：
     - `target_id`
     - `target_kind`
   - 第一阶段只允许：
     - `network`
     - `server`
     - `application`

4. `topology_node`
   - 表示物理拓扑里的节点
   - 最小字段只保留：
     - `node_id`
     - `node_kind`

5. `topology_edge`
   - 表示物理拓扑里的边
   - 最小字段只保留：
     - `a_node_id`
     - `b_node_id`

6. `observation_fact`
   - 表示 RCA 当前真正要消费的观测事实
   - 第一阶段只保留：
     - `node_observations`
     - `active_path_edges`
     - `blocked_path_edges`
     - `log_evidence_node_ids`

##### 最小关系

第一阶段监控地图只需要 6 类关系：

1. `source_runs_task`
   - `monitor_source -> monitor_task`

2. `task_covers_target`
   - `monitor_task -> monitor_target`

3. `server_attached_to_topology`
   - `server -> topology_node`
   - 这部分来自 `ArpTableServerV2` / `ArpTableServer2`

4. `application_hosted_on_server`
   - `application -> server`
   - 第一阶段允许手工维护

5. `application_depends_on_application`
   - `application -> application`
   - 第一阶段允许手工维护

6. `target_has_observation_fact`
   - `monitor_target -> observation_fact`

##### 当前读取顺序

为了避免监控地图再次泛化，第一阶段读取顺序只固定为：

1. 先读实际执行 remote 监控任务的 agent
2. 再读该 agent 上的监控任务
3. 再读这些任务当前覆盖的目标对象
4. 再读物理拓扑与服务器挂接关系
5. 再补应用承载与应用依赖手工关系
6. 最后只校验 RCA 一期所需观测事实是否齐全

这里固定这条顺序的目的只有一个：

- 监控地图第一阶段首先要回答“当前这套监控到底覆盖到了哪里，以及 RCA 能不能拿到足够输入”

##### 当前最小读取结果

第一阶段监控地图若要对外提供读取结果，当前只建议输出下面 4 个区块：

1. `sources`
   - 当前有哪些 agent 是有效监控源

2. `coverage`
   - 每个监控源下有哪些任务
   - 每个任务覆盖了哪些目标

3. `relations`
   - 物理拓扑
   - 服务器挂接
   - 应用承载
   - 应用依赖

4. `observation_readiness`
   - 对每个目标只回答：
     - 节点事实是否有
     - 路径事实是否有
     - 阻断边事实是否有
     - 日志证据是否有

这里刻意不输出：

1. 排序
2. 评分
3. 推荐动作
4. 自动结论

##### 当前主线价值

这个最小读模型当前只服务三件事：

1. 让监控地图从“概念”变成可读取的事实视图
2. 让观测归一化层知道去哪里取第一轮所需事实
3. 让后续验收时可以直接看出 RCA 上游输入缺在哪一层

因此当前阶段可以把监控地图的主线目标继续固定为：

- **先把监控覆盖关系和观测事实就绪度读出来**
- **不在这个阶段提前做 RCA 推理本体**

#### 7.10.18 监控地图最小接口边界

在读模型已经固定后，当前还需要再收口一件事：

- 监控地图在第一阶段以什么接口边界对外提供读取

这里同样只固定最小边界，不进入实现类设计。

##### 当前接口定位

第一阶段监控地图接口只承担一件事：

- **读取某个范围内的监控覆盖关系与观测事实就绪度**

它当前不是：

1. RCA 分析入口
2. 拓扑生成入口
3. 监控任务下发入口
4. 手工关系维护入口

##### 当前最小入参

第一阶段当前只建议保留 4 个最小入参：

1. `tenant_code`
   - 限定读取范围

2. `observed_at`
   - 统一观测时点

3. `target_ids`
   - 可选
   - 若传入，则只看指定目标

4. `source_ids`
   - 可选
   - 若传入，则只看指定监控源

这里当前明确不加入：

1. 排序参数
2. 分页策略设计
3. 模糊搜索表达式
4. 时间窗口聚合
5. 多快照对比

##### 当前最小返回结构

第一阶段若定义一个最小返回结构，当前只建议固定下面 5 个区块：

1. `sources`
   - 监控源列表

2. `tasks`
   - 监控任务列表

3. `targets`
   - 监控目标列表

4. `relations`
   - 当前范围内的：
     - 物理拓扑关系
     - 服务器挂接关系
     - 应用承载关系
     - 应用依赖关系

5. `readiness`
   - 当前范围内每个目标的观测事实就绪度

这里当前不建议把返回结构继续扩成通用图协议。

原因只有一条：

- 当前主线要的是“RCA 上游可读事实”，不是一个通用图平台协议

##### 当前就绪度检查项

`readiness` 当前只允许回答下面 6 项：

1. `has_monitor_source`
   - 是否已经能定位到实际执行任务的 agent

2. `has_monitor_task`
   - 是否已经能定位到覆盖该目标的监控任务

3. `has_topology_relation`
   - 是否已经能把该目标挂到物理拓扑语义上

4. `has_node_observation`
   - 是否已有节点 `up/down` 事实

5. `has_active_path`
   - 是否已有命中路径事实

6. `has_blocked_path_or_log_evidence`
   - 是否已有阻断路径边事实或日志证据

第一阶段当前不允许继续扩成：

1. 分数
2. 百分比
3. 风险等级
4. 自动原因分类

##### 当前接口与 RCA 的衔接关系

第一阶段这条边界当前应固定为：

1. 监控地图接口先返回覆盖关系与就绪度
2. 观测归一化层据此汇聚 `layered_input`
3. 若就绪度缺项，则在归一化层明确报缺，不继续伪造输入
4. `pkg/rca` 仍然只消费已经成型的 `layered_input`

这里要继续固定一条主线纪律：

- **监控地图接口不替代 `IMonitoringRootCauseObservationNormalizer`**
- **它只为归一化层提供可读取事实**

##### 当前建议的代码落点

虽然当前不写代码，但代码边界可以先收口一句话：

- 监控地图读取接口若后续落代码，仍应放在 `OneOps/app/platform/service` 主线内

原因仍然不变：

1. 它要同时消费监控任务、物理拓扑、服务器挂接、应用关系、日志证据
2. 它天然是跨域读取，不属于单一业务域
3. 它是 RCA 上游事实读取层，不是 RCA 机制层

##### 当前阶段结论

因此，第一阶段监控地图接口只需要闭合下面 3 件事：

1. 给定范围，能读出监控覆盖关系
2. 给定范围，能读出关系补齐情况
3. 给定范围，能读出 RCA 一期所需事实是否就绪

做到这 3 件事后，主线才继续进入下一步：

- 用它支撑 `IMonitoringRootCauseObservationNormalizer`

##### 当前已落地状态

截至当前代码状态，这一段主线已经继续推进到下面 5 点：

1. 监控地图最小读取接口已落地，并已稳定返回：
   - `sources`
   - `tasks`
   - `targets`
   - `relations`
   - `readiness`
2. `relations` 当前已接通两类应用层关系：
   - `application_hosted_on_server`
   - `application_depends_on_application`
3. layered RCA 当前会先读取监控地图，再自动把这两类关系映射为：
   - `hosting`
   - `dependency`
4. 若监控地图已经返回目标 `readiness` 项，RCA 当前会继续回填：
   - `has_node_observation`
   - `has_active_path`
   - `has_blocked_path_or_log_evidence`
5. 若同一个目标的 `readiness` 已明确缺少：
   - `has_monitor_task`
   - `has_topology_relation`
   - `has_active_path`
   - 或当前 layered 输入里缺少目标节点观测
   - 则 RCA 当前直接返回 `unable_to_conclude_yet`

这里再固定一条实现边界：

- 当前拒判只在监控地图已返回该目标 `readiness` 项时生效
- 若监控地图尚未返回该目标 `readiness`，当前实现不会伪造“缺失监控源/任务/关系”的结论

因此，当前主线已经从“监控地图可读取”推进到：

- **监控地图事实已经能直接影响 RCA 一期的可判/不可判**

#### 7.10.19 监控地图第一阶段验收口径

在能力定义、读模型、接口边界都已经固定后，当前还需要再固定一件事：

- 第一阶段到底怎样算“监控地图已闭合”

这里当前只按主线收口，不增加附加目标。

##### 验收输入

第一阶段验收输入只要求下面 5 类事实各至少有一份：

1. 至少 1 个实际执行 remote 监控任务的 agent
2. 至少 1 组该 agent 上的监控任务
3. 至少 1 份物理拓扑结果
4. 至少 1 份服务器挂接结果
5. 至少 1 组手工应用关系
   - `application -> server`
   - 或 `application -> application`

当前验收不要求：

1. 全租户全量覆盖
2. 所有应用都已自动建模
3. 所有目标都有路径事实

##### 验收动作

第一阶段验收动作只固定为 4 步：

1. 读取一个租户在某个 `observed_at` 的监控地图
2. 抽查 `sources / tasks / targets / relations / readiness`
3. 抽查至少一个网络设备目标、一个服务器目标、一个应用目标
4. 对照当前真实来源，确认每层关系是否被正确读出

##### 通过标准

第一阶段通过标准只保留下面 6 条：

1. 能读出至少一个监控源
   - 且该监控源能对应到实际执行任务的 agent

2. 能读出监控任务与目标覆盖关系
   - 即 `source -> task -> target` 关系链不为空

3. 能读出物理拓扑关系
   - 即至少存在一组 `topology_node / topology_edge`

4. 能读出服务器挂接关系
   - 即至少存在一组 `server_attached_to_topology`

5. 能读出手工应用关系
   - 即至少存在一组：
     - `application_hosted_on_server`
     - 或 `application_depends_on_application`

6. 能对目标输出观测事实就绪度
   - 即 `readiness` 至少能明确回答：
     - `has_monitor_source`
     - `has_monitor_task`
     - `has_topology_relation`
     - `has_node_observation`
     - `has_active_path`
     - `has_blocked_path_or_log_evidence`

##### 失败标准

当前任一情况出现，都应直接视为第一阶段未闭合：

1. 监控源仍然只能落到 `monitor_probe` 记录，无法收敛到实际执行任务的 agent
2. 任务存在，但无法读出任务覆盖了哪些目标
3. 物理拓扑存在，但服务器无法挂接到拓扑语义上
4. 应用目标存在，但既没有服务器承载关系，也没有应用依赖关系
5. `readiness` 不能对目标逐项回答“已有/缺失”
6. 为了补齐结果而在读取阶段伪造关系或伪造观测事实

##### 当前验收结论的使用方式

第一阶段验收当前只允许产出两类结论：

1. 已闭合
   - 可以进入 `IMonitoringRootCauseObservationNormalizer` 对接阶段

2. 未闭合
   - 必须明确缺口落在哪一层：
     - 监控源层
     - 任务覆盖层
     - 物理拓扑层
     - 服务器挂接层
     - 应用关系层
     - 观测事实层

这里当前不允许产出：

1. 模糊通过
2. 部分智能补偿后通过
3. 降级视为通过

##### 当前主线结论

因此，监控地图第一阶段的验收目标可以继续固定为一句话：

- **能稳定读出监控覆盖关系、层级关系、事实就绪度，就算闭合**
- **任何一层缺失，都明确报缺，不做伪闭合**

#### 7.10.20 监控地图与 ObservationNormalizer 的对接顺序

在监控地图第一阶段已经收口后，当前还需要固定最后一条主线：

- 监控地图怎样进入 `IMonitoringRootCauseObservationNormalizer`

这里当前只定义对接顺序，不定义实现拆分。

##### 当前对接原则

当前只固定 3 条原则：

1. 监控地图负责提供可读取事实
2. `IMonitoringRootCauseObservationNormalizer` 负责把这些事实收敛成 `layered_input`
3. 如果事实不齐，归一化层直接报缺，不继续伪造 RCA 输入

也就是说，当前边界必须继续保持：

- **监控地图不是 `layered_input`**
- **ObservationNormalizer 才是 `layered_input` 的直接生产方**

##### 当前对接顺序

第一阶段对接顺序只建议固定为 6 步：

1. 先读取监控地图
   - 输入范围仍然是：
     - `tenant_code`
     - `observed_at`
     - 可选的 `target_ids`
     - 可选的 `source_ids`

2. 再确定目标对象
   - 把当前分析范围内的目标先收敛成：
     - 网络设备
     - 服务器
     - 应用

3. 再确定监控源
   - 从监控地图里收敛到实际执行 remote 任务的 agent
   - 不再停留在 `monitor_probe` 静态记录层

4. 再确定层级关系
   - 物理拓扑
   - 服务器挂接
   - 应用承载
   - 应用依赖

5. 再确定一期观测事实
   - `node_observations`
   - `active_path_edges`
   - `blocked_path_edges`
   - `log_evidence_node_ids`

6. 最后再生成 `layered_input`
   - 若缺项，则返回明确缺口
   - 若齐全，才进入 `pkg/rca`

##### 当前字段收敛顺序

为了避免归一化层继续膨胀，第一阶段当前只允许按下面这个顺序收敛字段：

1. `monitor_node_id`
2. `target_node_id`
3. `extra_nodes[].application`
4. `extra_edges[].hosting`
5. `node_observations[]`
6. `active_path_edges[]`
7. `blocked_path_edges[]`
8. `log_evidence_node_ids[]`

这个顺序当前只表达一个约束：

- 先把监控源、目标、层级关系补齐，再判断观测事实是否够用

##### 当前缺口处理规则

第一阶段若对接过程中发现缺口，当前处理规则只允许下面两类：

1. 关系缺口
   - 例如：
     - 找不到服务器挂接关系
     - 找不到应用承载关系
     - 找不到应用依赖关系
   - 处理方式：
     - 明确返回缺哪一层
     - 不自行猜测关系

2. 事实缺口
   - 例如：
     - 没有节点事实
     - 没有路径事实
     - 没有阻断路径边事实
   - 处理方式：
     - 明确返回缺哪类事实
     - 不自行补默认值

当前明确不允许：

1. 因缺字段而自动降级成旧 RCA 路径
2. 因缺关系而自动猜测承载关系
3. 因缺路径而自动构造“伪实际路径”

##### 当前对接结果

第一阶段对接结果当前只允许两种：

1. `layered_input_ready`
   - 说明监控地图已足够支撑一期归一化

2. `layered_input_not_ready`
   - 说明监控地图事实还不够
   - 且必须返回明确缺口列表

这里当前不允许出现：

1. `best_effort_ready`
2. `partially_ready`
3. `fallback_to_old_path`

##### 当前主线结论

因此，监控地图与 `IMonitoringRootCauseObservationNormalizer` 的关系当前可以固定为一句话：

- **监控地图负责把事实读出来**
- **ObservationNormalizer 负责把事实压成 RCA 一期输入**
- **事实不齐就停，不做伪归一化**

#### 7.10.21 手工应用关系最小约束

在当前主线里，应用关系仍然有一部分不能依赖自动发现闭合。

因此还需要固定一条最小边界：

- 第一阶段允许手工维护应用关系
- 但手工关系只能服务于监控地图与 RCA 主线
- 不扩成通用知识图谱能力

##### 当前允许的关系类型

第一阶段当前只允许 2 类手工应用关系：

1. `application_hosted_on_server`
   - 即 `application -> server`
   - 用于补齐应用承载位置

2. `application_depends_on_application`
   - 即 `application -> application`
   - 用于补齐最小依赖关系

当前不允许继续扩成：

1. 调用方向
2. 流量比例
3. 主备权重
4. 依赖强度
5. 关系评分

##### 当前关系宿主判断

当前从代码现实看，最接近的宿主已经存在：

1. `application/v2`
   - 已经有 `device_code`
   - 已经有 `runs_on` 同步逻辑
   - 已经依赖 `EntityRelationSvc`

2. `platform/api/entity_relation.go`
   - 已经提供通用关系创建、查询、删除入口
   - 且会按 schema 校验关系类型与目标类型

因此当前主线不建议再新建一套专门的“手工应用关系”机制。

更合理的方向是：

- **优先复用现有 `EntityRelationSvc` 与 relation schema 能力**
- **只补当前主线所需的最小关系类型**

##### 当前最小字段

第一阶段手工应用关系当前只需要 4 个最小字段：

1. `source_id`
2. `source_type`
3. `target_id`
4. `relation_type`

其中：

- `source_type` 当前应固定为 `application_v2`
- `target_id` 可以是：
  - `device_v2`
  - `application_v2`

当前第一阶段不要求额外字段：

1. 描述文本
2. 权重
3. 方向标签
4. 生效时间
5. 失效时间

##### 当前校验规则

为了避免手工关系失控，第一阶段当前只允许下面 6 条校验规则：

1. `application_hosted_on_server`
   - 源必须是 `application_v2`
   - 目标必须是 `device_v2`

2. `application_depends_on_application`
   - 源必须是 `application_v2`
   - 目标必须是 `application_v2`

3. 同一条关系不允许重复创建

4. `application_hosted_on_server` 第一阶段每个应用只允许有 1 条有效关系
   - 即一个应用只承载在一个服务器上

5. 不允许应用依赖自己

6. 如果 schema 未声明该关系类型，则不允许写入

这里当前特别固定一条主线约束：

- **手工关系必须可校验，不允许裸写自由关系**

##### 当前与现有 `runs_on` 的关系

第一阶段对 `application -> server` 关系，当前必须把口径收紧：

1. `runs_on` 是当前最接近的现实锚点
2. 若已有自动同步出的 `runs_on`，则优先复用
3. 若当前自动同步还不能稳定闭合，则允许手工补齐
4. 手工补齐不应和已有自动 `runs_on` 产生冲突

因此当前更合理的主线处理是：

- 先读已有 `runs_on`
- 缺失时再读手工关系
- 若两者冲突，直接报关系冲突，不自动覆盖

##### 当前与监控地图的关系

第一阶段手工应用关系进入监控地图时，只承担两件事：

1. 把应用挂到服务器上
2. 把应用依赖链最小化补齐

它当前不承担：

1. 决定根因
2. 推断实际路径
3. 替代日志证据
4. 替代监控任务覆盖事实

##### 当前验收标准

第一阶段手工应用关系若要视为闭合，当前只要求下面 4 条：

1. 能创建 `application -> server` 关系
2. 能创建 `application -> application` 关系
3. 监控地图读取时能把这两类关系读出来
4. ObservationNormalizer 能消费这些关系生成：
   - `extra_nodes[].application`
   - `extra_edges[].hosting`

##### 当前主线结论

因此，手工应用关系当前可以继续固定为一句话：

- **它只是第一阶段补齐应用层关系的最小手段**
- **必须复用现有关系服务并受 schema 校验约束**
- **不扩成新的通用关系系统**

#### 7.10.22 监控地图第一阶段实施顺序

到这里，第一阶段主线定义已经基本闭合。

当前接下来最重要的不是继续扩文档，而是把实施顺序固定下来，避免实现时再次发散。

##### 当前实施原则

第一阶段实施只遵守 3 条原则：

1. 先打通读取，再打通归一化
2. 先闭合层级关系，再闭合观测事实
3. 任一层未闭合时，不提前进入下一层

##### 当前实施顺序

第一阶段当前只建议按下面 5 步推进：

1. 先落监控地图读取骨架
   - 目标只有一个：
     - 能按 `tenant_code + observed_at` 读出 `sources / tasks / targets`
   - 这一阶段先不要求读出完整关系和事实

2. 再落层级关系读取
   - 只补四类关系：
     - 物理拓扑
     - 服务器挂接
     - 应用承载
     - 应用依赖
   - 到这一步，监控地图至少应能回答“目标挂在哪一层关系上”

3. 再落就绪度读取
   - 只补：
     - `has_monitor_source`
     - `has_monitor_task`
     - `has_topology_relation`
     - `has_node_observation`
     - `has_active_path`
     - `has_blocked_path_or_log_evidence`
   - 到这一步，监控地图至少应能回答“RCA 一期输入还缺什么”

4. 再对接 `IMonitoringRootCauseObservationNormalizer`
   - 只做：
     - 读取监控地图
     - 收敛 `layered_input`
     - 缺项时报缺
   - 这一阶段不改 `pkg/rca`

5. 最后做定向验收
   - 只验收：
     - 监控地图读取
     - ObservationNormalizer 对接
     - RCA 一期入口是否能拿到正确输入

##### 每一步的闭合标准

为了避免实施过程反复回滚，当前每一步都要有单独闭合标准：

1. 第一步闭合标准
   - 能稳定读出：
     - 至少一个监控源
     - 至少一组任务
     - 至少一个目标

2. 第二步闭合标准
   - 能稳定读出：
     - 物理拓扑关系
     - 服务器挂接关系
     - 手工应用关系

3. 第三步闭合标准
   - 能稳定输出逐目标 `readiness`
   - 且缺项是明确缺项，不是空值蒙混

4. 第四步闭合标准
   - ObservationNormalizer 能基于监控地图生成：
     - `monitor_node_id`
     - `target_node_id`
     - `extra_nodes[].application`
     - `extra_edges[].hosting`
     - 以及当前已有的事实字段

5. 第五步闭合标准
   - 至少一组真实范围或伪集成范围能够走通：
     - 监控地图
     - ObservationNormalizer
     - RCA 一期入口

##### 当前不允许的推进方式

第一阶段当前明确不允许下面这些推进方式：

1. 先写完整 UI 再补读取能力
2. 先写 RCA 归一化猜测逻辑，再回头补监控地图
3. 先把应用依赖做复杂化，再回头补服务器挂接
4. 先引入多监控源融合，再把单监控源主线补齐
5. 先做通用图协议或图库抽象，再回头补主线事实

##### 当前阶段性里程碑

第一阶段当前只建议固定 3 个里程碑：

1. `M1`
   - 监控地图能读出覆盖关系

2. `M2`
   - 监控地图能读出层级关系与事实就绪度

3. `M3`
   - ObservationNormalizer 能消费监控地图并产出 RCA 一期输入

到 `M3` 之前，不进入：

1. 第二轮应用依赖 RCA
2. 多监控源联合判定
3. 分布式监控调度
4. 复杂前端交互

##### 当前主线结论

因此，监控地图第一阶段的实施顺序当前可以继续固定为一句话：

- **先读覆盖，再读关系，再读就绪度，最后才接入 RCA 归一化**
- **不跨阶段抢跑，不提前展开第二轮能力**

#### 7.10.23 第一阶段开发计划

在上面的主线已经闭合后，第一阶段开发计划当前可以直接固定为 `M1 / M2 / M3` 三段。

这里不再展开更多分支，只围绕主线任务推进。

##### `M1` 监控覆盖读取

`M1` 当前只做一件事：

- 把监控地图最小读取骨架落出来

当前任务清单只保留 4 条：

1. 定义监控地图最小读取接口
2. 读取监控源、监控任务、监控目标
3. 返回 `sources / tasks / targets`
4. 补最小伪集成验收数据

当前交付物只保留：

1. 监控地图读取接口边界
2. `sources / tasks / targets` 读取结果
3. 对应定向测试或夹具

`M1` 退出标准只有一个：

- 能稳定回答“谁在监控谁”

##### `M2` 层级关系与就绪度

`M2` 当前只做两件事：

1. 把层级关系读出来
2. 把事实就绪度读出来

当前任务清单只保留 5 条：

1. 接入物理拓扑读取
2. 接入服务器挂接读取
3. 接入手工应用关系读取
4. 输出 `relations`
5. 输出 `readiness`

当前交付物只保留：

1. `relations`
   - 物理拓扑
   - 服务器挂接
   - 应用承载
   - 应用依赖
2. `readiness`
   - 6 项最小检查项
3. 对应定向测试或夹具

`M2` 退出标准只有一个：

- 能稳定回答“目标挂在哪一层、RCA 一期还缺什么”

## 8. 主线验收清单

当前建议只按下面这份清单做真实数据验收。

### 8.1 拓扑输入验收

验收输入：

- 选定一个真实租户
- 该租户下至少存在 2 个真实连通节点
- `phy_node_map` 能产出原始 `L2NodeMap`

验收动作：

1. 调用 `app/topology` 生成统一拓扑
2. 抽查 `nodes`
3. 抽查 `edges`

验收标准：

- 每个节点都有 `device_code`
- 每条边都有 `a_device_code`
- 每条边都有 `b_device_code`
- 每条边都有 `a_interface_code`
- 每条边都有 `b_interface_code`
- 不满足以上任一项时，视为拓扑输入未闭合

### 8.2 RCA 主路径验收

验收输入：

- 一组真实告警编码
- 这些告警属于同一租户
- 告警标签里能解析出异常设备对象

验收动作：

1. 调用 `POST /alert/alarm/root-cause/analyze`
2. 观察是否能补齐 `tenant_code`
3. 观察是否能补齐 `observed_at`
4. 观察是否生成候选结果

验收标准：

- 请求能进入统一 RCA 主路径
- 返回结果中包含 `request_id`
- 返回结果中包含 `tenant_code`
- 返回结果中包含 `observed_at`
- 返回结果中包含 `candidates`
- 候选对象只出现节点候选、直接故障边候选、唯一最短路径桥接候选、多路径公共桥接候选

### 8.3 拓扑契约错误验收

验收输入：

- 一份故意缺少必需字段的拓扑数据

验收动作：

1. 让 RCA 消费这份不完整拓扑
2. 观察 `platform` 入口返回
3. 观察 `alert` 入口返回

验收标准：

- RCA 不继续生成候选
- 返回明确错误
- 错误内容直接指出缺失字段
- 入口层不再返回泛化失败消息

### 8.4 日志证据验收

验收输入：

- 一次真实异常时间窗
- 至少一个候选设备在该时间窗内存在日志证据

验收动作：

1. 调用 RCA
2. 对比有日志证据和无日志证据的候选结果

验收标准：

- 有日志证据的候选增加 `log` 证据源
- 没有日志证据的候选不伪造 `log`
- 日志当前只作为证据附加，不改变候选类型

### 8.5 主线闭合判定

当前主线只在以下条件同时满足时，才视为闭合：

- 统一拓扑输出稳定满足 RCA 最小契约
- 告警入口能稳定收敛到异常对象和观测时间
- RCA 能稳定输出候选与证据
- 拓扑契约错误能被明确暴露
- 日志证据能稳定挂载到候选结果

在此之前，不进入排序、置信度、历史因果分析、变更关联建模。

### 8.6 验收执行顺序

真实数据验收时，顺序固定为：

1. 先做 `8.1 拓扑输入验收`
2. 再做 `8.3 拓扑契约错误验收`
3. 然后做 `8.2 RCA 主路径验收`
4. 最后做 `8.4 日志证据验收`

若前一步不通过，不进入下一步。

在进入真实数据验收前，当前代码侧也应先按固定顺序跑完可执行夹具：

1. `pkg/rca` 平台无关机制夹具
2. `platform/service/impl` OneOps 侧伪集成夹具
3. `platform/api` 与 `alert/api` 入口回归

对应命令以 [ONEOPS_ROOT_CAUSE_TEST_DATASET.md](/home/jacky/project/OneOPS-ALL/docs/ONEOPS_ROOT_CAUSE_TEST_DATASET.md) 为准。

### 8.7 验收记录模板

每次验收只记录以下内容：

- 验收日期
- 租户编码
- 告警编码列表
- 观测时间
- 拓扑输入是否满足最小契约
- RCA 是否成功返回候选
- 候选数量
- 是否存在 `log` 证据
- 是否暴露明确错误
- 结论：通过 / 不通过
- 阻塞点

建议每次验收结果都按同一格式落一条记录，不再写泛化结论。

### 8.8 首条验收任务单

第一条真实验收任务只做一件事：

- 选定一个真实租户，完整跑通一次 `拓扑输入 -> RCA 主路径 -> 日志证据` 验收链路

执行要求：

1. 只选一个租户
2. 只选一组同租户告警
3. 只记录一次观测时间
4. 若拓扑输入不满足最小契约，当次验收直接结束
5. 若 RCA 入口返回明确错误，当次验收直接结束

首条验收完成标准：

- 已拿到一条真实验收记录
- 该记录能明确判断主线当前是通过还是不通过
- 若不通过，阻塞点能直接落到拓扑输入、入口归一化、RCA 候选生成、日志证据四者之一
