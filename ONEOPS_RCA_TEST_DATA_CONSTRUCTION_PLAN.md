# OneOPS RCA 测试数据构造计划

本文档只定义 RCA 前期与中期测试所需的最小数据构造计划。

约束固定：

1. 围绕核心主线，不泛化，不衍生
2. 不冗余，不降级，不兜底，不智能
3. 主线闭环前，不做过度细节设计

---

## 1. 目标

当前目标不是建设通用仿真平台，而是建设一套可控的数据构造链路，用来稳定推动 RCA 主线测试。

主线闭环固定为：

1. 页面创建拓扑与设备
2. 页面配置标签附着关系
3. 页面配置测试场景
4. 指标发生器生成 Prometheus 指标
5. Monitoring Map / RCA 读取这些事实
6. RCA 输出结论
7. 页面展示输入事实与 RCA 结果

只有这条链闭环，当前阶段才算完成。

---

## 2. 当前阶段不做的事

当前阶段明确不做：

1. 通用拓扑管理平台
2. 自动发现
3. 智能推断
4. 自动补全缺失关系
5. 排序、置信度、历史因果
6. 复杂权限与协作
7. 细粒度版本管理
8. 与真实生产数据强绑定
9. 超出 RCA 主线的高级编辑能力

---

## 3. 需要构造的最小对象

当前只构造五类对象：

1. 监控源
2. 设备
3. 拓扑关系
4. 标签附着
5. 测试场景

### 3.1 监控源

监控源表示 remote 监控任务的执行点，对应当前 RCA 的观测起点。

最小字段：

1. `source_id`
2. `source_name`
3. `device_code`
4. `agent_code`
5. `enabled`

### 3.2 设备

设备不是只指网络设备，也包括服务器和应用对象。

最小字段：

1. `device_code`
2. `device_name`
3. `device_type`
4. `ip`
5. `labels`

当前 `device_type` 只允许三类：

1. `network`
2. `server`
3. `application`

### 3.3 拓扑关系

当前只保留 RCA 主线需要的关系：

1. `network_connected_to_network`
2. `server_attached_to_network`
3. `application_hosted_on_server`
4. `application_depends_on_application`

若后续二层/三层真实转发拓扑接入，再单独扩 observed path 事实源，不在当前计划中提前泛化。

### 3.4 标签附着

标签附着要对齐 OneOps 当前模式，即通过标签把监控目标、设备、应用、场景关联起来。

当前最小要求：

1. 设备支持标签附着
2. 应用支持标签附着
3. 监控任务目标支持标签附着
4. Prometheus 发生器按标签决定指标归属

### 3.5 测试场景

测试场景表示“这一次要让哪些事实变成 up/down、哪些路径活跃、哪些路径阻断”。

最小字段：

1. `scenario_id`
2. `scenario_name`
3. `monitor_source_id`
4. `target_selector`
5. `node_states`
6. `active_paths`
7. `blocked_paths`

---

## 4. 页面最小能力

页面只做最小闭环，不做重型编辑器。

### 4.1 页面一：拓扑与设备编辑

最小能力：

1. 新建设备
2. 修改设备基本信息
3. 设置设备类型
4. 画关系边
5. 删除关系边
6. 保存草稿

### 4.2 页面二：标签附着编辑

最小能力：

1. 给设备附着标签
2. 给应用附着标签
3. 给监控目标附着标签
4. 查看标签命中对象

### 4.3 页面三：场景编辑

最小能力：

1. 选择监控源
2. 选择目标范围
3. 设置节点 up/down
4. 设置路径 active
5. 设置路径 blocked
6. 一键生成本次 Prometheus 指标
7. 一键触发 RCA 测试

### 4.4 页面四：结果回放

最小能力：

1. 显示本次构造的节点观测
2. 显示本次构造的 active path
3. 显示本次构造的 blocked path
4. 显示 Monitoring Map 摘要
5. 显示 RCA 结论

---

## 5. Prometheus 发生器最小职责

Prometheus 发生器不是监控系统本身，只是测试事实发生器。

当前只负责生成 RCA 主线真正消费的事实。

### 5.1 当前必须生成的事实

第一批只生成：

1. `ping_result_code`
2. 可供 `node_observations` 映射的最小标签
3. 与监控源、设备、目标对应的标签组合

其中当前明确使用：

1. `device_code`
2. 与监控源关联所需标签
3. 与租户或场景隔离所需标签

### 5.2 当前不提前生成的事实

当前不提前做：

1. 大量无消费方的指标
2. 智能派生指标
3. 自动补路径指标
4. 超出 RCA 主线的复杂时序噪声

### 5.3 发生器输出边界

发生器至少要支持两种输出：

1. 直接暴露给测试 Prometheus
2. 导出为固定夹具文件

原因：

1. 前者方便联调
2. 后者方便稳定回归

---

## 6. 路径事实构造原则

当前路径事实必须遵循已经确认的理解：

1. 路径优先表示实际报文转发路径
2. 简单场景下才允许使用受控拓扑路径近似
3. 多路径在真实场景中是常态，不允许强行压成单路径
4. 两台设备之间也可能存在多路径

因此当前测试数据构造分两层：

### 6.1 前期测试

前期测试允许手工直接构造：

1. `active_paths`
2. `blocked_paths`

这一步的目的不是还原真实网络采集，而是先稳定验证 RCA 机制。

### 6.2 中期测试

中期测试再逐步让路径事实从这些来源生成：

1. `L2NodeMap`
2. `ArpTable`
3. 二层转发拓扑
4. 三层转发拓扑

当前只有前两项已有基础入口，后二者尚未实现。

---

## 7. 测试场景分层

当前测试场景只分三层，不再扩展。

### 7.1 前期场景

目标是稳定验证 RCA 主判定顺序。

必须覆盖：

1. 网络路径阻断
2. 承载服务器异常
3. 应用依赖异常
4. 应用自身异常
5. 无法继续收敛

### 7.2 中期场景

目标是验证多路径与实际路径事实。

必须覆盖：

1. 单目标单路径
2. 单目标多路径
3. 两失败点之间多路径
4. 多路径存在公共桥接候选
5. 多路径不存在确定性候选

### 7.3 OneOps 适配场景

目标是验证标签附着与平台映射。

必须覆盖：

1. 标签命中设备
2. 标签命中应用
3. 标签命中监控目标
4. 单监控源场景
5. 监控源切换场景

---

## 8. 实施顺序

顺序固定，不跳步。

### 阶段 1：固定数据模型

只做：

1. 拓扑对象
2. 设备对象
3. 标签附着对象
4. 场景对象

完成标准：

1. 可以保存一份最小场景配置
2. 可以导出为固定 JSON

### 阶段 2：做最小页面

只做：

1. 拓扑与设备编辑
2. 标签附着编辑
3. 场景编辑

完成标准：

1. 可以从页面生成一份完整场景

### 阶段 3：做最小 Prometheus 发生器

只做：

1. 根据场景生成 `ping_result_code`
2. 生成当前 RCA 消费所需标签

完成标准：

1. `node_observations` 可从发生器数据稳定读取

### 阶段 4：做路径事实构造器

只做：

1. 手工 `active_paths`
2. 手工 `blocked_paths`

完成标准：

1. `MonitoringMapReader` 能稳定读取这些路径事实
2. RCA 能稳定消费这些路径事实

### 阶段 5：做结果联调页

只做：

1. 显示构造输入
2. 显示 Prometheus 指标摘要
3. 显示 Monitoring Map 摘要
4. 显示 RCA 结论

完成标准：

1. 一次操作可跑通“构造 -> 生成 -> 分析 -> 展示”

### 阶段 6：固化前期与中期回归数据集

只做：

1. 把典型场景导出为夹具
2. 接入脚本化回归

完成标准：

1. 页面场景与代码夹具可互相对应
2. RCA 回归可重复执行

---

## 9. 当前最小落地建议

下一步不要同时做页面、生成器、真实接入。

当前最合理的最小落地顺序是：

1. 先定义场景 JSON 契约
2. 再做场景到 Prometheus 指标的发生器
3. 再做场景到 Monitoring Map 的事实投影
4. 最后做可视化编辑页面

原因：

1. RCA 当前最缺的是可控输入，不是页面壳子
2. 先固定契约，后面页面才不会反复推翻
3. 页面应该只是编辑器，不应反过来主导模型

---

## 10. 计划完成标准

本计划完成只看四件事：

1. 能创建一个最小测试场景
2. 能生成 Prometheus 指标
3. 能生成 Monitoring Map / RCA 所需事实
4. 能稳定验证 RCA 前期与中期场景

若这四件事未闭环，不进入下一层设计。

---

## 11. 接续建议

按当前主线，下一步应先补一份独立的“场景 JSON 最小契约”文档，只定义：

1. 设备
2. 关系
3. 标签
4. 节点状态
5. 路径状态
6. 预期 RCA 结果

这份契约确定后，再开始代码与页面实现。
