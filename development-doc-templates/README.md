# 标准化开发文档模板

本目录从 `docs/prompts` 中提取开发文档结构，去除了角色提示词、执行提示词和生成过程说明，仅保留可复用的项目交付文档模板。

## 使用顺序

| 阶段 | 模板 | 主要用途 | 建议输出目录 |
|------|------|----------|--------------|
| 需求 | `01-需求概要模板.md` | 明确业务背景、角色、功能、规则、边界 | `rqdoc/` 或 `doc/` |
| 需求 | `02-功能清单模板.md` | 将需求拆成可排期、可验收的功能项 | `rqdoc/` 或 `doc/` |
| 需求/设计 | `03-实体属性清单模板.md` | 梳理业务实体字段、约束和关系 | `dbdesign/` 或 `doc/` |
| 原型 | `04-原型清单模板.md` | 拆分子系统、模块、页面和导航 | `prototype/` |
| 设计 | `05-数据库设计模板.md` | 输出表结构、索引、字典和数据支撑规则 | `dbdesign/` 或 `doc/` |
| 设计/开发 | `06-接口文档模板.md` | 定义 HTTP 接口、参数、响应、状态码 | `docs/api/` 或 `apidoc/` |
| 开发 | `07-后端开发设计模板.md` | 约定工程结构、分层职责和代码实现范围 | `doc/` 或 `devdoc/` |
| 测试 | `08-测试用例模板.md` | 定义功能测试、边界测试和性能测试用例 | `testcase/` |
| 测试/联调 | `09-接口调用文档模板.md` | 面向联调和测试的接口调用说明 | `apidoc/` |
| 测试 | `10-自动化测试脚本说明模板.md` | 约定 pytest 脚本结构、配置和报告输出 | `tests/`、`reports/` |
| 审查 | `11-Service代码审查报告模板.md` | 记录 Service 层代码质量、事务、性能和规范问题 | `review/` 或 `doc/` |

## 通用填写规则

- 文档中所有 `{占位符}` 必须替换为项目实际内容。
- 信息缺失但需要保留时，使用 `[待确认]` 标记，不要留空。
- 表格字段能填写“无”时写“无”，不要删除列。
- 版本号统一使用 `V1.0`、`V1.1` 这类格式。
- 日期统一使用 `yyyy-mm-dd`。
- 每份文档开头保留“文档信息”章节，便于追溯。
- 涉及接口、数据库、测试的文档应明确关联上游文档。

## 推荐交付链路

1. 先完成需求概要，确认业务目标、用户角色、系统边界。
2. 基于需求概要生成功能清单，形成排期和验收基础。
3. 基于需求和功能清单梳理实体属性，再深化为数据库设计。
4. 按子系统和模块拆分原型清单，补齐页面、导航、组件关系。
5. 基于 Service 或业务能力输出接口文档。
6. 基于需求和接口文档输出测试用例与接口调用文档。
7. 自动化测试脚本和代码审查报告作为开发质量闭环补充交付。
