# OneOPS 标准开发流程

本文档定义 OneOPS 的标准开发流程。它吸收了 `docs/prompts` 中需求、原型、数据库、接口、开发、测试等阶段的经验，并适配当前项目的 autodev 与探索驱动测试方式。

## 总体流程

```text
需求输入
  -> 需求概要
  -> 原型/流程/实体拆分
  -> 概要设计
  -> 数据/API/模块设计
  -> 实现
  -> 代码审核
  -> 测试用例与自动化测试
  -> evidence 回写
  -> 下一轮迭代
```

复杂项目允许边探索边测试，但探索必须有方法：

```text
目标方向
  -> flow line
  -> final goal
  -> critical environments
  -> weak-point hypotheses
  -> H0/H1/H2/H3 harm classification
  -> probe design
  -> evidence
  -> repair / continue / defer / approval
```

## 阶段 1：需求输入

需求输入可以来自用户描述、现有代码、客户 PoC 关注点、问题反馈、截图、接口联调失败、自动化 evidence。

每个需求至少要记录：

- 背景：为什么要做。
- 目标：最终要证明或交付什么。
- 范围：本次做什么，不做什么。
- 角色：谁使用，谁维护，谁审核。
- 风险：最担心失败在哪里。
- 验收：什么证据可以证明完成。

信息不足时，AI/autodev worker 只能生成候选假设或占位，不得把推断写成已确认需求。所有需要人类判断的内容必须使用 `[待人工确认]`、`[待补充]` 或 `[候选方案]` 标记，并说明需要确认的人或角色。

## AI 与人工职责边界

AI 可以做：

- 整理用户输入、历史 prompts、代码、测试和 evidence。
- 提炼可能的需求、设计问题、风险和文档缺口。
- 根据已有事实生成候选章节、候选字段、候选接口、候选测试用例。
- 对不确定位置放置占位，并标明需要谁确认。
- 发现冲突、缺口、H0/H1 风险并提出问题。

AI 不可以做：

- 把没有来源的推断写成正式需求。
- 代替产品或业务负责人决定业务规则、角色权限、验收口径。
- 代替架构负责人决定核心架构、数据模型、跨系统边界。
- 代替运维或安全负责人确认生产环境、凭证、外部系统和设备操作边界。
- 为了让文档完整而编造字段、接口、流程、性能指标或测试数据。
- 删除人类尚未确认但可能有价值的历史信息；应标记为 `[待人工确认]`、`[已废弃候选]` 或迁移到附录。

占位格式建议：

```text
[待人工确认: 产品负责人] 这里需要确认真实业务规则。
[待补充: 后端负责人] 这里需要补充实际接口 path 和响应结构。
[候选方案: AI 提炼] 以下内容来自历史 prompts，需要人工评审后才能进入正式设计。
```

## 文档可用性门禁

所有标准工程文档都必须先通过人工补充和确认，才能作为后续开发、测试或 autodev 执行依据。

默认规则：

- AI 初稿或自动化生成文档，默认状态为 `blocked-human-confirmation`。
- 只要存在 `[待人工确认]`、影响关键结论的 `[待补充]` 或未采纳的 `[候选方案]`，文档不可直接使用。
- 人工补充并确认后，文档才能进入 `draft`。
- 人工评审通过后，文档才能进入 `reviewed`。
- 只有 `reviewed` 或 `active` 文档，才能用于拆分 story、发布 autodev batch、驱动开发实现、作为测试验收依据。

执行类任务的 gate：

```text
输入文档状态 = blocked-human-confirmation
  -> 停止执行
  -> 输出待确认清单
  -> 等待人工补充/确认

输入文档状态 = draft
  -> 只能做评审、补充、问题发现
  -> 不发布执行类 story

输入文档状态 = reviewed/active
  -> 可以进入 story 拆分、开发、测试、自动化执行
```

## 阶段 2：需求概要

需求概要文档回答“系统要做什么”，不进入具体技术实现。

标准内容包括：

- 文档信息。
- 业务背景与目标。
- 用户角色。
- 功能需求。
- 非功能需求概要。
- 关键业务规则。
- 系统边界与约束。
- 用户故事。
- 术语表与参考文档。

输出建议：

```text
docs/requirements/【需求概要】<系统或模块名>_V<版本号>.md
```

## 阶段 3：原型、流程与实体拆分

当需求涉及页面、交互、数据录入、审核流、监控视图时，需要拆分原型或流程。

页面类需求应输出：

- 子系统划分，例如管理端、移动端、H5、监控端。
- 模块页面清单。
- 页面层级结构。
- 页面导航关系。
- 模块内公共组件。
- 与其他模块的页面或数据关联。

数据/后端类需求应输出：

- 核心业务实体。
- 实体关系。
- 状态机或业务流转。
- 关键字段来源。
- 外部系统依赖。

输出建议：

```text
docs/requirements/prototype/【原型清单】<系统>_<子系统>_<模块>_V<版本号>.md
docs/requirements/prototype/【原型页面导航】<系统>_V<版本号>.md
```

## 阶段 4：概要设计与模块设计

概要设计回答“怎么组织系统能力”，但仍要避免过早陷入实现细节。AI 可以提出设计候选，但不能反推并确认设计决策；未确认内容必须保留为占位。

至少包含：

- 模块边界。
- 核心流程。
- 数据流转。
- 关键接口。
- 权限与安全边界。
- 异常与降级策略。
- 可观测性要求。
- 测试策略。

对于 OneOPS，设计文档还应说明：

- 目标子项目：`OneOPS`、`OneOPS-UI`、`agent`、`teleabs`、`telegraf` 等。
- 是否涉及设备、监控、日志、拓扑、凭证、任务调度。
- 是否触碰外部系统或真实设备。
- 是否需要 autodev loop 或人工 approval。

## 阶段 5：数据库设计

数据库设计应基于需求概要、原型清单、实体关系和现有 schema。

标准内容包括：

- 核心实体识别。
- ER 关系。
- 数据表清单。
- 详细表结构。
- 索引设计。
- 数据字典。
- 关键业务规则的数据支撑。
- 数据量预估与扩展性说明。
- 软删除与审计字段策略。

约束：

- 不要只凭 prompt 新建表，必须检查现有 schema 和迁移文件。
- 新表/字段必须说明业务来源和查询路径。
- 不确定的字段类型、长度、关系使用 `[待人工确认]`，不得为了完整性自行定稿。
- 涉及生产数据迁移时必须单独写 migration plan。

输出建议：

```text
docs/design/database/【数据库设计】<系统>_<模块>_V<版本号>.md
```

## 阶段 6：API 设计与接口文档

API 文档必须先于或同步于接口实现，作为前后端联调和测试自动化的契约。

每个 API 至少说明：

- 接口名称与业务目的。
- HTTP method 与 path。
- 认证方式。
- 请求参数。
- 响应结构。
- 业务成功判断。
- 失败响应。
- 权限要求。
- 幂等性与副作用。
- 关联 Service/handler。

输出建议：

```text
docs/api/【接口文档】<系统>_<模块>_V<版本号>.md
```

## 阶段 7：实现

实现必须遵循当前子项目既有框架和代码风格。历史 prompts 中的 Java/Nebula 规范可作为 Java 模块参考，但不能机械套用到 Go、Vue、Agent、Telegraf 等模块。

实现前需要确认：

- 写入范围。
- 可修改文件。
- 不可修改文件。
- 是否需要迁移脚本。
- 是否会触碰真实设备、外部系统、凭证、任务下发。
- 是否需要人工 approval。

实现中需要保持：

- 小步提交。
- 代码与文档同步。
- 测试与 evidence 同步。
- 不把 read-only discovery 与 controlled action 混在同一任务中。

## 阶段 8：代码审核

代码审核优先关注风险，而不是格式表扬。

高优先级检查：

- 业务逻辑是否违反需求。
- 是否有错误的数据/设备/tenant 目标。
- 是否泄露 secret。
- 是否缺少权限校验。
- 是否错误吞掉异常。
- 是否把 HTTP 200 当作业务成功。
- 是否有事务、并发、缓存一致性风险。
- 是否缺少关键测试。

Java Service 类可参考历史 prompt 的规则：

- public 方法命名清晰。
- 参数校验完整。
- `@Transactional` 只用于有效 public 方法。
- catch 不吞掉导致事务应回滚的异常。
- 查询方法考虑 read-only transaction。
- 避免 `new Thread()` 和不受控线程池。
- Redis 大对象缓存需评估是否适合本地缓存，并说明多实例一致性风险。

## 阶段 9：测试与 evidence

测试分为：

- 功能测试。
- 接口测试。
- 性能测试。
- 回归测试。
- 探索驱动测试。
- 安全边界测试。

测试完成后必须留下 evidence：

- 测试命令。
- 环境。
- fixture。
- 输入。
- 输出。
- 业务状态判断。
- 失败原因。
- 下一步决策。

复杂项目的测试 evidence 应采用 `PASS/FAIL/BLOCKED/APPROVAL/SKIPPED` 状态词，并按 H0/H1/H2/H3 记录风险级别。

## 阶段 10：回写与迭代

任何自动化或人工测试产生的稳定结论，都应回写到对应文档：

- 需求变更回写需求文档。
- API 变化回写接口文档。
- 表结构变化回写数据库设计。
- 测试新增回写测试用例。
- 弱点与修复回写开发规范或模块设计。

这一步是让项目长期变清晰的关键，不做回写，文档会重新退化成一次性材料。
