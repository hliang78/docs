## 2. 业务背景与目标

### 2.1 业务背景

Device V2 采集入库后，需要继续完成监控纳管和日志纳管，才能形成完整纳管闭环。现有流程已覆盖入库、同步 Device V1、监控任务下发等部分能力，但日志纳管与远程配置链路仍需要补齐。

### 2.2 业务目标

- 为单台设备提供手动“继续纳管”入口。
- 对网络设备完成监控任务下发和 syslog 优先的日志纳管。
- 对服务器支持 agent 或 SNMP 监控模式选择。
- 对服务器日志支持 agent 优先；无 agent 时生成 SSH 配置计划。
- 以配置成功作为纳管成功依据，不要求 Prometheus 或 Loki 数据到达验证。
- 失败必须暴露明确原因，不做降级、兜底或智能推断。

## 4. 功能需求

### 4.1 单设备继续纳管

用户在 Device V2 管理页面对单台设备触发“继续纳管”，系统生成并执行该设备的监控纳管、日志纳管动作。

### 4.2 纳管证据记录

系统将 action 结果记录在 `device_v2_store_run.summary_json.onboarding.actions` 中，后续重试只处理 `failed` 或 `unknown` 的 ensure action。

### 4.3 网络设备日志纳管

首期优先支持 syslog。系统确认区域内有开启 syslog 监听器的 agent 后，按照明确模板远程配置网络设备 syslog 目标。

### 4.4 服务器日志纳管

有 agent 时下发日志转发任务；无 agent 时生成 SSH 配置计划。业务远程服务器操作必须通过 controller 提供的 API 实现。

## 5. 非功能需求概要

### 5.1 性能需求

首期仅执行单台设备，不定义批量并发执行指标。

### 5.2 可用性需求

远程访问延迟或失败时，界面必须展示失败 action 和明确错误，不允许静默成功。

### 5.3 安全性需求

账号凭据不得写入开发文档、故事文件或工具配置。开发测试工具只能用于授权测试目标。

### 5.4 可扩展性需求

SNMP trap、更多厂商命令模板和日志策略可后续扩展，但首期不做推断式模板。

## 6. 关键业务规则

| 编号 | 规则 |
| --- | --- |
| BR-001 | 首期只对单台设备执行，批量只生成计划或逐台展示结果。 |
| BR-002 | 纳管成功依据是配置成功，不要求 Prometheus/Loki 数据到达。 |
| BR-003 | 重试只处理 `failed` 或 `unknown` 的 ensure action。 |
| BR-004 | 网络设备日志首期优先 syslog。 |
| BR-005 | 服务器监控模式由用户明确选择 agent 或 SNMP。 |
| BR-006 | 有 agent 时服务器日志优先 agent；无 agent 时生成 SSH 配置计划。 |
| BR-007 | 业务功能中的远程服务器操作必须通过 controller API。 |

## 7. 系统边界与约束

### 7.1 系统边界

| 边界类型 | 内容 |
| --- | --- |
| 前端边界 | 仅 Device V2 管理页面局部变更，不改全局样式或路由。 |
| 后端边界 | 仅 Device V2 纳管相关 API 和本地服务，不改全局 API 语义。 |
| 工具边界 | `docs/openclaw-autodev/tools/d2on/` 仅用于开发测试探索。 |
| 外部依赖 | controller API、monitor task、agent syslog listener、netlink 网络设备库。 |

### 7.2 约束条件

- 不做降级、兜底分支、mock 成功或智能推断。
- 不吞错误，缺少契约或模板时必须 BLOCKED/failed。
- 不新增全局前后端变更、依赖升级、迁移或生产配置变更。

## 8. 用户故事

### 8.1 运维管理员用户故事

#### 8.1.1 单设备继续纳管

作为运维管理员，我希望对一台已入库设备点击继续纳管，以便补齐监控和日志配置。

作为运维管理员，我希望看到每个 action 的执行结果和错误详情，以便决定重试或补充配置。
