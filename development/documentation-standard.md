# OneOPS 开发文档规范

本文档定义 OneOPS 正式开发文档的类型、命名、结构和维护规则。

## 文档原则

- **文档服务开发闭环:** 文档不是归档物，而是需求、实现、测试、修复的共同输入。
- **文件优先:** 重要结论必须写入文件，不能只留在对话或 prompt 中。
- **证据优先:** 结论应尽量引用代码、接口、测试、截图、日志、evidence。
- **标记不确定性:** 无法确认的内容使用 `[待人工确认: <角色>]`、`[待补充]` 或 `[候选方案]`。
- **人工确认优先:** AI 可以补结构、列候选、标风险、放占位，但不得替人确认需求、设计、业务规则、权限、安全边界、验收口径。
- **保留历史:** 过时内容使用 `[已废弃]` 标记，并指向替代文档。
- **中文优先:** 面向团队长期维护的正式工程文档产物必须以中文为主。
- **机器可读优先:** 自动化要消费的字段、状态、enum 不随意翻译或改名。

## 文档语言规范

正式工程文档产物必须以中文为主。英文可以保留，但只能用于必要的技术标识和机器可读内容。

必须使用中文的内容：

- 背景、目标、范围、非目标。
- 业务规则、人工确认占位、风险说明。
- 流程、验收标准、测试说明。
- 面向团队阅读的模板正文和维护规则。

允许保留英文的内容：

- 文件名、目录名、命令、类名、函数名、字段名。
- API method/path、JSON key、enum、状态值。
- 代码片段和配置字段。
- 团队共识的关键概念，例如 `PRD`、`OpenClaw`、`autodev`、`story`、`loop`、`evidence`、`fixture`、`probe`、`read-only`、`blocked-human-confirmation`、`reviewed`、`active`。

禁止：

- 正式工程文档主体使用英文段落输出。
- 为了复用历史 prompts 而保留英文模板主体。
- 把面向人工确认和维护的规则写成英文。

## 人工确认与占位规则

当文档需要的信息无法从用户输入、代码、现有文档、测试 evidence 中直接确认时，必须使用占位。占位不是文档缺陷，而是保护长期正确性的机制。

推荐标记：

| 标记 | 使用场景 |
|---|---|
| `[待人工确认: 产品负责人]` | 业务目标、用户角色、业务规则、验收口径 |
| `[待人工确认: 架构负责人]` | 架构边界、模块职责、跨系统设计、数据模型 |
| `[待人工确认: 后端负责人]` | API path、DTO、Service 边界、数据库字段 |
| `[待人工确认: 前端负责人]` | 页面结构、交互、可视化呈现 |
| `[待人工确认: 测试负责人]` | 测试范围、性能指标、fixture、通过标准 |
| `[待人工确认: 运维/安全负责人]` | 生产环境、凭证、设备命令、外部系统写操作 |
| `[待补充]` | 明确缺少事实材料 |
| `[候选方案: AI 提炼]` | AI 根据 prompts/代码/evidence 提炼出的候选内容 |

禁止写法：

- 用“应当”“必须”“系统已经支持”等措辞描述未经确认的推断。
- 为了让章节完整而编造字段、接口、流程、性能指标或测试数据。
- 删除不确定内容后不留占位，导致后续人工不知道这里缺少决策。

推荐写法：

```text
[待人工确认: 产品负责人] 是否需要按租户维度隔离该业务规则。
[候选方案: AI 提炼] 历史 prompts 建议使用 sdk/local/local.starter 三段式模块，仅适用于 Java/Nebula 模块。
[待补充: 测试负责人] 需要补充性能测试并发数、目标 TPS 和错误率。
```

## 推荐目录

```text
docs/
  development/            # 开发规范与流程
  requirements/           # 需求概要、用户故事、原型清单
  design/                 # 架构设计、模块设计、数据库设计
  api/                    # 接口文档
  testing/                # 测试用例、测试策略、测试报告
  openclaw-autodev/       # 自动化任务、stories、evidence
  prd-autodev/            # PRD/autodev program 文档
```

当前仓库已有多类历史文档，不要求一次性迁移。新增正式文档优先使用上述目录；旧文档在被引用时逐步整理。

## 文档类型

| 类型 | 目的 | 推荐位置 |
|---|---|---|
| 需求概要 | 说明业务背景、目标、角色、功能、边界 | `docs/requirements/` |
| 原型清单 | 说明页面、导航、组件、子系统拆分 | `docs/requirements/prototype/` |
| 概要设计 | 说明模块边界、流程、架构、约束 | `docs/design/` |
| 数据库设计 | 说明实体、表、字段、索引、数据规则 | `docs/design/database/` |
| API 文档 | 说明接口契约、请求、响应、错误 | `docs/api/` |
| 测试用例 | 说明功能、接口、性能、边界测试 | `docs/testing/cases/` |
| 测试报告 | 说明实际执行结果与证据 | `docs/testing/reports/` |
| 探索 evidence | 说明 flow line、probe、weak point、决策 | `docs/openclaw-autodev/evidence/` |

## 命名规范

正式中文文档可以使用以下命名：

```text
【需求概要】<系统或模块>_V<版本号>.md
【原型清单】<系统>_<子系统>_<模块>_V<版本号>.md
【原型页面导航】<系统>_V<版本号>.md
【概要设计】<系统或模块>_V<版本号>.md
【数据库设计】<系统>_<模块>_V<版本号>.md
【接口文档】<系统>_<模块>_V<版本号>.md
【测试用例】<系统>_<模块>_V<版本号>.md
```

面向自动化和代码库引用的文档，建议使用稳定英文 slug：

```text
device-v2-ingest-design.md
monitoring-v2-api.md
ws-01-exploration-evidence-schema.md
```

## 通用文档头

每份正式文档建议包含：

| 项目 | 内容 |
|---|---|
| 文档名称 | `<名称>` |
| 适用范围 | `<系统/模块/接口/测试范围>` |
| 版本 | `V1.0` |
| 创建日期 | `YYYY-MM-DD` |
| 最近更新 | `YYYY-MM-DD` |
| 维护人 | `[待人工确认: 项目负责人]` |
| 关联文档 | `<PRD/API/测试/evidence>` |
| 状态 | `blocked-human-confirmation/draft/reviewed/active/deprecated` |

## 文档状态门禁

标准工程文档必须有明确状态。状态不是装饰字段，而是后续开发、测试、autodev 是否可以使用该文档的门禁。

| 状态 | 含义 | 是否可作为后续依据 |
|---|---|---|
| `blocked-human-confirmation` | 文档由 AI 或自动化初步生成，仍包含 `[待人工确认]`、`[待补充]` 或关键 `[候选方案]`，缺少人工确认 | 否 |
| `draft` | 人工已开始补充，但仍未完成评审；可以作为讨论材料 | 否 |
| `reviewed` | 人工已补齐关键占位并完成评审，可作为开发、测试、story 拆分依据 | 是 |
| `active` | 已在当前版本或当前自动化体系中正式生效 | 是 |
| `deprecated` | 已废弃，只能作为历史参考 | 否 |

状态转换规则：

```text
AI 初稿
  -> blocked-human-confirmation
  -> 人工补充/确认
  -> draft
  -> 人工评审通过
  -> reviewed
  -> 纳入当前工程流程
  -> active
```

门禁规则：

- 只要文档存在未解决的 `[待人工确认:*]`，状态必须是 `blocked-human-confirmation` 或 `draft`，不得标记为 `reviewed/active`。
- 只要文档存在影响需求、设计、API、数据、权限、安全、测试通过标准的 `[待补充]`，不得进入 `reviewed/active`。
- `[候选方案: AI 提炼]` 只有被人工明确接受、改写为正式内容，或移入“未采纳候选/附录”，文档才可以进入 `reviewed`。
- `blocked-human-confirmation` 文档不能直接作为开发实现、数据库设计、API 实现、测试验收、autodev story 发布的依据。
- autodev/story worker 如果发现输入文档状态为 `blocked-human-confirmation`，必须停止推进执行类任务，只能产出问题清单、占位清单或请求人工确认。
- 人工确认后，应记录确认人、日期和确认范围。

建议在文档头增加：

| 项目 | 内容 |
|---|---|
| 状态 | `blocked-human-confirmation` |
| 人工确认人 | `[待人工确认: 项目负责人]` |
| 人工确认日期 | `[待人工确认: 项目负责人]` |
| 未解决占位数 | `<数字>` |

## 状态词

人类文档可使用中文描述，但自动化 evidence 统一使用：

- `PASS`
- `FAIL`
- `BLOCKED`
- `APPROVAL`
- `SKIPPED`

风险等级统一使用：

- `H0`: unsafe / invalidates gate。
- `H1`: gate-untrustworthy / blocks proof。
- `H2`: coverage gap。
- `H3`: efficiency / maintainability。

## 引用规则

- 引用代码时写明文件路径、函数、结构体/类名。
- 引用接口时写明 method/path。
- 引用测试时写明命令、环境、fixture。
- 引用自动化 evidence 时写明 evidence 文件路径。
- 引用历史 prompt 时只作为来源，不作为正式规范本身。

## 更新规则

以下变更必须更新文档：

- 新增/修改 API。
- 新增/修改数据库表或字段。
- 修改业务状态机或核心流程。
- 修改权限、安全边界、设备操作边界。
- 新增自动化测试或改变测试判定方式。
- 发现 H0/H1 风险或完成对应修复。

## AI/autodev worker 使用规则

AI/autodev worker 在执行开发任务前，应优先读取：

1. 相关 PRD 或需求概要。
2. 模块设计/API/数据库文档。
3. 当前 story scope 和 allowed paths。
4. 既有 evidence。
5. 本目录的开发规范。

不得仅凭 `docs/prompts` 中的旧提示词直接生成或修改代码。

AI/autodev worker 输出正式文档时，必须把以下内容保留为人工占位：

- 业务规则。
- 用户角色和权限。
- 核心架构和模块边界。
- 数据库字段和数据关系。
- 外部系统、真实设备、凭证、生产环境边界。
- 性能指标和验收标准。
- 任何会影响客户 PoC 结论或上线判断的内容。
