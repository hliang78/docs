# GPU 服务器故障模式知识条目

## 数据来源

本文基于 `tmp/服务故障事件记录表.xlsx` 提炼，共包含 475 条有效记录。

事件结构特征：

- 可明确归类为“故障/事件”的记录：474
- 故障：284
- 事件：190
- 另有 1 条记录分类字段缺失，但故障描述仍可作为案例参考
- 主要机型：`4090`、`A800二期扩容`、`A800`、`A10`、`H100`、`910B`

## 这份台账告诉了我们什么

原始表不是标准 RCA 系统，但已经足够说明一线值班的高频现实：

1. 真正高频的问题高度集中在少数几类硬件/驱动/网络症状。
2. 很多处理动作并不复杂，复杂的是证据采集、是否该升级、是否该更换备件。
3. 不同机型的高频故障结构不一样，runbook 不能只写“通用服务器”。

## 高频故障模式

### Top cause -> solution

按记录频次看，最常见的原因与处理组合是：

| 排名 | 故障模式 | 高频处理 |
| --- | --- | --- |
| 1 | GPU故障 | 更换故障 GPU |
| 2 | 服务器宕机 | 冷重启 |
| 3 | GPU掉卡 | 更换故障 GPU |
| 4 | GPU ECC | 更换故障 GPU |
| 5 | 内存故障 | 更换故障内存 |
| 6 | GPU掉卡 | 对调 GPU |
| 7 | 风扇故障 | 更换故障风扇 |
| 8 | CPU故障 | 更换故障 CPU |
| 9 | 硬盘故障 | 更换故障硬盘 |
| 10 | 服务器重启后网卡名变化 | 修改网卡名 |

这意味着知识库和 SOP 至少要优先覆盖：

- GPU 在位 / 掉卡 / ECC / XID
- 宕机和自动重启
- 内存 / CPU / 风扇 / 硬盘
- IB 或 ROCE 网卡命名和对应关系异常

## 按机型观察

### 4090

高频模式：

- `GPU故障 -> 更换故障GPU`
- `GPU掉卡 -> 更换故障GPU`
- `服务器宕机 -> 冷重启`
- `内存故障 -> 更换故障内存`

知识含义：

- 4090 的问题更偏 GPU 本体和内存本体
- runbook 里应明确 GPU 物理定位、SN 获取限制和换卡验证流程

### A800

高频模式：

- `服务器宕机 -> 冷重启`
- `风扇故障 -> 更换故障风扇`
- `GPU故障 / GPU ECC -> 更换故障GPU`

知识含义：

- A800 除 GPU 外，风扇/散热链路也很值得重点巡检
- 需要把 `fabricmanager`、`nvidia_peermem`、OFED 这类检查前置

### A800 二期扩容

高频模式：

- `GPU掉卡 -> 更换故障GPU`
- `风扇故障 -> 更换故障风扇`
- `GPU ECC -> 更换故障GPU`
- `服务器重启后网卡名变化 -> 修改网卡名`

知识含义：

- 这批设备呈现出“GPU + 风扇 + IB 命名”三类并发高频问题
- 说明修复后验证不能只看 `nvidia-smi`，还要看网络命名和业务验证

### A10

高频模式：

- `服务器宕机 -> 冷重启`
- `GPU掉卡 -> 对调GPU`
- `CPU故障 -> 更换故障CPU`

知识含义：

- A10 里“对调 GPU 验证”出现频率高，适合单独写成设备级判定动作
- 说明并非所有 GPU 问题都直接换卡，先做模组/位置交叉验证有价值

### H100

高频模式：

- `GPU故障 -> 更换故障GPU`
- `GPU ECC -> 更换故障GPU`

知识含义：

- H100 当前样本中 GPU 硬故障占比明显高
- runbook 对 H100 应尽量缩短“是否升级更换”的判断路径

### 910B

高频模式：

- `内存故障 -> 更换故障内存`
- `NPU掉卡 -> 冷重启`
- `NPU掉卡 -> 整机替换`

知识含义：

- 910B 应与英伟达 GPU 系列分开看待
- 一旦进入 NPU 掉卡且冷重启无效，整机替换是现实中的高频路径

## 可直接沉淀到 runbook 的结论

### 1. 先分类，再深挖

一线排障不应先从“全量命令收集”开始，而应先把问题分进以下桶：

- GPU/NPU 在位异常
- GPU ECC/XID/驱动异常
- 宕机/自动重启
- 风扇/电源/散热异常
- 内存/CPU/硬盘/RAID 异常
- IB/ROCE/网卡命名/存储网络异常

### 2. “冷重启”是高频动作，但不是终点

宕机、NPU 掉卡、首次 GPU 掉卡都经常先走冷重启。但知识库要明确：

- 冷重启只是第一响应，不是 RCA
- 冷重启后必须补采集证据和验证结果
- 同类问题重复出现时应直接升级厂商或备件路径

### 3. 更换备件前的证据最关键

表里高频解法是换件，但换件前至少要保证：

- 故障设备编码、机柜位置、机型明确
- 系统、带外、业务三侧症状一致
- GPU/NPU/内存/风扇等部件的识别证据已留存

### 4. 业务恢复不等于技术闭环

表里包含“业务验证后反馈未排除”“业务侧反馈故障排除时间”等字段，说明真正闭环需要同时满足：

- 技术动作完成
- 业务侧确认恢复
- 记录被补全

## 适合补成 OneOPS 结构化标签的字段

后续如果要把这类台账产品化，建议在 OneOPS 里优先结构化这些标签：

- 故障类型：故障 / 事件
- 故障域：GPU / NPU / 内存 / CPU / 硬盘 / 风扇 / 电源 / 网络 / 系统 / 驱动
- 机型：4090 / A800 / A10 / H100 / 910B
- 首次动作：冷重启 / 对调 / 修改网卡名 / 重装 / 更换备件
- 最终动作：恢复 / 换件 / 整机替换 / 厂商升级
- 闭环状态：待验证 / 已恢复 / 已复发 / 待复盘

## 当前知识缺口

这份台账能告诉我们“什么问题多、通常怎么解”，但还缺三类信息：

1. 每条记录关联的真实设备、告警、日志、拓扑上下文
2. 更换前后的证据链和附件
3. 可机器检索的标准化字段

因此本条目适合作为经验知识，不适合作为审计级事实源。
