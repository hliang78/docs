# PRD

## 目标

- 把 Device V2 的前端体验从“可入库、可采集、可监控的散点页面”收敛成“围绕关键因素工作的闭环页面”。
- 优先优化：
  - [DeviceV2IngestPipelineRedesign.vue](/Users/huangliang/project/OneOPS-ALL/OneOPS-UI/src/views/device/DeviceV2IngestPipelineRedesign.vue)
  - [DeviceV2ManagementGrouped.vue](/Users/huangliang/project/OneOPS-ALL/OneOPS-UI/src/views/device/DeviceV2ManagementGrouped.vue)
- 前端展示以业务文案优先，尽量避免把 code 作为主表达；code 只保留为辅助信息和精确定位手段。

## 非目标

- 本轮不优先实现“手工新增设备”的全新主流程。
- 本轮不接入 OpenClaw 自动化任务拆分或发布。
- 本轮不假设存在“同步删除 V1”后端接口；若契约不存在，只做删除提示增强，不做伪实现。

## 关键因素模型

### 1. 身份定位

- 设备名称 / 业务名称
- 设备编码
- 业务编码
- 主机名
- SN
- 资产编号

### 2. 采集接入

- 带内 IP
- 带外 IP
- 带内凭据
- 带外凭据
- SNMP 凭据
- WinRM 凭据 / WinRM 端口

### 3. 纳管同步

- 平台
- 设备类别
- 租户
- 区域
- 机房
- 机架
- V1 关联状态 / `device_v1_code`

### 4. 分组与检索

- group tags
- labels
- 关键位置类 attributes

### 5. 设备事实

- 厂商
- 型号
- 机架号
- 系统版本 / OS 版本
- CPU / 内存

### 6. 来源追溯

- 来源
- 来源批次
- 任务 ID
- 运行 ID
- 原始编码

## 当前真实代码事实

- 导入页已经承接大量字段，但“可纳管”仅根据地址 + 凭据判断，缺少对同步 V1、位置归属、来源追溯的显式表达。
- 管理页已有编辑、删除、采集、监控、分组，但筛选维度偏少，关键因素展示不成体系。
- 已确认前端存在 `create/update/delete/get/list/sync-to-v1` 等 Device V2 能力。
- 已确认当前后端支持 `DELETE /device/v2/:code?delete_v1=true`：默认删除 V2；当显式携带 `delete_v1=true` 且存在 `device_v1_code` 时，会继续调用 V1 独立删除能力。

## 方案

## A. 导入页优化

### A1. 把草稿表从“字段录入表”升级为“关键因素工作台”

- 每行以业务文案展示当前关键因素状态，而不是只展示编码和凭据文本。
- 至少新增以下状态表达：
  - 身份信息是否完整
  - 采集接入是否就绪
  - 纳管同步是否就绪
  - 来源追溯是否完整

### A2. 把结果文案拆清楚

- `可提交` 只表示“可以写入 V2 清册”。
- `可采集` 表示地址和凭据已具备。
- `可同步 V1` 表示平台、设备类别、机房、机架等前置字段已具备。
- `可先登记` 表示允许先入库，但后续仍需补关键因素。

### A3. 前端表达尽量业务化

- 平台、设备类别、机房、机架优先显示名称。
- 凭据用“已填写 / 未填写”或友好摘要，不把 credential ref 当作主视觉。
- 来源类信息以“Excel 上传 / 历史批次 / 手工提交 / 批次号”表达。

## B. 管理页优化

### B1. 增加关键因素筛选

- 在现有关键词 + 监控状态基础上，补充：
  - 平台
  - 设备类别
  - 设备状态
  - 区域
  - 机房
  - 机架
  - 关键因素状态

### B2. 增加关键因素状态筛选

- 建议至少支持：
  - 可采集
  - 待补采集条件
  - 可同步 V1
  - 待补纳管条件
  - 已关联 V1

### B3. 列表与详情加强关键因素展示

- 列表页增加对“采集接入 / 纳管同步 / 来源追溯”的摘要表达。
- 详情侧栏补充：
  - 关键因素总览
  - 采集接入状态
  - 纳管同步状态
  - 来源与追溯

### B4. 删除闭环增强

- 行内增加单台删除入口。
- 删除确认文案明确：
  - 当前执行的是删除 Device V2
  - 默认不会自动删除 V1
  - 对已绑定 `device_v1_code` 的设备，提供“同时删除已关联 V1”可选项
  - 当前端勾选该选项时，调用 `DELETE /device/v2/:code?delete_v1=true`
  - 若 V2 删除成功但 V1 删除失败，页面要以“部分失败”形式明确提示

## 验收标准

### 导入页

1. 用户可以在草稿表中直接看出每台设备是：
   - 可提交
   - 可采集
   - 可同步 V1
   - 仅可先登记
2. 用户无需理解大量 code，也能看懂平台、位置、设备类别和来源状态。
3. 用户能更快定位“为什么这台设备还不能继续采集或纳管”。

### 管理页

1. 用户可以按关键因素组合筛选设备。
2. 用户可以在列表中看出设备是否缺少采集接入因子或同步 V1 因子。
3. 用户可以从详情面板中快速判断当前设备的资料缺口。
4. 用户可以执行单台删除，且删除提示不会误导为“已经联动删除 V1”。

## 风险

- 当前管理页使用前端已加载列表做部分筛选，数据量很大时需要注意体验。
- 位置名称可能同时存在 code/name 两套来源，需优先展示业务名并避免冲突。
- “同步删 V1”若无后端能力，本轮只能停留在提示层，不能伪造交互。
