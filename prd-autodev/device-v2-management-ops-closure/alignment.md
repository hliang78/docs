# Alignment

## Proposed Direction

- 推荐仍走 `方向 C：分两阶段做闭环`，但范围改成：
- 第一阶段：`动作闭环 + 编辑体验`
- 第二阶段：`新增入口 / 分组可靠性 / 更深结果分析`
- 不再把“治理能力建设”作为这一页的主问题定义。

### 推荐的第一阶段范围

1. 围绕 `监控未下发` 设备补动作后复核能力
2. 采集/监控动作补上结果承接、失败原因与回流
3. 把采集 / 继续纳管失败收敛成“分层错误协同表达”，尤其把 `credential_ref_in_band`、`controller_detect_failed`、`no_data_collected` 这类线索前置
4. 旧页里可复用的任务结果、自动刷新、执行明细、采集证据能力做“挖矿式迁移”
5. 编辑弹窗重做宽度、布局、字段分层和下拉优先策略
6. 修正名称/编码展示优先级，尽量不用编码作为主阅读信息
7. 清理零值时间、无意义占位信息、技术化 hover 文案
8. 列表保持普通设备列表语义，不额外承担“下一步处理信号”
9. 明确并发采集 / 并发监控推送时的设备级状态表达、重复触发控制和结果归属

### 推荐的第二阶段范围

1. 手动新增/克隆等补录入口
2. 左侧全量分组可靠性升级
3. 更深的结果分析或采集数据查看入口
4. 需要时再评估更多批量能力

## Options

- 选项 1：先做“动作闭环”
  - 优点：和 `监控未下发` 场景直接对齐，旧页已有能力可迁。
  - 代价：编辑体验问题会继续拖住高频使用。
- 选项 2：先做“编辑体验”
  - 优点：直接改善高频操作，收益立刻可见。
  - 代价：监控/采集动作后的闭环仍然偏弱。
- 选项 3：把“动作闭环 + 编辑体验”作为同一批 PRD 的两条主线
  - 优点：最贴合用户当前反馈。
  - 代价：需要在 PRD 和 stories 上严格控范围，避免重新膨胀成旧页。

## User Decisions

- 已确认：
  - `监控未下发` 过滤出的设备，是动作闭环的核心目标群体。
  - 监控下发是重要但低频场景，成功要有显示，失败要知道原因，少数情况下才会看采集具体数据。
  - 旧页里批量、单设备采集、监控下发内容可以挖矿迁移。
  - 当前页问题不定义为“治理”。
  - 编辑页面是高频重点，需要更好的交互体验、更多下拉、弹窗更宽、重新组织组件布局。
  - 页面还有大量细节要调，尤其是阅读语义问题。
  - 尽量不用编码，编码对阅读理解是灾难。
  - 页面美化是基本功能完成后的重点，不应先于基础可用性和语义正确性。
  - 第一批暂不考虑新增设备。
  - 动作结果默认在右侧详情里做简明复核。
  - 采集必须提供“查看最后一次采集详细数据”的入口。
  - “最后一次采集详细数据”不放进右侧详情常驻区，使用独立查看层承载。
  - 当前专项实现文件以 `DeviceV2ManagementRedesign.vue` 为准。
  - 当前更重要的主题是采集、监控过程中的各种错误能被前后端协同地准确承接出来。
  - 对这类失败，前端要优先把用户带向“正式链路接入配置 / 凭据引用问题”这类更接近真实排查方向的线索，而不是笼统提示设备不可采。
  - 当前不强行把页面定义成“诊断系统”，更适合定义为“错误分层表达与排障协同”。
  - 左侧分组第一批只做名称语义和细节级整改，不进入结构性重做。
  - 旧页对并发采集和并发监控推送处理得也不够好，第一批 PRD 需要把这部分明确补强，而不是直接照搬旧交互。
