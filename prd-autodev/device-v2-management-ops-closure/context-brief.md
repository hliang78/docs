# Context Brief

## Goal

把 `Device V2` 设备管理正式页收敛成一个适合运维日常高频使用的设备管理页。它的核心不是“讲流程”，而是让运维能围绕设备快速完成 `定位 -> 判断 -> 编辑 -> 执行动作 -> 查看结果/继续处理` 的闭环。

本轮先完成闭环前的分析，输出可直接进入 PRD 的问题清单、场景优先级和候选方向。

## Boundaries

- 页面标题保持“设备管理”，不加副标题。
- 页面结构保持三段式：左侧设备分组、中间设备清单、右侧设备详情。
- 右侧详情默认不展开，只能通过操作列“详情”打开。
- 采集、监控是操作，不是主叙事。
- 页面仅保留“监控未下发 / 监控已下发”这条主状态视角，不回到旧工作台风格。
- 左侧分组配置继续使用“后端保存的全局唯一一套”，不改回本地状态。
- 第一批范围暂不考虑新增设备。
- 页面阅读尽量不用编码作为主展示，优先显示业务名称、业务语义和人可理解的文案；编码只作为次级信息或补充信息。
- 页面美化不是第一优先，基础功能和阅读语义正确之后，再重点打磨视觉和细节。
- 本轮先做分析与 PRD 准备，不直接生成自动化 stories。

## Focus

- 以运维工程师的真实设备管理场景评估当前页面。
- 第一优先分析“监控未下发”这批设备的动作闭环，但明确这是重要低频场景，不把整个页面做成围绕监控下发运转的专项页。
- 第二优先分析编辑弹窗，重点是减少填写负担、增加下拉选择、拉宽弹窗、重做组件布局。
- 识别当前页在功能设计和操作体验上的关键不足。
- 明确哪些不足已经有后端/前端能力储备但尚未接到当前页，哪些是后续需要新增能力的缺口。
- 重点识别采集 / 继续纳管失败时，哪些后端证据已存在但仍未被前端前置成“可定位线索”，以及哪些字段需要前后端协同稳定下来。
- 给出 1 到 3 个适合进入 PRD 的演进方向。

## Background

- 当前专项实现范围已收敛到 `DeviceV2ManagementRedesign.vue`，本轮不再把 `Grouped.vue` 作为主要实现目标。
- 当前页已经具备左侧分组、中心列表、右侧详情、批量采集/监控/删除、完整度较高的本页编辑弹窗。
- `grouping_config` 已落到 `device_v2_schema` 单例表，由后端提供 `GET/PUT /device/v2/grouping-config`。
- 旧页 `DeviceV2Management.vue` 仍保留较多围绕入库、任务、变更记录、证据的能力，可作为“借能力但不借叙事”的参考来源。

## Concerns

- 当前页已经摆脱旧工作台叙事，但离“运维连续做事”的闭环还有距离。
- 页面目前更像“可浏览、可改、可发动作”的管理列表，还不够像“可判断、可回溯、可批量治理”的正式运维设备管理页。
- 当前错误承接更偏“结果摘要”，对 `controller_detect_failed`、`no_data_collected`、`credential_ref_in_band` 这类排障锚点前置不足，容易把用户带到错误排查方向。
- 如果不先按 `前端 / 编排层 / Vault / Controller / DC2 / sync-to-v1 / monitor push / Agent / 目标设备` 这些真实参与方拆解责任面，后续前端再多加字段，也仍可能把上游问题说成下游问题。
- 用户已明确当前页问题不定义为“治理平台建设”，后续 PRD 不能再把方向写偏。
- 如果不先把场景优先级讲清楚，后续容易继续做局部美化，缺少 PRD 级闭环。

## Primary Users / Systems

- 一线运维工程师：定位设备、修正台账、补齐归属、发起采集、下发监控、判断失败原因。
- 二线平台/CMDB 运维：校验模型字段、补录和修正基础信息、处理低频但关键的动作异常。
- 相关系统：`device_v2` 实体、`device_v2_schema`、采集入库任务、监控下发、变更记录与采集证据接口。

## Known Constraints

- 当前仓库 `npm run typecheck` 慢，不把慢误判成当前改动错误。
- 页面中禁止再次引入“工作台、推进、采集验证、过程结果、待补前提、待推送任务、查看结果”等叙事词。
- 产品方向已确认，不再回退到旧页的复合型流程页。
- 正式页中类似 `0001-01-01 08:05` 这类无意义占位时间，不能作为正常信息继续暴露给用户。

## Evidence Needed

- 能否支持典型运维场景：快速找设备、动作后复核、定位失败原因、人工补录/修正、值班交接。
- 能否把技术字段翻译成人可读语言，避免页面大量出现“编码可见、语义不可读”的情况。
- 能否证明后续 PRD 方向是“提升运维闭环效率”，而不是重新做一个视觉更精致但工作流依然断裂的页面。

## Open Questions

- 编辑弹窗第一批是否需要把 `labels / metadata` 进一步做“推荐维护区 + 高级区”分层？

## Confirmed Decisions

- 第一批暂不考虑新增设备。
- 动作结果默认在右侧详情里做简明复核，不把右栏做成结果信息墙。
- 采集场景必须提供“查看最后一次采集详细数据”的入口，详细数据使用右栏之外的独立查看层承载。
- 采集 / 继续纳管失败时，前端第一优先要做“分层错误承接”，尤其要把正式链路凭据引用问题与设备 SSH 不可用区分开。
- 要覆盖这类错误承接，必须以采集、`sync-to-v1`、`monitor push`、`ensureOnboarding` 的后端全过程为依据，而不是只按前端现有摘要做文案修补。
- 错误承接的第一分层不再只是“采集 / 监控”，而要先回答“属于哪个参与方”，再回答“停在哪个阶段 / 方法”。
