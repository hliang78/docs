# Device V2 详情与抽屉字段方案

## 1. 目标

这份文档把上一轮交互方案继续收敛到字段级，回答三个问题：

1. 右侧详情到底显示哪些块
2. `最后一次采集详细数据` 抽屉到底展示哪些字段
3. `本次监控结果` 抽屉到底展示哪些字段

目标不是生成最终视觉稿，而是把：

- 字段范围
- 信息层级
- 数据来源
- 空态规则
- 是否默认展示

先定清楚，避免实现时边做边补。

## 2. 右侧详情方案

右侧详情仍然坚持一句话：

- `简明复核`

它只做“设备级结论承接”，不做大结果中心。

## 2.1 区块结构

建议固定为 6 块：

1. 头部概况
2. 基础信息
3. 网络与位置
4. 采集复核
5. 监控复核
6. 设备操作

## 2.2 头部概况

### 目标

让用户一眼确认：

- 我现在看的是什么设备
- 当前监控状态是什么
- 当前设备有没有明显异常

### 字段

1. 设备名称
2. 设备编码
3. 监控状态 tag
4. 简短状态说明

### 数据来源

1. `selectedDeviceView.name`
2. `selectedDeviceView.code`
3. `monitor_push_status / monitor_push_error / monitor_push`
4. 由前端基于设备当前状态计算

### 状态说明文案规则

只允许很短：

- `监控尚未下发`
- `监控推送失败`
- `最近采集有记录`
- `最近采集暂无记录`

不允许：

- 长段原因
- 下一步建议
- 技术化任务描述

## 2.3 基础信息

### 目标

保持设备管理页的常规可读性。

### 字段

1. 平台
2. 设备类别
3. 状态
4. 租户
5. 区域
6. 标签摘要

### 显示规则

- 名称优先
- 编码退后
- 没有名称时才回退编码
- 不在这里展示过程状态

## 2.4 网络与位置

### 字段

1. 主 IP
2. 机房
3. 机柜
4. 位置全路径
5. 最近更新时间

### 显示规则

- 更新时间清理零值时间
- 位置优先名称
- 路径可显示 `安全区 / 机房 / 机柜`

## 2.5 采集复核

这是右栏最关键的新块之一。

### 目标

回答：

- 最近有没有采集
- 最近采集结果是什么
- 如果出问题，大概为什么
- 是否可以进一步查看采集详细数据

### 建议字段

1. 最近采集结论
2. 最近采集时间
3. 采集状态
4. 原因摘要
5. 查看最后一次采集详细数据入口

### 数据来源优先级

1. 设备持久化 `process_result`
2. 设备已有简要采集字段
3. 若当前页内刚执行过采集，可用最近任务上下文补短时状态

### 字段说明

#### 最近采集结论

来源：

- `process_result.headline`

回退：

- `暂无采集记录`

示例：

- `采集验证已完成`
- `采集被阻塞`
- `采集执行失败`
- `暂无采集记录`

#### 最近采集时间

来源优先级：

1. 最近一次采集任务结果时间
2. `last_store_collection_at`
3. 空态

空态：

- `暂无记录`

#### 采集状态

来源：

- `process_result.overall_status`

映射：

- `success` -> `成功`
- `running` -> `进行中`
- `blocked` -> `被阻塞`
- `failed` -> `失败`
- `skipped` -> `已跳过`

#### 原因摘要

来源优先级：

1. `process_result.steps` 中非 success 步骤的 `user_message`
2. `operator_message`
3. 空

展示规则：

- 只显示首条最关键原因
- 不展示完整 steps 列表

示例：

- `缺少访问凭据`
- `缺少接入目标`
- `采集结果写入失败`

#### 查看最后一次采集详细数据

显示条件：

- `getDeviceV2LastStoreCollectionDc2Req` 返回 `found=true`

不显示条件：

- 没有最近采集记录

按钮文案：

- `查看最后一次采集详细数据`

## 2.6 监控复核

这是右栏另一个关键块。

### 目标

回答：

- 当前监控到底是什么状态
- 最近一次监控动作有没有失败
- 失败是同步失败还是推送失败
- 是否可以查看本次结果

### 建议字段

1. 当前监控状态
2. 最近一次监控结果
3. 原因摘要
4. 本次监控结果入口

### 数据来源优先级

1. 设备持久化 `monitor_push_status`
2. 设备持久化 `monitor_push_error`
3. 当前页最近一次监控动作返回上下文

### 字段说明

#### 当前监控状态

建议映射：

- `success` -> `监控已下发`
- `failed` -> `监控推送失败`
- `running / pending` -> `监控下发中`
- 空 -> `监控未下发`

#### 最近一次监控结果

这个字段只给一句结论：

- `监控推送成功`
- `V1 同步失败`
- `监控推送失败`
- `尚无推送记录`

### 原因摘要

来源优先级：

1. 本次监控返回上下文中的 item/message
2. `monitor_push_error`
3. 空

显示规则：

- 只展示最关键的一条
- 不展示设备列表级长表格

#### 本次监控结果入口

显示条件：

- 当前用户刚执行过监控动作，且本页还保留最近动作上下文

按钮文案：

- 单设备：`查看本次监控结果`
- 批量：`查看批量监控结果`

## 2.7 设备操作

### 按钮

1. 采集
2. 监控
3. 编辑

### 状态规则

- 采集中时禁用采集
- 监控推送中时禁用监控
- 详情按钮区不附加长说明

## 3. 最后一次采集详细数据抽屉

## 3.1 目标

提供排障必需的过程数据，但不污染主详情。

## 3.2 抽屉结构

建议固定为 6 块：

1. 顶部摘要
2. 过程结论
3. 设备采集证据
4. 判断事实
5. 后续可采集项
6. 深链接

## 3.3 顶部摘要

### 字段

1. 设备名称
2. 设备编码
3. task id
4. store run id
5. dc2 run id
6. overall status
7. headline

### 数据来源

- `last-store-collection-dc2`
- `task summary.process_result`

### 显示规则

- `dc2 run id` 可点击或作为深链接入口
- `overall status` 用 tag
- `headline` 作为抽屉主结论

## 3.4 过程结论

### 来源

- `process_result.steps`

### 展示方式

固定展示 3 行：

1. 前置准备
2. 执行采集
3. 写入结果

每行字段：

1. 步骤名
2. 状态
3. 用户提示
4. 下一动作标签

### 显示规则

- 默认折叠 operator message
- 失败或阻塞时展开用户提示
- 不展示原始 details JSON

## 3.5 设备采集证据

### 来源

- `layers.collection_execution`
- `last-store-collection-dc2`

### 字段

1. 是否执行 DC2
2. DC2 run id
3. 事实写入数
4. observation count
5. message

### 显示规则

- 这是摘要区，不放大表格
- 若无 DC2 证据，显示 `未发现最近一次 DC2 采集记录`

## 3.6 判断事实

### 来源

- `observations`

### 默认展示

1. 总数
2. capability 分布摘要
3. 关键字段样本 5 条

### 展开态

支持表格分页查看：

1. capability
2. field_key
3. status
4. source
5. collected_at
6. value 预览

### 禁止项

- 默认首屏直接展示全部 observation

## 3.7 后续可采集项

### 来源

- `collection plans`

### 默认展示

1. 总数
2. 当前可执行数
3. 第一条 next action

### 展开态

表格字段：

1. action
2. collection_key
3. availability_status
4. preferred_method
5. 原因摘要

## 3.8 深链接

### 入口

1. `查看 DC2 运行详情`
2. `查看更完整采集结果`

### 规则

- 只有存在可用 run id 时出现
- 当前页不重建 DC2 专业详情

## 4. 本次监控结果抽屉

## 4.1 目标

承接本次监控动作的批量或单设备结果，让用户能排查失败，但不把监控过程做成主页面常驻结构。

## 4.2 抽屉结构

建议固定为 3 块：

1. 顶部汇总
2. 失败归因
3. 设备结果列表

## 4.3 顶部汇总

### 来源

- `syncDeviceV2ToV1Req` 返回体

### 字段

1. 总数
2. created
3. updated
4. skipped
5. failed
6. monitor_push_status

### 显示规则

- 成功态：绿色摘要
- 有失败：黄色或红色摘要

## 4.4 失败归因

### 目标

明确分开：

1. `sync-to-v1` 失败
2. `monitor push` 失败

### 字段

1. 失败类型
2. 首条原因
3. 失败数量

### 显示规则

- 若只有 monitor push 失败，不要误写成同步失败
- 若两类都存在，分两块展示

## 4.5 设备结果列表

### 来源

- `resp.items`

### 字段

1. 设备编码
2. V1 编码
3. 同步结果
4. 监控结果
5. message

### 显示规则

- 单设备时仍用表格，但只一行
- 批量时支持按失败优先排序

### 交互

1. 一键筛选失败设备
2. 一键关闭并回到当前列表

## 5. 空态规则

## 5.1 右侧详情

### 采集复核空态

- `暂无采集记录`
- 不显示详细数据入口

### 监控复核空态

- `尚无监控推送记录`
- 不显示本次结果入口

## 5.2 采集详细抽屉

- 无最近采集记录时不允许打开
- 若 task 有值但明细不可读，显示：
  - `已找到最近采集任务，但暂无可展示的详细数据`

## 5.3 监控结果抽屉

- 只为本次动作服务
- 没有最近动作上下文时不出现入口

## 6. 字段来源映射

## 6.1 右侧采集复核

1. `process_result.headline`
2. `process_result.overall_status`
3. `process_result.steps`
4. `last-store-collection-dc2`

## 6.2 右侧监控复核

1. `monitor_push_status`
2. `monitor_push_error`
3. 最近一次 `sync-to-v1` 返回结果上下文

## 6.3 采集详细抽屉

1. `getDeviceV2LastStoreCollectionDc2Req`
2. `getDeviceV2StoreTaskSummaryReq`
3. `listDeviceV2StoreRunsReq`
4. `listDeviceV2StoreObservationsReq`
5. `listDeviceV2StoreCollectionPlansReq`

## 6.4 监控结果抽屉

1. `syncDeviceV2ToV1Req` 返回体
2. 当前页缓存的最近监控动作上下文

## 7. 第一批实现顺序建议

1. 右侧采集复核块
2. 右侧监控复核块
3. 最后一次采集详细数据抽屉
4. 本次监控结果抽屉
5. 并发按钮状态与动作上下文缓存

这个顺序的好处是：

- 先把右栏承接做好
- 再把下钻层补上
- 最后再收并发细节
