# 原始想法

工程文档标准化需要以 `docs/development-doc-templates/01-07` 为候选标准文档主结构，并通过深度阅读真实代码生成长期可维护的中文候选开发文档体系。

关键要求：

- 不希望一个 PRD 承担两个任务。
- `docs/prd-autodev/oneops-poc-concern-autotest` 主要服务于 `poc-ct`。
- 工程文档生成、文档状态门禁、人工确认机制，需要放在一个独立 PRD 目录中管理。
- AI/autodev 只能整理、提炼、占位、标风险，不能反推需求、设计、业务规则、架构、API、数据模型、验收标准。
- 未经人工补充/确认的标准工程文档，必须处于不可直接使用状态。
- 输出的工程文档产物必须以中文为主；必要的英文术语、字段名、状态值、代码标识可以保留英文。
