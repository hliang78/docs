# Platform 创建任务页面分析

## 范围

本分析只聚焦 `任务中心` 中的“创建任务”弹窗，不讨论：

- `template-debug-v2/v3`
- 模板创作工作台整体
- 任务列表本身
- 定时任务页面

对应页面文件：

- [TaskManagement.vue](/Users/huangliang/project/OneOPS-ALL/OneOPS-UI/src/views/platform/TaskManagement.vue)

## 结论先行

当前“创建任务”不是字段不够，也不是提示不够，而是**把多个层级的决策同时暴露给用户**。它实际上混合了：

1. 任务类型选择
2. 模板入口选择
3. 仓库与脚本来源选择
4. 目标范围选择
5. 参数配置
6. 执行环境配置
7. 高级继承 / 默认值 / 复用来源

这些决策都合理，但当前在一个长表单里同时展开，导致用户很难建立稳定心智。页面现在更像“功能控制台”，不是“创建任务流程”。如果目标是简洁、专业、容易使用，那么这里最需要的不是删字段，而是**重新组织决策顺序**。

## 当前真实代码事实

### 1. 创建任务入口很多

当前可进入“创建任务”的方式至少有 4 条：

1. 页面主按钮 `创建任务`
2. 推荐入口卡片 `直接创建`
3. 列表行内 `复用`
4. query 参数预填 `openCreate/templateId/variableSetId`

涉及逻辑：

- `openCreateModal`
- `startFromRecommendedTemplate`
- `reuseTask`
- `prefillCreateModalFromQuery`

这说明页面已经有“快速起步”的意识，但不同入口进入后，最终仍落到同一套长表单。

### 2. 当前弹窗承担了超过 20 个表单块

从模板结构看，当前弹窗可见或条件可见的块包括：

1. 快速预设
2. 操作提示
3. 云工具主线
4. 当前进度
5. 执行预览
6. 项目/分组
7. 区域
8. 应用类型
9. 示例入口
10. 仓库来源
11. 默认脚本仓
12. 仓库 URL
13. 分支
14. Shell 入口
15. 路径 / 工作目录 / Playbook 路径
16. Inventory
17. Extra Vars
18. Parameter Specs
19. Arguments
20. 执行位置
21. Agent 编码
22. 高级选项
23. 模板
24. 变量组
25. 仓库记录
26. 库存来源
27. 凭证

这还不包括每个块中的 alert、hint、link action、preview。

### 3. 当前页面已经在尝试“引导式创建”

这点其实很好，说明方向是对的。页面已经具备这些设计意图：

- `快速预设`
- `操作提示`
- `当前进度`
- `执行预览`
- `推荐区域`
- `推荐 Agent`
- `主线建议`
- `示例入口`

问题不在“没引导”，而在**引导是附着在长表单上的，不是表单结构本身的一部分**。

### 4. 当前页面把“高级能力”做成了默认主视图的一部分

例如：

- `Parameter Specs`
- `Extra Vars`
- `仓库来源`
- `Shell 入口`
- `执行位置`
- `Agent`

这些本身并不一定该隐藏，但它们在创建流程中的优先级并不相同。现在用户一打开弹窗，就同时面对“主路径决策”和“进阶配置能力”。

### 5. 当前页面强依赖即时 message 提示

代码里有大量：

- `message.info`
- `message.warning`
- `message.success`

这说明创建流程很依赖即时消息来修正用户路径。问题是：

- 消息是瞬时的
- 消息不能替代稳定结构
- 同样的提醒会在不同路径上重复出现

专业工具更理想的状态是：**结构先减少犯错，提示再补充说明**。

## 用户真正要做的决策

从产品视角，“创建任务”本质上只是在回答下面 5 个问题：

### 1. 我要跑什么

对应：

- 应用类型
- 模板
- 示例入口
- Playbook 路径 / 工作目录 / 命令

### 2. 我要对谁跑

对应：

- 项目/分组
- 区域
- Inventory
- 库存来源

### 3. 我需要带什么参数

对应：

- 变量组
- Extra Vars
- Parameter Specs
- Arguments

### 4. 我要在哪里跑

对应：

- Controller 本机 / Agent
- Agent 编码

### 5. 需要什么权限和上下文

对应：

- 仓库 URL / 分支
- 仓库记录
- 凭证

当前问题就在于，这 5 个问题没有被页面明确分段，而是穿插在一起。

## 当前页面为什么不够简洁

### 1. 决策顺序被打乱

当前实际顺序更像：

```text
先看预设
再看提示
再看主线
再看当前进度
再看执行预览
然后才开始填项目/区域/类型
中间再切仓库来源
再切 shell 模式
再补 inventory / vars / arguments / agent
最后还有高级选项
```

用户还没确定“我要跑什么”，就已经被要求理解“默认脚本仓 vs 自定义仓库 vs 无仓库”、“命令模式 vs 脚本路径模式”、“本机 vs Agent”。

### 2. 页面里有太多“说明层”

当前创建弹窗里有多层说明：

- alert
- hint
- link 按钮
- preview
- progress

它们单独看都合理，但叠在一起之后，页面会变得很吵。简洁不是少字段，而是少并列层级。

### 3. “模板入口”和“手工入口”没有被拉开

现在用户既可以：

- 从推荐模板直接进
- 从模板预填进
- 从复用任务进
- 完全手工创建

但这些入口最后仍然进入同一个视觉结构，没有显式告诉用户：

- 你现在是“模板驱动创建”
- 还是“手工脚本创建”

这会削弱专业感，因为工具没有显式表达当前工作模式。

## 当前页面为什么不够专业

### 1. 专业感没有落在“掌控感”上

真正的专业工具应该让用户在提交前非常清楚：

- 本次执行来源是什么
- 会跑哪类脚本
- 跑到哪里
- 带哪些关键参数
- 在哪里执行

当前虽然有 `执行预览`，但它仍然是表单过程中的一个辅助块，不是整个流程的骨架。

### 2. 高级能力和高频能力的视觉权重太接近

例如：

- 项目/区域/类型 是高频关键决策
- Parameter Specs / 仓库记录 / 凭证 是进阶配置

但当前它们在整体表单中并没有形成足够清楚的层级落差。

### 3. 复用能力强，但模式感弱

当前页面其实已经支持：

- 模板预填
- 任务复用
- 推荐入口
- 自动推荐区域 / Agent

这已经很接近专业工具了，但页面没有把这种“智能预填”和“主线创建”组织成清晰模式，所以用户感受到的是复杂，不是专业。

## 当前页面为什么不够容易使用

### 1. 用户必须自己判断当前该填哪些字段

虽然页面通过 `mainlineSummary` 在提示缺项，但从体验上，用户仍然要自己扫描整页，判断：

- 我现在到底该填哪几项
- 哪些先填，哪些可以后填
- 哪些只是高级选项

### 2. 页面提供了很多捷径，但捷径本身也增加了认知成本

例如：

- 快速预设
- 套用当前类型示例
- 示例入口
- 默认脚本仓
- 使用推荐区域
- 使用推荐 Agent

这些捷径说明设计很用心，但当它们同时出现时，用户会需要先理解这些捷径之间的关系。

### 3. 表单是“条件显示”的，但不是“阶段显示”的

当前已经有不少 `v-if`：

- ansible 才显示 inventory
- shell 才显示 shell 入口
- run_on_agent 才显示 agent

这降低了一部分噪音，但还不够。因为用户仍然面对的是一整个大表单，而不是一个逐步展开的任务流。

## 我建议的产品结构

## 目标

让“创建任务”变成一个稳定的 5 步流程，而不是一个带很多提示的长表单。

## 推荐结构

```text
Step 1 选择来源
Step 2 选择目标
Step 3 参数配置
Step 4 执行环境
Step 5 确认执行
```

### Step 1 选择来源

回答：我要跑什么

包含：

- 创建模式
  - 从模板创建
  - 手工脚本创建
  - 复用历史任务
- 应用类型
- 模板 / 示例入口
- 仓库来源
- 路径 / 命令 / 工作目录

设计重点：

- 明确“模板模式”和“手工模式”是两条不同主线
- 推荐入口只出现在这一步

### Step 2 选择目标

回答：我要对谁跑

包含：

- 项目/分组
- 区域
- Inventory / 库存来源

设计重点：

- 目标选择应该独立成一步
- 这里不夹杂参数和仓库说明

### Step 3 参数配置

回答：要带什么参数

包含：

- 变量组
- Extra Vars
- Arguments
- 高级参数
  - Parameter Specs

设计重点：

- 普通用户先看到“变量组 + Extra Vars + Arguments”
- `Parameter Specs` 放到高级项，不应默认占主视图

### Step 4 执行环境

回答：在哪里跑，需要什么权限

包含：

- Controller / Agent
- Agent 推荐
- 凭证
- 仓库记录

设计重点：

- 让执行位置和凭证在一个上下文里被理解
- 这里才出现 Agent 和凭证，不提前干扰前面步骤

### Step 5 确认执行

回答：本次到底会怎么跑

包含摘要：

- 来源
- 目标
- 路径 / 命令
- 参数摘要
- 执行位置
- 凭证摘要

设计重点：

- 这是专业感最强的一步
- 要把瞬时消息替换成稳定的执行摘要

## 页面布局建议

### 方案 A：Stepper Drawer

```text
[Drawer Title: 创建任务]
[Step Bar]
1 来源
2 目标
3 参数
4 执行环境
5 确认

[Main body]
当前步骤内容

[Bottom bar]
上一步 | 下一步 | 创建任务
```

适合：

- 真正降低认知负担
- 把现在的长表单彻底改成流程

### 方案 B：单页分段 + 固定右侧摘要

```text
[Left]
来源
目标
参数
执行环境

[Right]
当前执行摘要
当前缺项
推荐动作
```

适合：

- 不想引入明显 stepper
- 但仍想让结构更专业

我更推荐 **方案 A**，因为这页本质上就是一个顺序决策流程。

## 具体字段怎么重排

### 放前面的

- 创建模式
- 模板 / 类型 / 示例
- 路径 / 命令
- 项目 / 区域 / 目标

### 放后面的

- 仓库记录
- 凭证
- Agent
- Parameter Specs

### 默认折叠的

- Parameter Specs
- 仓库记录
- 高级 Inventory 覆盖
- 高级凭证配置

## 需要保留的优点

当前页面有一些很好的东西，不应该丢：

1. 推荐入口
2. 推荐区域
3. 推荐 Agent
4. 模板预填
5. 任务复用
6. 执行预览
7. 缺项提示

但这些能力应该服务于流程，而不是平铺在流程前面。

## 我的建议

如果我们下一步真的要设计“创建任务”页，我建议直接按这个顺序推进：

1. 先确认 `创建任务` 是否接受改成 stepper 式流程
2. 然后我给你出一版 **创建任务页面的低保真线框**
3. 再把当前字段映射到新的 5 步结构里

这样后面不管是做 UI 还是做代码拆分，都会很稳。
