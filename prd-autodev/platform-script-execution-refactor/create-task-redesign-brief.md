# 创建任务弹窗重设计分析

## 目标页面

- 页面实现：[TaskManagement.vue](/Users/huangliang/project/OneOPS-ALL/OneOPS-UI/src/views/platform/TaskManagement.vue)
- 当前对象：`创建任务` 弹窗

## 基于截图的直观判断

这个弹窗当前最明显的问题，不是“不完整”，而是**太像配置表单，不像任务创建流程**。

用户进入后，第一眼看到的是：

1. 快速预设
2. 两块蓝色提示卡
3. 项目 / 区域 / 应用类型
4. 仓库来源
5. Playbook / Vars / Parameter Specs / Arguments

也就是说，页面把“说明、状态、配置、进阶参数”几乎同权摆在一起。结果就是：

- 信息很多，但重点不够清楚
- 有专业术语，但不够专业利落
- 用户知道这里能做很多事，但不知道先做什么

## 具体问题

## 1. 页面开头太重，用户还没开始填，就先被两大块说明压住

截图里最占视觉面积的是两块浅蓝色 `Alert`：

- `操作提示`
- `当前进度`

它们的问题不是内容错，而是：

- 面积太大
- 语义太接近
- 都在重复表达“先选任务类型，再走主线字段”

结果：

- 页面开场像“培训说明”
- 不像“开始创建”

### 设计判断

这里应该只保留一个更短的“流程导航”区，不应该放两块大提示卡。

## 2. 缺少一个真正的“当前步骤感”

用户现在会看到很多字段，但不知道自己当前处于：

- 选来源
- 选目标
- 配参数
- 选执行环境

页面虽然写了“当前进度”，但它是文案，不是结构。

### 设计判断

创建任务页不该靠解释告诉用户“现在到哪一步了”，而应该让布局本身表达阶段。

## 3. 高优先级字段和低优先级字段混在一起

从截图看，用户高频真正要填的通常是：

- 项目/分组
- 区域
- 应用类型
- 仓库来源
- 路径 / 命令

但当前用户很快就会看到：

- Extra Vars
- Parameter Specs
- Arguments

尤其 `Parameter Specs` 这种字段，明显属于高级配置，不应该在主视图里这么早出现。

### 设计判断

主流程应该先帮助用户把任务“跑起来”，不是先把所有配置能力都摆出来。

## 4. 视觉层级不够稳，专业感被“说明块”稀释了

从截图里看，当前视觉上最强的是：

- 大提示卡
- 蓝色仓库来源按钮
- 蓝色“执行位置”标签

但真正应该最强的，其实应该是：

- 当前创建模式
- 当前关键必填项
- 提交前摘要

现在页面更像“有很多状态块”，而不是“一个干净的任务创建界面”。

### 设计判断

专业感应该来自：

- 克制
- 稳定
- 决策顺序清楚

不是来自更多提示框。

## 5. 表单密度不低，但节奏松散

问题不是“太挤”，而是“每段都像平铺”：

- 字段之间没有明确分组标题
- 当前块和下一块之间的逻辑边界不够强
- 每个字段都像独立项，而不是一个步骤中的一部分

### 设计判断

需要把单个字段组织成几个“任务分组”，而不是继续做一条长表单。

## 6. 当前页面不够“像执行工具”

一个好的执行工具在创建前，应该让用户很快看清：

- 这次跑的是什么
- 对谁跑
- 用什么方式跑
- 关键输入有哪些

而当前截图里的界面更像“你先慢慢配”，缺少一个清晰的执行摘要区。

### 设计判断

应该尽早出现执行摘要，不应该等用户脑内自己拼。

## 这页应该变成什么

我建议把它从：

```text
长表单 + 大量说明 + 部分预览
```

改成：

```text
任务创建流程 + 少量主线提示 + 固定执行摘要
```

## 推荐结构

## 方案：双栏分组式弹窗

因为这是弹窗，不一定非要上完整 Stepper。我更推荐先做一个**专业控制台式双栏布局**。

### 左侧：主创建流程

```text
[分组 1] 任务来源
- 快速预设
- 应用类型
- 模板 / 示例入口
- 仓库来源
- 路径 / 命令

[分组 2] 执行目标
- 项目/分组
- 区域
- Inventory / 库存来源

[分组 3] 参数
- Extra Vars
- Arguments
- 高级参数（折叠）

[分组 4] 执行环境
- Controller / Agent
- Agent 编码
- 凭证
```

### 右侧：固定摘要面板

```text
[任务摘要]
- 类型
- 来源
- 路径/命令
- 项目/区域
- 执行位置

[当前状态]
- 还缺什么
- 推荐下一步

[执行前提醒]
- 风险提示
- 提交说明
```

## 为什么我推荐双栏

因为它最适合当前这个页面的真实状态：

- 已经有很多配置项
- 又需要保留专业用户的效率
- 又不想一步步点太多次

Stepper 更强流程化，但在弹窗里会拉长交互路径。双栏结构可以先大幅提高清晰度，同时保留熟练用户速度。

## 页面应该删掉或弱化什么

## 1. 删掉两块重复提示卡

把：

- `操作提示`
- `当前进度`

合并成右侧摘要区里的一个小状态块：

```text
当前状态：还未选择任务类型
建议：先选择预设或应用类型
```

## 2. 弱化“执行位置”小蓝标签

它现在像一个漂浮状态，不够成体系。

应该把“执行位置”放进右侧摘要：

- 当前执行：Controller 本机

或者放进“执行环境”分组，而不是单独漂浮。

## 3. 折叠 `Parameter Specs`

这个字段太技术向，不适合常驻主视图。

建议变成：

- `高级参数`
  - 展开后出现 `Parameter Specs`

## 4. 把仓库来源做成更清晰的模式卡

当前“默认脚本仓 / 自定义仓库 / 不使用仓库”是按钮组，功能没问题，但理解成本略高。

建议做成 3 个轻量模式卡：

- 默认脚本仓
  适合快速验证
- 自定义仓库
  适合业务脚本
- 不使用仓库
  适合本机命令

这样更专业，也更容易扫读。

## 5. 把“快速预设”升级成“创建方式”

现在“快速预设”像一排工具按钮，不像页面主入口。

建议变成更明确的第一块：

```text
创建方式
[Shell 命令] [Ansible] [Terraform] [Tofu] [Terragrunt]
```

并在选中后，页面立即切换对应字段布局。

## 页面应该强化什么

## 1. 强化分组标题

不要只靠字段 label，应该明确出现：

- 任务来源
- 执行目标
- 参数配置
- 执行环境

这样页面马上就专业起来。

## 2. 强化摘要面板

摘要面板应该成为这页的核心专业感来源。

它应该一直告诉用户：

- 你正在创建的任务是什么
- 当前还差什么
- 最终会怎么执行

## 3. 强化“推荐动作”

例如：

- 当前未选应用类型：先选择预设
- 当前已选 shell 且无仓库：建议使用命令模式
- 当前是 ansible 且无 inventory：请补 Inventory 或选择库存来源

不要靠飘消息，尽量变成页面内稳定提示。

## 低保真线框

```text
[创建任务]

| 左侧主区                                            | 右侧摘要区                     |
|-----------------------------------------------------|--------------------------------|
| [任务来源]                                          | [任务摘要]                     |
| 创建方式: Shell / Ansible / Terraform / ...         | 类型: Shell                    |

| 路径/命令                                            | 路径: /                        |
|                                                     | 执行: Controller 本机          |
| [执行目标]                                          |                                |
| 项目/分组                                            | [当前状态]                     |
| 区域                                                 | 还缺: 项目/分组, 路径          |
| Inventory / 库存来源                                 | 建议: 先补任务来源和目标       |
```

## 视觉建议

### 色彩

- 用克制的产品化配色
- 不要大面积浅蓝提示块
- 只保留一种强调蓝用于主动作或当前模式

### 信息块

- 用轻分组，不要堆大 Alert
- 分组标题清楚，说明文字缩短

### 字体与节奏

- 标题更明确
- 组内间距收紧
- 组与组之间拉开

这样会更像专业控制台，而不是“说明型后台表单”。

## 优先级建议

如果真要把这个页面搞好，我建议优先做这 4 件事：

1. 把长表单改成 `左流程 + 右摘要`
2. 去掉重复大提示卡
3. 折叠 `Parameter Specs` 等高级项
4. 用分组标题重建页面层级
