# 创建任务弹窗线框与字段映射

## 目标

把当前“长表单 + 提示块”重构为一个更像专业执行工具的创建界面，同时尽量不打断已有能力：

- 保留快速预设、推荐入口、模板预填、复用、推荐 Agent、推荐区域
- 改善页面层级、节奏、主次关系
- 让用户更快完成第一次成功创建

## 第一设计原则

每一步操作以后，都要降低用户下一步的使用心智。

这条原则应该直接主导这个弹窗的结构和交互，而不是停留在说明文案层面。

### 这句话在页面里的真实含义

用户每完成一个动作后，界面不能只是“记录了一个值”，而应该同步完成下面至少一件事：

1. 自动隐藏已经不重要的选项
2. 自动补全高概率正确的默认值
3. 自动把注意力收敛到下一组必须决策的字段
4. 自动让右侧摘要更具体
5. 自动减少用户下一步需要理解的概念数量

### 反例

如果用户已经选了 `Shell`，页面还让他同时面对：

- Inventory
- 库存来源
- Parameter Specs
- Agent 复杂度
- 多套仓库逻辑

那就说明上一步操作并没有真正降低下一步心智。

### 正例

理想状态下：

- 选完 `Shell`
  下一步页面主要只剩：
  - 命令模式还是脚本路径模式
  - 是否需要仓库
  - 路径或命令本身
- 选完 `Ansible`
  下一步页面主要只剩：
  - Playbook 路径
  - Inventory / 库存来源
  - 必要参数
- 选完 `模板`
  下一步页面主要只剩：
  - 模板没覆盖的字段
  - 当前还缺哪些关键项
- 选完 `Controller 本机`
  下一步页面就不该继续让用户处理 Agent 的复杂度

## 围绕这条原则的交互规则

### 规则 1：每做一步，下一步只出现必要问题

例如：

- 选完创建方式，只强调“来源”和“目标”
- 填完目标，只强调“参数”
- 选完执行位置，只强调“执行环境剩余缺项”

### 规则 2：能推断的，不要让用户再做一次决定

例如：

- 选了推荐模板，自动带出类型、仓库、路径、默认参数
- 选了区域，自动推荐可用 Agent
- 选了 shell 命令模式，自动弱化脚本路径相关 UI

### 规则 3：当前模式之外的信息必须降噪

例如：

- 当前是 `Shell 命令`
  就不要让 `Inventory`、`库存来源`、`Parameter Specs` 抢主视图
- 当前是 `Controller 本机`
  就不要让 `Agent` 相关内容继续高亮

### 规则 4：右侧摘要必须随着每一步变得更确定

摘要区不是回显区，而是“降低下一步心智”的核心装置：

- 未选类型：类型未知
- 已选类型：类型明确
- 已选模板：来源明确
- 已填项目/区域：目标明确
- 已选执行位置：运行环境明确

用户会从这个收敛过程里感受到“越来越接近可执行”，这正是专业感的重要来源。

## 推荐方案

优先采用：

```text
双栏弹窗
左侧主流程
右侧固定摘要
```

原因：

- 比完整 Stepper 更容易直接落地到现有弹窗
- 不会让熟练用户多点很多次
- 能明显改善专业感和可扫读性

## 总体布局

```text
[Modal Header]
创建任务

[Body: 2-column]
---------------------------------------------------------
| Left: 主流程区                     | Right: 摘要区     |
|-----------------------------------|-------------------|
| 任务来源                           | 任务摘要          |
| 执行目标                           | 当前缺项          |
| 参数配置                           | 推荐下一步        |
| 执行环境                           | 执行前提醒        |
---------------------------------------------------------

[Footer]
取消 | 创建任务
```

## 区块结构

## 左侧主流程区

### A. 任务来源

回答：我要跑什么

包含：

- 创建方式
  - Shell
  - Ansible
  - Terraform
  - Tofu
  - Terragrunt
- 示例入口
- 模板来源
  - 从模板创建
  - 手工配置
- 仓库来源
  - 默认脚本仓
  - 自定义仓库
  - 不使用仓库
- 路径 / 命令 / 工作目录

### B. 执行目标

回答：我要对谁跑

包含：

- 项目/分组
- 区域
- Inventory 或 库存来源

### C. 参数配置

回答：我需要带什么参数

包含：

- 变量组
- Extra Vars
- Arguments
- 高级参数（折叠）
  - Parameter Specs

### D. 执行环境

回答：在哪里跑，用什么上下文跑

包含：

- 执行位置
  - Controller 本机
  - 指定 Agent
- Agent 编码
- 凭证
- 仓库记录

## 右侧固定摘要区

### 1. 任务摘要

始终显示：

- 类型
- 模板 / 手工模式
- 仓库模式
- 路径 / 命令
- 项目/区域
- 执行位置

### 2. 当前缺项

基于已有 `mainlineMissingItems` 逻辑，但换成稳定展示，不用大 alert：

例如：

```text
还缺：
- 项目/分组
- 应用类型
- Playbook 路径
```

### 3. 推荐下一步

根据当前模式给一条最重要建议：

- 未选类型：先选创建方式
- shell + 无仓库：建议命令模式
- ansible + 无 inventory：补 Inventory 或库存来源
- agent 模式 + 无 agent：先选 Agent

### 4. 执行前提醒

短文本，最多 2 条：

- 默认脚本仓适合快速验证
- 高级参数仅在脚本真实消费时声明

## 当前字段到新布局的映射

## 任务来源

### 保留并前置

- `快速预设`
- `应用类型`
- `示例入口`
- `从模板创建`
- `仓库来源`
- `默认脚本仓`
- `仓库 URL`
- `分支`
- `Shell 入口`
- `Playbook 路径`

### 设计处理

- `快速预设` 改名为 `创建方式`
- `从模板创建` 从高级选项提升到主流程区
- `仓库 URL / 分支` 仅在自定义仓库模式下展示
- `Shell 入口` 保留，但视觉上并入“路径 / 命令模式”

## 执行目标

### 保留并集中

- `项目/分组`
- `区域`
- `Inventory`
- `库存来源`

### 设计处理

- `区域提示` 保留，但缩成字段下方一行提示
- `使用推荐区域` 保留为小链接动作
- `Inventory` 与 `库存来源` 合并到同一组里，不要散在页面不同位置

## 参数配置

### 保留

- `变量组`
- `Extra Vars (JSON)`
- `Arguments`
- `Parameter Specs`

### 设计处理

- `变量组` 从高级选项提升到主流程区
- `Parameter Specs` 放入折叠项
- `生效变量预览` 保留，但缩成 `查看预览` 或小型代码块

## 执行环境

### 保留

- `执行位置`
- `Agent 编码`
- `凭证`
- `仓库记录`

### 设计处理

- `执行位置: Controller 本机` 不再做漂浮 tag
- `Agent` 推荐逻辑保留
- `凭证` 与 `仓库记录` 收拢在执行环境组，不再藏得太深

## 删除 / 合并 / 弱化建议

## 1. 合并 `操作提示` 和 `当前进度`

当前：

- 两块大 alert

改为：

- 右侧 `当前缺项`
- 右侧 `推荐下一步`

## 2. 弱化 `执行预览`

当前：

- 一个独立块，带 alert 和 grid

改为：

- 右侧 `任务摘要`
- 摘要用简洁键值展示

## 3. 弱化 `云工具主线`

当前：

- 一个完整独立块

改为：

- 仅在 `terraform / tofu / terragrunt` 模式下，在“任务来源”组内显示一个轻量模式提示条

## 推荐文案重写

## 顶部

当前：

- `快速预设`

建议：

- `创建方式`

## 任务来源组说明

当前：

- 多段解释

建议：

- `先确定本次运行的脚本类型和来源。`

## 执行目标组说明

建议：

- `选择本次任务作用的项目、区域和目标来源。`

## 参数配置组说明

建议：

- `只填写本次执行真正需要的参数。`

## 执行环境组说明

建议：

- `确认任务在哪里执行，以及是否需要额外凭证。`

## 交互规则

## 1. 创建方式必须先选

未选创建方式前：

- 后续区块可以展示，但弱化
- 右侧提示：`先选择创建方式`
- 不要让用户同时理解完整表单

## 2. 模板模式与手工模式明确区分

若选择模板：

- 自动回填默认配置
- 右侧摘要显示：`来源: 模板 <name>`

若手工模式：

- 右侧摘要显示：`来源: 手工配置`

## 3. 高级参数不默认展开

默认只展示：

- 变量组
- Extra Vars
- Arguments

用户点击“高级参数”后再展开：

- Parameter Specs

## 4. 提交按钮应带状态感

### 正常

- `创建任务`

### 缺项时

- 按钮可点击或不可点击二选一
- 更建议：
  - 可点击，但点击时聚焦第一个缺项

## 状态设计

## 空白初始态

右侧摘要显示：

```text
任务摘要
- 类型：未选择
- 来源：未选择
- 目标：未填写
- 执行：Controller 本机

推荐下一步
先选择创建方式
```

## 已选 Shell 命令态

右侧摘要显示：

```text
类型：Shell
来源：不使用仓库
命令：echo hello
目标：project-x / DefaultArea
执行：Controller 本机
```

## 已选 Ansible 态但缺 Inventory

右侧摘要显示：

```text
当前缺项
- Inventory

推荐下一步
填写 Inventory，或选择库存来源
```

## 对前端实现最重要的拆分建议

即使先不改交互模型，也建议先把 UI 拆成这些组件：

1. `TaskCreateSourceSection`
2. `TaskCreateTargetSection`
3. `TaskCreateParamsSection`
4. `TaskCreateRuntimeSection`
5. `TaskCreateSummaryPanel`

这样即使第一版只是“单栏改双栏”，后续也能继续演进。
