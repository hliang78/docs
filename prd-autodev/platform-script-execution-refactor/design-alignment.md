# Platform 脚本执行前端重构 - Design Alignment

## 问题总结

当前 `platform` 的脚本执行前端，不是“功能不够多”，而是“功能组织方式不够产品化”。用户要在 `脚本模板工作台`、`任务中心`、`任务模板管理`、`定时任务管理` 之间来回切换，自己理解“哪里负责生成、哪里负责沉淀、哪里负责执行、哪里负责周期化”，同时还要在多个页面重复面对相似的执行配置字段。这导致页面虽然能力完整，但整体感受不够简洁，不够专业，也不够顺手。核心问题不是视觉不精致，而是信息架构没有把复杂能力组织成清晰的工作流。

## 目标用户与使用场景

### 主要用户

1. 平台运维 / 自动化工程师
2. 编写脚本模板的人
3. 日常执行任务的人
4. 配置周期任务的人

### 他们真正想完成的事

1. 快速找到正确入口
2. 少填重复字段
3. 明白当前这一步是在“写模板”还是“跑任务”
4. 能放心执行，知道会跑到哪里、用什么参数、结果在哪看

## 关键观察

### 1. 入口按“系统能力”划分，不按“用户任务”划分

现状里：

- `脚本模板工作台` 承担“生成 + 编辑 + 测试 + 发布”
- `任务中心` 承担“推荐入口 + 列表 + 详情 + 创建任务”
- `任务模板管理` 承担“模板目录 + 模板配置”
- `定时任务管理` 承担“周期配置 + 执行配置覆盖 + 最近运行”

从研发视角这是合理的，但从用户视角，第一反应往往是：

- 我要跑个脚本，去哪里？
- 我要改模板，去哪里？
- 我要基于模板临时跑一次，和定时跑一次，差别在哪里？

也就是说，页面命名和边界是“系统结构”，不是“用户意图”。

### 2. 高价值操作没有被突出，低频复杂字段暴露过早

以 `TaskManagement.vue` 和 `ScriptTemplateStudio.vue` 为例，页面一开始就暴露大量字段：

- 仓库
- 分支
- Inventory
- Extra Vars
- Parameter Specs
- Agent
- 凭证
- 提权策略

这些字段对高级用户很重要，但不是每次操作都需要先看见。现在的问题不是字段太多，而是没有“按决策顺序出现”。

理想顺序应该是：

1. 你要做什么
2. 用哪个模板 / 脚本
3. 跑到哪一组目标
4. 需要哪些参数
5. 在哪里执行
6. 最终确认

而不是一开始就把所有可配置面都平铺给用户。

### 3. 同一个“执行配置”在不同页面里语义不稳定

同样的执行概念在多个页面重复出现：

- `inventory_content`
- `inventory_grouping_selection_set_id`
- `repo_url / repo_branch`
- `extra_vars_json`
- `parameter_specs_json`
- `run_on_agent / agent_code`
- `credential_ref`

但它们在不同页面里的角色并不一致：

- 在 `工作台` 里，它们是“生成 / 测试 / 发布上下文”
- 在 `任务模板` 里，它们是“默认执行配置”
- 在 `任务中心` 里，它们是“本次执行配置”
- 在 `定时任务` 里，它们又变成“模板基础上的周期覆盖”

这会让用户产生疑问：

- 这个字段我是现在填一次，还是以后都生效？
- 我改的是模板默认值，还是这次运行值？
- 我是在定义能力，还是在定义实例？

专业工具最怕“同名字段，不同语义”。

### 4. 页面里混合了“创作流”和“运行流”

`ScriptTemplateStudio.vue` 现在把下面几件事放在一起：

1. 创建会话
2. 和 LLM 对话
3. 编辑草稿
4. 做本地校验
5. 做任务执行测试
6. 发布成模板

这对于懂系统的人很强，但对于多数使用者来说，心智切换很大。它同时像：

- IDE
- AI Copilot
- 草稿管理器
- 测试面板
- 发布面板

结果就是每个区块都成立，但整页缺少“当前阶段”的强引导。

### 5. `任务中心` 更像后台集合页，不像执行中心

`TaskManagement.vue` 当前同时有：

- 主线推荐入口
- 筛选
- 表格
- 日志详情
- 创建任务弹窗

这让它像“所有东西都在这里”，但并没有形成一个特别稳定的主任务体验。对于日常执行用户，更理想的体验应该是：

- 先看到“最常执行的入口”
- 再看到“最近任务和状态”
- 创建任务时按步骤完成
- 结果页有清晰的状态与日志反馈

现在则是“列表页 + 创建器 + 详情页”叠在同一处。

### 6. 当前页面专业感更多来自业务复杂度，不来自交互秩序

页面里有很多专业概念和实用提示，这说明业务深度已经够了。但专业 UI 不只是“字段多、术语多、能力强”，更重要的是：

- 层级清晰
- 决策路径清晰
- 默认安全
- 关键动作明确
- 高级能力有边界地展开

现在的页面更像“把能力诚实地摆出来”，还没有把这些能力整理成一套让人有掌控感的专业工具界面。

## 组件级观察

### `脚本模板工作台`

涉及页面：[ScriptTemplateStudio.vue](/Users/huangliang/project/OneOPS-ALL/OneOPS-UI/src/views/platform/ScriptTemplateStudio.vue)

问题：

1. 页面纵向太长，用户很难感知“我现在在哪个阶段”
2. 会话、对话、草稿、测试、发布都很重要，但主次不分
3. 测试和发布都像“第二主流程”，容易互相争注意力
4. 参数协议展示是有价值的，但当前更像说明区，不像执行前检查区

建议定位：

- 让它成为“模板创作工作台”，不是“全链路万能页”
- 主路径只保留：`写 -> 校验 -> 发布`
- 任务执行测试保留，但降为“验证方式”之一，不和主编辑流抢主导

### `任务中心`

涉及页面：[TaskManagement.vue](/Users/huangliang/project/OneOPS-ALL/OneOPS-UI/src/views/platform/TaskManagement.vue)

问题：

1. 页面首页没有明确回答“你现在可以做什么”
2. 创建任务弹窗字段很多，缺少阶段式组织
3. 日志抽屉和列表切换虽强，但整体节奏偏后台表管理
4. 推荐入口是对的，但它和正式创建流程之间的连接不够像产品主线

建议定位：

- 让它成为“执行中心”
- 首屏优先展示：
  - 常用执行入口
  - 最近运行状态
  - 一键新建任务
- 创建任务改成分组或 stepper，而不是长表单

### `任务模板管理`

涉及页面：[TaskTemplateManagement.vue](/Users/huangliang/project/OneOPS-ALL/OneOPS-UI/src/views/platform/TaskTemplateManagement.vue)

问题：

1. 现在更像“模板字段编辑器”，不像“模板目录”
2. 模板的价值没有被强调，例如适用场景、推荐入口、关联变量组、执行方式
3. 用户在这里容易继续陷入配置细节，而不是先判断“我该用哪个模板”

建议定位：

- 让它成为“模板目录 + 默认配置管理”
- 首先帮助用户选模板，其次才是改模板细节

### `定时任务管理`

涉及页面：[ScheduledTaskManagement.vue](/Users/huangliang/project/OneOPS-ALL/OneOPS-UI/src/views/platform/ScheduledTaskManagement.vue)

问题：

1. 它现在承接了太多模板执行字段，导致“定时”这个主题不够突出
2. 用户会把它理解成“又一个创建任务页”
3. 实际上定时任务最关键的是：
   - 基于哪个模板
   - 对哪个项目 / 区域生效
   - 何时运行
   - 是否启用
   - 最近有没有成功

建议定位：

- 让它成为“调度页”
- 默认只显示周期运行必要信息
- 高级覆盖字段折叠到“高级覆盖配置”

## 设计方向

### 方向 1：保守优化

做法：

- 不改页面边界
- 只优化布局层级、表单分组、帮助文案、按钮层级
- 把高级字段折叠起来

优点：

- 风险最低
- 改造成本可控

缺点：

- 页面定位问题不会根治
- 还是会保留“能力堆叠”的感受

适合：

- 想先快速止血

### 方向 2：工作流优化

做法：

- 保留 4 个页面，但重新定义每页角色
- 按“创作 / 沉淀 / 执行 / 调度”重排页面职责
- 长表单改为分段式流程

优点：

- 最符合“简洁、专业、容易使用”
- 能明显改善用户路径

缺点：

- 需要对现有页面结构做较多调整

适合：

- 这次确实想把产品体验做上一个台阶

### 方向 3：执行中心优先

做法：

- 先重点改 `任务中心`
- 把它做成最强主入口
- 其他页面暂时只做配合式整理

优点：

- 对日常使用影响最大
- 能最快体现“更易用”

缺点：

- 工作台和模板页的问题仍会保留一部分

适合：

- 当前最痛的是“执行不好用”，而不是“创作链路不好用”

## 我推荐的方向

我推荐 **方向 2：工作流优化**。

原因很简单：你的目标不是“页面看起来更漂亮”，而是“页面更简洁、专业、容易使用”。这类目标最依赖信息架构，不先改页面角色和操作顺序，只做样式优化，效果会很有限。

## 推荐信息架构

### 1. 脚本模板工作台

建议主结构：

```text
[Header: 模板创作 / 当前会话 / 主动作]
[Step rail: 1 需求描述 | 2 草稿编辑 | 3 校验 | 4 发布]
[Main stage]
  - 当前步骤主内容
[Right aside]
  - 参数协议 / 草稿版本 / 发布注意事项
```

原则：

- 一次只强调一个阶段
- 历史信息和辅助说明进入侧边区
- 测试结果和发布结果做成独立状态块

### 2. 任务中心

建议主结构：

```text
[Header: 执行中心 / 新建任务]
[Top strip: 常用模板入口]
[Main]
  [Left: 最近任务 / 筛选列表]
  [Right or Drawer: 任务详情 / 日志]
[Create drawer]
  1 选择模板或脚本
  2 选择目标范围
  3 填参数
  4 选择执行位置与凭证
  5 执行前确认
```

原则：

- “创建任务”必须是一个顺序化决策过程
- 日志详情不应干扰列表主视图
- 推荐入口应真正变成高频起点

### 3. 任务模板管理

建议主结构：

```text
[Header: 模板目录 / 新建模板]
[Filter: 类型 / 场景 / 搜索]
[Template list]
  - 名称
  - 用途说明
  - 适用类型
  - 默认执行方式
  - 关联变量组 / 仓库
[Template editor]
  - 默认配置
  - 高级默认项
```

原则：

- 先帮助用户选模板
- 再进入默认配置编辑

### 4. 定时任务管理

建议主结构：

```text
[Header: 调度中心 / 新建计划]
[Main list]
  - 名称
  - 模板
  - 项目 / 区域
  - Cron
  - 启用状态
  - 最近运行
[Schedule editor]
  - 基础信息
  - 时间策略
  - 覆盖配置(折叠)
```

原则：

- 把“调度”放到页面中心
- 执行细节默认继承模板，只在必要时覆盖

## 交互原则建议

### 1. 默认展示基础模式，高级配置渐进展开

高级字段放入：

- `高级参数`
- `执行环境`
- `凭证与提权`
- `调试信息`

而不是默认全展开。

### 2. 重要动作前给“执行摘要”

在任务执行、模板发布、定时任务保存前，统一展示：

- 模板 / 脚本
- 目标范围
- 参数摘要
- 执行位置
- 凭证方式

这会显著增强专业感和可控感。

### 3. 页面术语统一

要严格区分：

- 模板默认值
- 本次执行值
- 周期覆盖值
- 生成上下文

同一个字段不要在不同页面里隐含不同语义而不提示。

### 4. 结果反馈做成“状态块”，不是消息弹窗为主

当前很多动作成功后依赖 `message.success`。建议更重视页面内状态区：

- 已创建任务
- 已通过校验
- 已生成发布记录
- 最近一次运行成功 / 失败

专业工具应该让结果留在界面里，而不是只飘一下。

## 按钮 / 交互验证矩阵

### `脚本模板工作台`

- 创建会话：应作为唯一首要入口
- 打开最近会话：应归入“继续工作”
- 发送对话：应挂在“生成 / 修改草稿”动作下
- 保存草稿：应区分“覆盖当前”与“另存版本”
- 测试：应明确是“校验”还是“真实执行测试”
- 发布：应明确是“更新模板”还是“新增模板”

### `任务中心`

- 直接创建：应是主按钮
- 推荐入口：应一键预填
- 复用任务：应明确复用范围
- 停止任务：应在运行态下高可见，但不抢主动作
- 日志：应作为详情视图，而不是列表主流程的一部分

### `任务模板管理`

- 创建任务：应是模板卡 / 行内最核心动作
- 新建定时：应是次核心动作
- 编辑模板：应进入默认配置编辑，不应抢主入口

### `定时任务管理`

- 新建计划：应是主按钮
- 启用 / 禁用：应直接可操作
- 最近运行：应能快速跳到任务结果
