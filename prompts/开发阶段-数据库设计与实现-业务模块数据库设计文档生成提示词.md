# 设计原则
## 1. 规范化原则
遵循数据库设计第三范式（3NF） 作为基准，减少数据冗余
在特定场景下（如频繁查询、统计分析）可适当反范式化，但需在文档中注明原因
每张表应有主键，建议UUID或雪花ID作为主键

## 2. 命名规范
表名：使用小写字母和下划线，采用单数形式（如 user、project、order_item）
字段名：使用小写字母和下划线（如 user_name、create_time、status）
主键：统一命名为 id
外键：命名为 {关联表名}_id（如 user_id、project_id）
时间字段：create_time、update_time（datetime类型）golang需要使用create_at、update_at、delete_at
状态字段：status（string类型，配合注释说明状态含义）

## 3. 完整性约束
明确每个字段的数据类型、长度、是否为空、默认值
标注必要的唯一约束、外键约束（逻辑外键或物理外键根据场景选择）
提供字段注释，清晰说明字段的业务含义

## 4. 扩展性考虑
预留必要的扩展字段（如 ext_info JSON类型存储扩展信息）
考虑未来可能的业务变化，避免过度设计

# 分析要求
1、全面提取字段信息：从原型界面中提取表单字段、列表展示字段、详情展示字段；从需求文档中提取业务属性；确保所有必要的数据字段都被涵盖。

2、合理拆分表结构：根据实体关系合理拆分表，避免单表字段过多（建议不超过30个字段），多对多关系应使用中间表。

3、字段类型选择合理：
文本类：根据长度选择 varchar、text、longtext
数值类：int（整数）、decimal（金额/精度要求高的数值）
时间类：datetime（业务时间）、timestamp（系统时间）
状态类：string（配合注释说明）

4、软删除设计：关键业务表建议增加delete_time或delete_at字段，支持软删除。
5、审计字段：每张业务表应包含 create_time或create_at，update_time或update_at，必要时增加 create_by、update_by。
6、不确定性处理：对于无法确定的数据类型、长度或业务关系，基于合理推断进行补充，并用[待确认]标记。
