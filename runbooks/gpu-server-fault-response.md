# GPU 服务器故障响应 Runbook

Date: 2026-06-21

## 目的

这份 runbook 用于 OneOPS 场景下的 GPU/NPU 服务器故障响应。它把原始运维手册里的硬件检查、系统排障、故障取证和事件闭环整理成一条适合值班同学执行的标准路径。

## 适用范围

适用机型：

- `4090`
- `A800`
- `A800二期扩容`
- `A10`
- `H100`
- `910B`

适用症状：

- GPU/NPU 掉卡或不在位
- GPU ECC / XID / 驱动异常
- 服务器宕机、自动重启、无法拉起业务
- 风扇、电源、内存、CPU、硬盘、RAID 硬件异常
- IB/ROCE/存储网络异常

## 响应原则

### OneOPS 主线优先

优先从 OneOPS 能力进入：

- 告警管理：确认告警是否已触发、是否存在同批次异常
- 设备监控 / 监控面板：看设备状态、资源异常和趋势
- 日志中心：收集系统与任务日志
- 设备管理：确认设备编码、机型、位置、责任域
- 统一凭证：获取登录和执行所需凭证
- 任务模板 / 变量集 / 任务中心：执行标准采集命令
- 操作审计：补足操作留痕

以下方式作为补充或兜底：

- Zabbix 或遗留监控页面
- 带外管理界面
- 跳板机 SSH
- 厂商日志导出工具

### 先恢复，再闭环，但不要跳过证据

允许先做最小恢复动作，例如冷重启或重试，但不能跳过下面三件事：

1. 记录故障对象
2. 采集关键证据
3. 要求业务侧验证

## 标准流程

### 1. 确认影响和对象

先确认：

- 哪台设备异常
- 是单机还是批量
- 是业务上报、平台告警，还是巡检发现
- 当前影响是“故障”还是“不影响业务的事件”

至少记录这些字段：

- 发生时间
- 设备名称 / 编码
- 机型
- 所属机房 / 机柜 / 项目
- 业务影响描述

如果 OneOPS 里已有设备详情、分组、拓扑、区域信息，优先从平台取，不要人工重复抄录。

### 2. 先做快速分桶

把问题先归到下面一个主类：

- `A. GPU/NPU 在位异常`
- `B. GPU ECC / XID / 驱动异常`
- `C. 宕机 / 自动重启 / 无法开机`
- `D. 风扇 / 电源 / 散热异常`
- `E. 内存 / CPU / 硬盘 / RAID 异常`
- `F. IB / ROCE / 网卡名 / 存储挂载异常`

如果 5 分钟内还不能分桶，先按 `C` 和 `F` 两类排查，因为这两类最容易同时拖垮业务恢复。

### 3. 采集证据

#### 3.1 平台证据

从 OneOPS 收集：

- 告警截图或告警详情
- 设备监控指标异常点
- 日志中心相关日志片段
- 最近操作审计
- 最近任务执行记录

#### 3.2 带外证据

带外重点看：

- 服务器 SN
- GPU/NPU 状态
- 风扇、电源、CPU、内存、硬盘、RAID 状态
- 一键导出的黑盒/系统日志

#### 3.3 系统证据

系统侧至少确认：

```bash
nvidia-smi
lspci
lsblk
df -h
ip addr
ibdev2netdev
systemctl status nvidia-fabricmanager
lsmod | grep nvidia_peermem
```

如果需要补充 GPU 细节，可再看：

```bash
nvidia-smi -L
nvidia-smi -q
nvidia-smi --query-gpu=persistence_mode --format=csv
```

记录原则：

- 能截图就截图
- 能导出日志就导出
- 能通过任务中心复用标准采集模板，就不要人工散打命令

## 分类处置

### A. GPU/NPU 在位异常

典型症状：

- `nvidia-smi` 卡数少于应有值
- `nvidia-smi` 报错，驱动无法访问设备
- 910B 场景下 NPU 不在位

首轮动作：

1. 核对业务影响和当前可否停机
2. 通过系统和带外确认具体掉卡位置
3. 采集 GPU/NPU、设备 SN 和日志证据
4. 获得业务许可后执行冷重启

升级条件：

- 冷重启后仍不恢复
- 同一台设备重复掉卡
- 带外或系统已能明确定位为部件异常

升级路径：

- 英伟达系：进入换卡 / 对调 / 底板排查
- 910B：冷重启无效时优先评估整机替换

### B. GPU ECC / XID / 驱动异常

典型症状：

- ECC 告警
- `nvidia-smi` 显示异常
- Driver / library mismatch
- 驱动与硬件无法通信

先查：

- `nvidia-fabricmanager` 是否启动
- `nvidia_peermem` 是否在位
- OFED 和驱动版本是否匹配
- 最近是否有驱动、内核、CUDA 变更

处置优先级：

1. 先排除服务未起、模块缺失、版本不匹配
2. 再判断是否是硬件故障
3. 如果重复出现 ECC / XID 且影响业务，倾向换卡而不是无限重试

### C. 宕机 / 自动重启 / 无法开机

典型症状：

- OneOPS 告警显示主机离线
- 业务无法连接
- 带外可见系统重启、掉电或启动异常

首轮动作：

1. 看带外是否在线
2. 确认是否有电源、风扇、主板、RAID、系统引导类异常
3. 冷重启一次
4. 系统恢复后立刻补采集启动前后日志

如果出现以下情况，直接升级：

- 重启后进入救援/紧急模式
- RAID / 启动盘异常
- 同机短时间内多次自动重启

### D. 风扇 / 电源 / 散热异常

典型症状：

- 带外显示风扇转速异常或为 0
- 电源功率异常
- 温度过高导致 GPU 掉卡

首轮动作：

1. 在带外确认部件状态
2. 截图并保留 SN/部件位置
3. 评估是否影响继续运行
4. 联系厂商或备件更换

如果只是恢复后观察，不代表闭环。必须补一轮业务侧验证和复测。

### E. 内存 / CPU / 硬盘 / RAID 异常

高频现实：

- 内存故障通常直接走更换
- CPU 故障和硬盘故障也以换件为主
- RAID 问题要先区分“卡故障、盘故障、误删配置、系统盘偏移”

处置要求：

1. 明确故障件位置和证据
2. 判断是否需要停机窗口
3. 修复后做最小业务验证
4. 必要时补磁盘、RAID 和启动链路校验

### F. IB / ROCE / 网卡名 / 存储挂载异常

典型症状：

- 业务报 NCCL 异常
- `ibdev2netdev` 对应关系错误
- 存储目录无法挂载
- 服务器重启后网卡名变化

先查：

```bash
ibdev2netdev
ip addr
mount | grep nvfile-data
```

高频修复动作：

- 执行标准检查脚本
- 必要时执行网卡名修复脚本
- 修复 bond / 网卡对应关系
- 检查 OFED 后重新验证

对于 A800 系列，这类问题在事件表里属于高频复发项，修复后一定要做业务侧验证。

## 修复后验证

### 基础验证

至少完成：

- 设备恢复在线
- 告警收口或转为已知观察
- `nvidia-smi` / `lspci` / 网络状态恢复正常
- 关键挂载或业务依赖恢复

### 业务验证

要求业务侧明确反馈：

- 是否已恢复训练/推理/作业运行
- 是否还有 NCCL、驱动、掉卡、性能异常

没有业务验证时，不要把状态写成“完全闭环”，最多写“技术恢复，待业务确认”。

### 深度验证

按场景补测：

- NCCL test
- GPU burn / 模型压测
- CPU / 内存测试
- DCGM 检测

## 记录和闭环

至少补齐这些字段：

- 故障发现时间
- 响应时间
- 定位时间
- 临时方案时间
- 解决时间
- 通知业务验证时间
- 业务反馈恢复时间
- 原因分析
- 解决方案

建议在 OneOPS 里统一归档为：

- 事件记录
- 相关设备
- 告警链接
- 日志证据
- 操作审计
- 最终知识条目

## 升级条件

满足以下任一条件，建议直接升级厂商或进入更高等级处理：

- 同机重复掉卡 / 重复宕机
- 冷重启无效
- 带外或系统证据指向硬件故障
- 已影响多台设备或同一业务批量作业
- 需要换件、整机替换、系统重装
