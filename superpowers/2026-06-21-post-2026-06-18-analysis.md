# docs/superpowers 2026-06-18 之后内容分析整理

## 范围

- 本文按“`2026-06-18` 以后”理解，只整理 `2026-06-19` 与 `2026-06-20` 的文档。
- 共 `34` 份文档：
  - `22` 份 plans
  - `10` 份 specs

## 总体判断

这一批文档和 `2026-06-18` 之前最大的变化，不是单个功能继续加页面，而是整个 `docs/superpowers` 的主线开始收敛成一套更明确的平台叙事：

```text
共享事实底座
  -> IPAM 作为第一个业务闭环应用
  -> NetPath 作为路径分析应用
  -> AIOps / RCA / SOP 作为知识与诊断消费层
```

换句话说，重点已经从“单点功能实现”转到“统一事实模型 + 上层应用复用 + 闭环验收”。

## 四条主线

### 1. 共享事实底座成为核心中轴

这条线是整批文档里最关键的变化。

核心观点：

- 不再新造一套平行的 fact platform。
- 直接以 `Device Collection 2` 的 canonical facts 作为共享事实层。
- `obsflow` 继续承担 batch / snapshot 主链路。
- `IPAM` 先做第一个 fact projection 消费者。
- `NetPath` 后续消费 engine-ready snapshot。

这条线回答的是：以后 OneOPS 上层应用到底应该“吃什么数据、遵守什么契约、如何表达质量/溯源/最新态”。

### 2. NetPath 从 MVP 讨论进入产品化路线图

`2026-06-19` 到 `2026-06-20` 的 NetPath 文档明显不是零散实现，而是一整套分阶段推进：

- engine port
- SDK adapter
- SnapshotProvider
- route_table canonical fact
- tenant-safe latest fact seam
- policy fact foundation
- quick_env seed
- user journey acceptance

这说明 NetPath 已经从“能不能跑起来”进入“怎样以 OneOPS 产品形态落地”。

这里最重要的约束有三条：

- NetPath 不能自己再造一套 route/interface/topology/policy 数据模型。
- 未覆盖 ACL / NAT / PBR / firewall policy 时，不能静默给出“allowed”结论。
- 生产读取 latest facts 时必须带 tenant 和显式 device scope，不能再走全局无范围读取。

- 路线图和约束已经比较清晰。

- 这意味着 NetPath 已经不再只是“engine 能跑起来”，而是进入“事实快照可落盘、分析结果可追溯、SDK/runtime 走 store-first 主路径”的产品化阶段。
- 真正还没闭合的重点，已经收敛到 policy family processors、交互式结果页、evidence/probe/ticket 闭环，以及生产运营期的 rollout / observability / guardrail。

### NetPath 2026-06-22 补充进展

相对本文最初整理时，NetPath 又往前走了一步，关键变化不是“再补一个 adapter”，而是把 snapshot 从中间态推进成了分析主入口：

- 已补上 snapshot schema 与 durable persistence，分析不再只能依赖内存态或临时拼装结果。
- 已补上显式 snapshot create/get API，开始具备“先产快照、再分析、再回看证据”的产品链路雏形。
- 已补上 fact-backed snapshot builder，并把质量状态显式化；当事实不足时可以 degraded / blocked，而不是静默给出过度乐观结论。
- analyze 主链已经切到 snapshot-first，SDK/runtime 也开始优先消费持久化 snapshot，而不是每次重新现场拼数。
- 对 flow-aware snapshot seed scope 也补上了授权收口，tenant-safe 这条线比前一版更完整。

这组变化的意义在于：NetPath 的“数据入口契约”正在稳定下来。后面无论接结果页、evidence、探测、工单，都会更容易围绕 snapshot id、quality、provenance 来组织，而不是继续让上层直接碰 latest facts。

### 3. IPAM 被定义为第一个完整的事实消费型业务应用

这批 IPAM 文档不再停留在页面补齐，而是开始按“企业级地址规划 -> 地址池 -> 申请分配 -> 释放回收 -> 现网事实 -> 稽核治理 -> 上线验收”来组织。

它的战略位置很明确：

- 不是单纯的地址台账。
- 而是 shared canonical facts 上的第一个业务闭环。
- 后续会把 `interface_ip / mac / arp` 等事实投影成 `ipam_address_fact`，再产出 audit findings。

### 4. AIOps 分成两条并行但可汇合的线

这一批 AIOps 文档分成两部分：

- `SOP / Knowledge Base`：独立产品能力
- `Alert AI Diagnosis + RCA V2`：把既有告警 RCA 链路接到 AI diagnosis

它们的关系不是互相依赖上线，而是“先独立可用，后续再汇合”。

#### SOP / Knowledge Base

定位很明确：

- 做一个独立的“知识与 SOP”中心。
- 先走 deterministic retrieval，不强依赖重型 RAG 平台。
- runbook 只能给只读建议，不能自动执行修复。

代表文档：

- `plans/2026-06-19-oneops-sop-knowledge-base.md`
- `specs/2026-06-19-oneops-sop-knowledge-base-design.md`
- `evidence/2026-06-19-oneops-sop-knowledge-base-smoke-coverage.md`

当前判断：

- 这条线的 MVP 边界最清楚。
- smoke coverage 文档说明“文档入库 -> parse -> chunk -> runbook -> publish -> retrieval -> frontend wrappers”主链已经有覆盖。
- 但真实文件上传、MySQL 实机迁移、浏览器点击链路、alert AI 检索增强仍未覆盖。

#### Alert AI Diagnosis + RCA V2

这条线的关键不是“重新做 RCA”，而是：

- 直接复用当前 alert 页面已经在跑的 `RCA V2` 链路。
- AIOps 只负责把 RCA 结果转换为 cited evidence，再喂给本地 LLM 做解释。
- LLM 不替代 RCA，只做解释和归纳。

代表文档：

- `plans/2026-06-19-alert-ai-diagnosis-rca-v2-integration.md`
- `specs/2026-06-19-alert-ai-diagnosis-rca-v2-integration-design.md`

当前判断：

- 模块边界定得比较好，风险低于“另起一个 RCA 引擎”。
- 核心收益是让“故障定位”和“AI 分析”不再两张皮。

## 依赖关系整理

### NetPath 依赖链

```text
canonical fact contract
  -> route_table / policy fact families
  -> tenant-safe latest fact reader seam
  -> SnapshotProvider quality gate
  -> engine runtime wiring
  -> persisted analyze runs
  -> trace / evidence / probe / ticket closure
```

### IPAM 依赖链

```text
canonical fact contract
  -> interface_ip / mac / arp projection
  -> ipam_address_fact
  -> audit finding generation
  -> frontend facts/audit closure
  -> production launch gate
```

### AIOps 依赖链

```text
knowledge/sop retrieval core
  -> runbook + citation
  -> alert AI later consumes retrieval contract

alert RCA V2
  -> normalize RCA evidence
  -> LLM explanation with citations
```

## 当前阶段判断

### 已经比较清晰的部分

- 平台底座方向已经统一到 canonical facts，而不是继续多头建模。
- NetPath 的产品边界、验收旅程、质量门禁、tenant-safe 原则都已经成型。
- NetPath 的 durable snapshot + snapshot-first analyze 主链已经出现真实落地，不再只是 roadmap 上的目标状态。
- IPAM 已经从“页面拼装”升级为“生产闭环规划”。
- SOP / Knowledge Base 的 MVP 范围很克制，容易先落地。

### 仍然分散或有重复的部分

- NetPath 同时存在 roadmap、long-task、phase plan、phase design、acceptance checklist，多份文档之间有信息重叠。
- IPAM 有 long-task、production launch、three-closure、single-page implementation 几层文档，需要依赖本整理文档才能快速对齐。
- “implemented in isolated worktree / source-level complete / pending integration” 这种状态分散在多文档中，没有统一状态板。

### 主要风险

- 文档扩张速度比集成速度快，容易出现“设计已统一、实现仍分支化”。
- NetPath 虽然已经补齐 snapshot 主链，但如果继续同时推进 policy、结果页、probe、ticket，全量 scope 仍然偏大，需要严格分阶段。
- `dc2` 的 contract / dataset 定义仍然偏薄，尤其哪些 canonical facts 先进入“应用可依赖契约”，哪些字段属于 quality/provenance 必填，仍缺少更硬的字典与版本边界。
- tenant scope 目前更多是 NetPath 侧 fail-closed seam，还不是平台统一能力。
- SOP / AI / NetPath / IPAM 都在指向同一套事实底座，后续如果 contract 稍有变化，影响面会很大。

## 建议的后续整理动作

如果后续还要继续收敛文档，建议优先做三件事：

1. 做一个统一状态板，把 `planned / source-level complete / verified / integrated` 明确写在一个索引里。
2. 把 NetPath 的后续目标从“补 persistence / wiring”切到“policy family 最小闭环 + 结果页/evidence 闭环”，避免继续并行摊大饼。
3. 给 shared fact foundation 补一份最小可执行的 `dc2 contract/dataset` 字典，至少先锁定 route_table、interface_ip、arp/mac、topology 邻接、policy seed 这几类高频事实的字段、质量、溯源与版本边界。

如果按推进优先级来排，我会建议：

1. NetPath 先做“可演示闭环”：snapshot 列表/详情、analysis result、evidence drill-down 串起来。
2. IPAM 继续做“可上线闭环”：projection、audit finding、allocation/reclaim smoke 串起来。
3. `dc2` 再补“最小契约闭环”：不要一次定义所有 dataset，先定义最直接服务 NetPath/IPAM 的那一小撮 canonical contract。

## 结论

`2026-06-19` 到 `2026-06-20` 这一批内容，本质上是在把 OneOPS 从“多个网络运维功能点”重构成“共享事实底座之上的应用平台”。

其中：

- `Fact Foundation` 是底座
- `IPAM` 是第一个业务闭环样板
- `NetPath` 是最重的分析型应用主线，而且现在已经从 engine/adapter 阶段进入 snapshot-first 产品化阶段
- `SOP / RCA / Alert AI` 是知识与诊断消费层

后续如果继续推进，这份整理建议作为入口文档使用。
