# Device V2 Detail And Collection Drawer Optimization Design

## 1. 背景

Device V2 设备管理页当前已经具备比较完整的设备清单、分组浏览、批量采集、监控推送、任务结果保留、日志下载和原始采集数据跳转能力。

这次优化的目标不是重新设计整个设备管理页，也不是调整批量采集和监控推送的触发机制。

当前主要问题集中在两个区域：

1. 设备详情展示的信息过少，无法承担“设备详情”的职责。
2. 采集结果抽屉已有大量证据，但信息组织偏任务技术视角，用户不容易判断“这次基础信息采集采到了什么、哪些设备失败、证据在哪里”。

本设计先优化设备详情和采集结果抽屉。批量采集入口、批量任务工作台、刷新后任务不丢、日志下载等现有机制先保留，待第一轮效果验证后再决定是否调整触发点。

## 2. 页面定位

Device V2 设备管理页是偏设备管理的综合运营视图。

页面主对象是 `Device V2 设备`，不是采集任务、监控任务或纳管流程。

设备详情是采集后的结构化设备档案。

采集结果抽屉是基础信息采集任务的过程与证据中心。

采集和监控推送属于低频操作，但每次操作都很重要。它们涉及复杂过程，必须支持：

- 执行过程可见
- 页面刷新后任务不丢
- 批量任务可回看
- 可按设备排查
- 可下载排障日志
- 可跳转查看最近一次采集数据和原始证据

## 3. 本期范围

### 3.1 本期要做

1. 优化设备详情的信息架构。
2. 在设备详情中更好展示采集后的结构化数据。
3. 支持服务器和网络设备按类型展示不同字段。
4. 保留最近一次采集数据跳转。
5. 优化采集结果抽屉的信息组织和阅读顺序。
6. 保留采集抽屉现有关键能力：自动刷新、手动刷新、批量设备聚焦、日志下载、原始数据展示、后续可采集数据展示。

### 3.2 本期不做

1. 不调整设备列表一级视图主列。
2. 不把采集状态、监控推送状态提升为设备列表一级主信息。
3. 不调整批量采集触发点。
4. 不调整批量任务工作台的任务保留机制。
5. 不重做监控推送结果抽屉。
6. 不新增 mock、前端兜底成功态或静默默认值。

## 4. 设备详情设计

### 4.1 设计目标

设备详情要从当前的“基础字段小卡片”升级为“结构化设备档案”。

用户打开详情后，应能回答：

1. 这台设备的身份、位置、归属是什么。
2. 最近一次基础信息采集采到了哪些结构化数据。
3. 这台设备是服务器还是网络设备，类型相关字段是什么。
4. 哪些数据来自清册，哪些来自最近采集。
5. 如果要看完整采集证据，应从哪里跳转。

### 4.2 展示原则

1. 先展示通用信息，再展示设备类型专属信息。
2. 有数据才展示对应区块，不用大量 `-` 撑页面。
3. 不把原始 JSON 直接平铺在详情页。
4. 最近一次采集数据跳转必须保留。
5. 采集、监控、WebShell、操作录屏等运维动作靠后展示。
6. 字段来源不明确时，不新增前端猜测映射；应记录为待确认字段。

### 4.3 推荐分区

#### A. 概览区

目的：快速确认用户打开的是哪台设备。

建议展示：

- 设备名称
- 设备编号
- 设备类型
- 平台
- 管理状态
- 最近更新时间

概览区只展示稳定管理信息，不展示资料完备和是否可纳管。

#### B. 基础身份

建议展示：

- 设备名称
- 设备编号
- 设备类型
- 平台
- 厂商
- 型号
- 序列号
- 主机名

字段可以来自 Device V2 清册，也可以来自最近一次采集后的结构化结果。若两者都有，应优先展示当前设备档案中的有效值，并在需要时标注最近采集值或更新时间。

#### C. 位置与归属

建议展示：

- 区域
- 机房
- 机架
- 租户
- 安全区
- 业务归属
- 标签或分组摘要

该区块服务于设备管理，不服务于纳管前置检查。

#### D. 网络与访问

建议展示：

- 管理 IP
- 带内地址
- 带外地址
- 主要访问协议
- SSH / WinRM / SNMP 等访问方式摘要
- 凭证绑定摘要

凭证明文绝不能展示。只展示凭证引用、绑定状态或用途摘要。

#### E. 最近一次基础信息采集

目的：让用户在详情页直接看到“最近采集发生过什么”，并保留深查入口。

建议展示：

- 采集时间
- 采集协议
- 任务 ID
- run ID / store run ID
- DC2 run ID
- 采集结果摘要
- 采集到的关键字段数量
- 失败或缺失摘要
- 查看详细数据入口

必须保留当前跳转能力：

- 从详情页跳转到最近一次 DC2 采集数据
- 携带 `run_id`、`target_id`、`device_code`、`contract_key`、`store_task_id`、`store_run_id` 等已有上下文

#### F. 类型专属区

详情页必须支持按设备类型展示字段。

服务器设备和网络设备不能使用完全相同的一组字段。

### 4.4 服务器设备专属区

服务器设备重点展示主机与系统层面的结构化数据。

建议字段：

- 操作系统
- OS 版本
- 内核版本
- 主机名
- CPU 型号 / 核数
- 内存容量
- 磁盘 / 文件系统摘要
- 网卡列表摘要
- IP / MAC 摘要
- 虚拟化信息或宿主信息
- SSH / WinRM 访问摘要

字段为空时跳过对应行或区块。

### 4.5 网络设备专属区

网络设备重点展示 SNMP、接口、链路和设备角色。

建议字段：

- 厂商
- 型号
- 设备角色
- 固件版本 / 软件版本
- `sysName`
- `sysDescr`
- `sysObjectID`
- `sysLocation`
- `sysContact`
- 接口数量
- 接口状态摘要
- 接口速率摘要
- MAC / IP 摘要
- 端口 / 链路 / 物理连线入口
- SNMP 凭证或采集协议摘要

接口明细较多时，详情页只展示摘要和入口。完整接口表可进入现有运维工作台、采集详情或后续专门区块。

### 4.6 运维动作区

运维动作保留，但不作为详情主体。

建议放在详情页靠后位置：

- 基础信息采集
- 监控推送
- WebShell
- 操作录屏
- 接口
- 物理连线

动作执行后仍进入现有结果抽屉或工作台，不在详情页内重造执行过程。

## 5. 采集结果抽屉设计

### 5.1 设计目标

采集结果抽屉要从“任务技术证据堆叠”调整为“基础信息采集结果与证据中心”。

用户打开抽屉后，应能按顺序理解：

1. 这次采集任务整体是否完成。
2. 批量采集时每台设备分别是什么结果。
3. 当前聚焦设备采到了哪些结构化信息。
4. 哪些字段失败，失败原因是什么。
5. 原始采集数据和排障日志在哪里。

### 5.2 必须保留的现有能力

以下能力不能删除或弱化：

- 手动刷新
- 执行中自动刷新
- 批量任务页面刷新后不丢
- 批量采集工作台保留任务
- 批量设备结果切换和聚焦
- 任务 ID、run ID、阶段、状态展示
- 下载采集排障日志
- 查看最近一次 DC2 采集详细数据
- 展示字段级 observation
- 展示标准化值
- 展示原始采集数据
- 展示后续可采集数据或采集计划

### 5.3 推荐信息顺序

#### A. 顶部任务概况

展示任务层面的摘要：

- 任务标题
- 任务 ID
- 任务状态
- 当前阶段
- 设备范围
- 更新时间
- 刷新
- 下载日志

执行中任务展示自动刷新提示。

#### B. 批量结果摘要

展示：

- 本次采集范围
- 已进入执行
- 成功
- 失败
- 执行中
- 待开始

批量采集时应明确当前聚焦设备。

#### C. 设备结果列表

批量采集时，设备结果列表应靠前展示。

每行建议展示：

- 设备编号
- 设备名称或 IP
- 状态
- 阶段
- 采集协议
- 采集时间
- 失败摘要
- 查看详情 / 当前查看

点击某台设备后，抽屉下方内容切换到该设备的结构化结果和原始证据。

#### D. 当前设备结构化采集结果

这是采集抽屉最需要优化的核心区域。

当前 observation rows 不应只以字段表格出现，应先按业务区块汇总：

- 基础身份
- 网络与访问
- 位置与归属
- 硬件与系统
- 接口与链路
- 协议证据

每个区块只展示有数据的字段。

字段级 observation 表可以保留，但应作为“字段明细”位于结构化结果之后。

#### E. 字段明细

展示字段级结果：

- 采集能力
- 字段
- 状态
- 值预览
- 报错
- 采集时间
- 查看明细

选中字段后展示：

- 报错信息
- 标准化值
- 原始采集数据

#### F. 技术排障信息

技术锚点、原始 JSON、后端任务证据、后续可采集计划应保留，但默认靠后或折叠。

它们服务于专业排障，不应打断普通用户理解采集结果。

## 6. 字段来源要求

所有新增或重组字段必须能追溯来源。

允许来源：

- Device V2 清册字段
- Device V2 `attributes`
- Device V2 `metadata`
- 最近一次采集结构化结果
- DC2 observation 标准化值
- DC2 原始采集数据
- V1 关联设备数据
- 用户编辑字段

不允许：

- 前端编造字段
- 用空数组或默认对象掩盖接口失败
- 用默认成功状态掩盖任务失败
- 字段缺失时猜测语义

字段来源不清楚时，应在实现任务中记录为接口或字段映射问题。

## 7. 交互规则

1. 从设备详情点击最近一次采集，应继续跳转到现有 DC2 采集详情页面。
2. 从采集抽屉点击查看详细数据，应继续使用现有跳转逻辑。
3. 批量采集后自动打开采集结果抽屉的行为保留。
4. 采集抽屉关闭后，批量任务仍应保留在当前页面工作台。
5. 页面刷新后，已保存的批量采集任务仍可恢复。
6. 下载日志时，单设备任务下载任务日志；批量任务优先下载当前聚焦设备日志。
7. 采集抽屉中的设备切换不应重新创建任务，只重新加载该任务下对应设备的数据。

## 8. 验收标准

### 8.1 设备详情

- 打开设备详情后，能看到比当前更多的结构化设备档案信息。
- 通用区、服务器专属区、网络设备专属区边界清楚。
- 服务器设备不会展示大量网络设备专属 SNMP 字段空位。
- 网络设备不会展示大量服务器 OS / CPU / 内存字段空位。
- 最近一次采集摘要可见。
- 最近一次采集详细数据跳转保留可用。
- 运维动作保留，但不抢占详情主体。

### 8.2 采集结果抽屉

- 任务概况、批量结果、设备结果列表、当前设备结构化结果、字段明细、技术排障信息层级清楚。
- 批量采集时能快速找到失败设备。
- 切换设备后能查看该设备的字段级结果和原始采集数据。
- 下载日志能力保留。
- 自动刷新和手动刷新能力保留。
- 页面刷新后任务不丢能力保留。
- 后续可采集数据或采集计划展示能力保留。

### 8.3 质量约束

- 不新增 mock 数据。
- 不新增静默 fallback。
- 不吞掉接口失败。
- 不隐藏真实错误为空态。
- 不删除现有任务证据链。
- 不破坏现有 `npm run typecheck:d2` 验证门禁。

## 9. 后续实现提示

后续实现应优先做信息重组，而不是大规模改动接口。

推荐顺序：

1. 梳理当前详情页可直接使用的 Device V2 字段。
2. 梳理最近一次采集和 observation 中可映射的结构化字段。
3. 为详情页增加按设备类型分区的展示模型。
4. 重排采集抽屉结构，先不删除底层数据加载逻辑。
5. 保留旧字段级表格和原始证据，只调整展示位置和标题。
6. 运行 `npm run typecheck:d2`。

若实现中发现结构化采集字段不足，应记录字段契约缺口，而不是用前端猜测补齐。
