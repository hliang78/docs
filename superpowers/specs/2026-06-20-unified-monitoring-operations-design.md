# 统一监控运维设计说明

## 背景

本设计面向 OneOPS 前端演示场景，目标是在当前平台前后端能力基础上，快速构建一条可信、可演示、可连续操作的“统一监控运维”用户旅程。该能力不以汇报大屏为目标，而以一线运维值班台为核心视角，强调：

- 先发现新设备
- 再判断当前阻塞项
- 再执行探测、入库、方案下发
- 最后在需要时衔接告警与拓扑能力

本次只设计“统一监控运维”这一项需求，不展开 IPv4/IPv6 地址管理、网络路径可观测、网络变更全流程自动化、网络数智化运维能力。

## 外部参考

本设计参考了主流网络运维产品的官方能力表达，但不复制其页面形态，重点提炼其工作流和信息组织方式：

- PRTG 自动发现：强调周期发现与手动触发并存，发现过程先做连通性和协议识别，再补监控对象
- ManageEngine OpManager：强调发现结果进入待处理队列，再执行 approve / ignore / monitor
- Zabbix Discovery / Problems：强调当前问题列表与发现规则驱动的后续动作
- Auvik：强调网络发现基于 ICMP、SNMP、LLDP、CDP 等协议组合，并将发现结果持续纳入库存与关系图

设计上的落点是：借鉴这些产品“发现 -> 队列 -> 探测 -> 纳管 -> 下发”的闭环，而非做图表中心或营销式大屏。

## 目标

### 产品目标

让用户在一个统一模块里完成以下事情：

1. 发起网段发现或邻居扩展发现
2. 查看新发现设备
3. 快速定位当前阻塞项
4. 在当前页面内执行 Ping / SNMP 探测
5. 完成设备入库
6. 确认下发监控方案和告警方案

### 演示目标

- 操作路径完整，不能是孤立页面堆砌
- 尽量复用现有 OneOPS 页面和接口能力
- 页面数据看起来像真实系统，而不是演示假稿
- 页面文案要像运维台，不要有“demo”“样例”“智能赋能”等表演式语言

## 范围

### 本次纳入设计

- 一级入口 `统一监控运维`
- 子页 `设备发现与纳管`
- 子页 `告警监控` 的入口衔接
- 子页 `关联拓扑` 的入口衔接
- 设备发现、待纳管队列、设备探测、设备入库、方案推荐与确认下发

### 本次不重做

- 当前告警列表页面本体
- 当前拓扑页面本体
- 设备入库底层流程能力
- 监控方案与告警方案底层编排能力

## 信息架构

一级入口：

- `统一监控运维`

左侧菜单：

- `设备发现与纳管`
- `告警监控`
- `关联拓扑`

默认落点：

- `设备发现与纳管`

衔接规则：

- `告警监控` 直接跳当前已有告警列表页，不重新设计告警页
- `关联拓扑` 直接复用当前已有拓扑页，不重做拓扑页，只设计入口衔接

## 总体用户旅程

### 主旅程

1. 用户进入 `统一监控运维`
2. 默认看到 `设备发现与纳管`
3. 通过顶部主操作区决定是否发起：
   - 网段发现
   - 邻居扩展
   - 查看周期任务
4. 页面主体显示待纳管设备表格
5. 用户按阻塞项优先浏览设备
6. 选中某台设备后，右侧详情面板展示当前结论、阻塞项、建议动作和探测结果
7. 若设备需验证，用户原地执行 `立即探测`
8. 探测通过后，执行 `立即入库`
9. 入库后，在 `推荐方案` 页签确认下发监控方案与告警方案
10. 若需进一步查看设备关系，从 `识别结果` 页签进入 `关联拓扑`

### 次级旅程

- 从菜单进入 `告警监控`，直接查看当前告警列表
- 从 `设备发现与纳管` 右侧 `识别结果` 页签跳转 `关联拓扑`

## 页面定位

`设备发现与纳管` 不是统计页，也不是概览看板，而是值班台工作面。

页面必须优先回答三件事：

1. 新发现了什么设备
2. 当前哪些设备卡住，卡在哪里
3. 哪些设备已经具备继续纳管的条件

## 首页布局

### 顶部主操作区

采用 `一个主按钮 + 两个次按钮横排` 的结构：

- 主按钮：`发起网段发现`
- 次按钮：`发起邻居扩展`
- 次按钮：`查看周期任务`

设计原则：

- 首页先给动作，不先给统计解释
- 不做复杂导览，不做多层入口卡片
- 操作区要短、硬、直接

### 中部主体

采用 `左侧待纳管设备表格 + 右侧详情处理面板` 的两栏布局。

左侧：

- 待纳管设备表格

右侧：

- 摘要卡 + 页签切换的详情处理面板

## 顶部发现入口设计

### 发起网段发现

进入方式：

- 右侧弹出配置面板

字段分组：

1. 扫描范围
   - 区域
   - 扫描网段
   - 扫描地址范围
2. 凭证模板
   - SNMP 凭证模板
   - SSH/Telnet 凭证模板
3. 发现方式
   - ICMP
   - SNMP
   - 端口探测
4. 任务属性
   - 任务名称
   - 备注
   - 执行人

### 发起邻居扩展

进入方式：

- 右侧弹出配置面板

字段分组：

1. 种子设备
   - 种子设备名称
   - 管理 IP
   - 所属区域
2. 扩展策略
   - 扩展深度
   - 是否跨区域
   - 是否仅发现网络设备
3. 凭证模板
   - SNMP 凭证模板
   - SSH/Telnet 凭证模板
4. 任务属性
   - 任务名称
   - 备注
   - 执行人

### 周期任务

通过 `查看周期任务` 进入现有或拟复用的任务列表，承接以下信息：

- 周期发现任务名称
- 发现方式
- 扫描范围或种子设备
- 最近执行时间
- 最近执行结果
- 手动立即执行入口

周期任务页面不要求本轮重做，但首页入口要成立。

## 待纳管设备表格

### 排序原则

默认排序为：

- `阻塞项优先`

目的不是用“高中低风险”做主观评分，而是把最影响继续处理的设备排在前面。

### 列结构

主列顺序固定为：

1. `设备名称`
2. `阻塞项`
3. `发现来源`
4. `建议动作`

### 第一列：设备名称

显示规则：

- 主行显示设备名称
- 设备名称缺失时，显示 `未命名设备（IP）`
- 副行显示 `IP · 设备类型`
- 设备类型缺失时，显示 `IP · 类型待识别`

设计原则：

- 第一眼先认设备
- 不把任务号、来源、状态放在第一列

### 第二列：阻塞项

显示规则：

- 只显示一个主标签
- 不显示多标签并排
- 其余附加问题进入右侧详情

阻塞项优先级：

1. `凭证缺失`
2. `疑似重复`
3. `SNMP待验证`
4. `类型待识别`

设计原则：

- 表格不是诊断报告
- 第二列只承担“当前最关键卡点”的角色

### 第三列：发现来源

只显示最简来源类型：

- `网段发现`
- `邻居扩展`

任务号、发现时间、种子来源等信息进入右侧详情的 `发现证据` 页签，不在表格首屏堆叠。

### 第四列：建议动作

显示规则：

- 单个主按钮
- 不显示多个并行动作
- 文案采用动作直白型

动作映射：

- `凭证缺失` -> `补充凭证`
- `疑似重复` -> `人工确认`
- `SNMP待验证` -> `立即探测`
- `类型待识别` -> `立即探测`
- `可直接入库` -> `立即入库`

## 右侧详情面板

### 整体结构

采用：

- `摘要卡 + 页签切换`

顶部摘要卡固定展示：

- 当前主结论
- 主阻塞项标签
- 关键状态标签
- 主动作按钮
- 次动作按钮

### 页签顺序

1. `风险与建议`
2. `发现证据`
3. `识别结果`
4. `推荐方案`

## 风险与建议页签

内容顺序：

1. `结论`
2. `当前阻塞项`
3. `建议动作`
4. `最近探测结果`

### 探测顺序

默认执行：

- `先 Ping，再 SNMP`

### 探测结论规则

- `Ping 失败` -> `设备不可达，建议先检查管理 IP 连通性`
- `Ping 成功，SNMP 失败` -> `设备在线，需补充或修正 SNMP 凭证`
- `Ping 成功，SNMP 成功` -> `设备可纳管，建议立即入库`

## 发现证据页签

### 第一组字段

- 发现来源
- 发现任务
- 首次发现时间
- 最近发现时间

### 第二组字段

如果来源为 `网段发现`：

- 扫描网段
- 扫描地址范围

如果来源为 `邻居扩展`：

- 种子来源
- 邻居端口关系

### 第三组字段

- 发现协议摘要

展示形式：

- `协议标签 + 状态`

示例：

- `ICMP 可达`
- `SNMP 已发现`
- `LLDP 发现`
- `CDP 未启用`

## 识别结果页签

### 第一组：设备身份识别

- 厂商
- 设备类型
- 型号
- 系统名称

### 第二组：管理面信息

- 管理 IP
- MAC
- SNMP 版本

### 第三组：识别补充说明

展示形式：

- 自然语言说明列表

示例：

- 型号：sysDescr
- 系统名称：sysName
- 设备类型：LLDP 邻居特征

### 拓扑入口

在 `识别结果` 页签中提供：

- `查看关联拓扑`

点击后跳转当前已有拓扑页，并带入当前设备上下文。

## 推荐方案页签

展示顺序：

1. `推荐监控方案`
2. `推荐告警方案`

两块统一结构：

- 方案名称
- 简短说明
- 主按钮

推荐方案页签的目的不是解释“平台多强”，而是让用户明确知道：

- 这台设备入库后将推什么能力
- 用户是否需要确认下发

## 告警监控

本轮不重做当前告警页面。

设计结论：

- 左侧菜单保留 `告警监控`
- 点击后直接进入当前已有告警列表页
- 不在本轮重新设计告警列表结构

## 关联拓扑

本轮不重做当前拓扑页面。

设计结论：

- 左侧菜单保留 `关联拓扑`
- 直接进入当前已有拓扑页
- 同时在 `设备发现与纳管 -> 识别结果` 页签中提供 `查看关联拓扑` 动作按钮

## 现有 OneOPS 能力复用策略

本设计最大限度复用当前平台已有前后端能力：

- 设备入库：复用现有设备导入与设备管理链路
- 告警监控：复用当前已有告警列表页
- 关联拓扑：复用当前已有拓扑页
- 监控方案和告警方案：复用现有监控编排与指标资产能力
- SNMP 相关能力：复用现有 SNMP 采集目标和策略能力

本轮重点是做“统一入口和连续工作流”，不是平地重起新系统。

## 数据真实性规则

页面上的数据必须像真实系统。

### 设备命名

- 采用真实网络设备命名风格
- 同一地域、同一设备层级命名规则一致
- 例如：
  - `SW-CORE-SZH-B12`
  - `FW-DC1-EDGE-03`
  - `RTR-HZ-WAN-07`

### 地址与区域

- 管理 IP 与区域必须匹配
- 同一批任务内地址分布应有规律
- 不允许出现明显乱填

### 任务编号

- 网段发现与邻居扩展使用不同前缀
- 例如：
  - `DISC-20260620-014`
  - `NEIGHBOR-20260620-006`

### 时间逻辑

- 首次发现时间不得晚于最近发现时间
- 探测时间、任务时间、入库时间应符合处理顺序

### 识别完整度

必须保留部分“不完整识别”的设备，避免页面过于理想化：

- 无设备名
- 类型待识别
- 型号部分识别
- SNMP 版本未知

### 探测结果一致性

- Ping 失败时，不允许 SNMP 成功
- SNMP 成功时，应该能回填 sysName、sysDescr、版本线索等
- 错误原因必须具体，例如：
  - `community 不匹配`
  - `SNMP 超时`
  - `管理 IP 不可达`

### 推荐方案一致性

- 方案必须和设备类型匹配
- 防火墙、交换机、路由器、负载均衡的推荐不能混淆

## 文案原则

### 保留的文案类型

- 字段名
- 状态词
- 标签
- 直接动作
- 必要的极短结论

### 避免的文案类型

- 解释性长段落
- 营销式或方案式表达
- “demo”“样例”“智能赋能”“全景洞察”等表演词

### 推荐语言风格

- `补充凭证`
- `人工确认`
- `立即探测`
- `立即入库`
- `查看关联拓扑`

### 结论文案要求

必须可执行，而不是抽象描述。例如：

- `设备不可达，建议先检查管理 IP 连通性`
- `设备在线，需补充或修正 SNMP 凭证`
- `设备可纳管，建议立即入库`

## 视觉与交互原则

- 不是大屏
- 不是图表墙
- 不是解释页
- 是高密度、可连续处理的运维工作面

交互上遵循：

- 当前页能处理的，尽量不跳走
- 当前行能给主动作的，不要求用户进深层页再选
- 详情面板先给结论，再给证据
- 表格强调处理优先级，不强调说明文字

## 风险与边界

### 主要风险

- 如果文案过多，页面会像方案稿，不像系统
- 如果数据过于理想，页面会像假数据演示
- 如果表格动作过多，页面会显得凌乱

### 设计边界

- 不重做告警页
- 不重做拓扑页
- 不重构现有后端能力
- 不在本轮引入新的复杂分析引擎表达

## 设计结论

`统一监控运维` 的核心不是做一个漂亮的大盘，而是把 OneOPS 当前已有能力，重新组织成一条适合一线运维值班台的连续旅程：

- 发现新设备
- 判断阻塞项
- 原地探测
- 完成入库
- 确认方案下发
- 必要时衔接告警与拓扑

这条链路一旦前端成立，后续无论是扩到 SDN 运维，还是继续补告警、拓扑、路径分析，都能保持同一个用户心智。
