# Device Discovery Drawer Design

## Goal

在现有设备清单页新增 `设备探测` 入口，通过右侧大抽屉完成探测计划定义、执行与结果承接；探测范围与种子来自当前数据库中的设备数据，执行后允许生成数据库中尚未纳入的新设备，再回流到设备清单继续做采集和监控推送。

## User Journey

1. 值班人员进入设备清单页。
2. 点击顶部工具栏新增按钮 `设备探测`。
3. 右侧打开 `设备探测` 大抽屉，默认选中一条探测计划。
4. 在抽屉中查看或编辑计划，计划类型为：
   - `网段探测`
   - `设备种子`
5. 点击 `立即执行` 后，页面短暂呈现：
   - `排队中`
   - `探测中`
   - `识别中`
6. 执行完成后，结果进入抽屉内 `待入库设备` 页签，同时主页面设备清单可切换看到 `来源=设备探测` 的新设备。
7. 用户在抽屉中将设备 `入设备清单`，随后继续使用现有的 `批量采集`、`批量监控`、`配置管理` 等能力。

## Entry And Layout

### Main Page

- 入口位于设备清单页顶部工具栏，放在 `监控任务` 与 `设备导入` 之间。
- 主页面保持现有设备清单，不新建独立发现页面。
- 增加一个轻量提示：当存在新发现设备时，在设备清单说明文案中体现 `其中 X 台来自设备探测`。

### Drawer

- 形态：右侧大抽屉，宽度约 `1120px`
- 顶部：标题 `设备探测`
- 主体结构：
  - 左侧计划列表
  - 右侧计划详情

### Left Rail

- 顶部按钮：`新建计划`
- 按计划卡片分组展示：
  - `网段探测`
  - `设备种子`
- 每个计划卡片展示：
  - 计划名
  - 探测类型
  - 最近执行时间
  - 最近一次发现设备数
  - 当前状态

### Right Panel

- 顶部摘要卡：
  - `启用计划`
  - `24小时新发现`
  - `待入设备清单`
- 下方页签：
  - `计划详情`
  - `执行记录`
  - `待入库设备`

## Data Strategy

这次不依赖后端新增探测接口，前端基于当前数据库中已经加载到页面的设备列表组织演示数据。

### Existing Database Data Reuse

- `网段探测` 的默认范围：
  - 从当前设备的主 IP 自动归纳 `/24` 网段
  - 取出现频次最高的若干网段作为候选
- `设备种子` 的候选：
  - 直接来自当前设备清单已有设备
  - 优先选择具备设备名、IP、区域位置信息的核心/汇聚/路由类设备
- `区域 / 机房 / 机架 / 平台`：
  - 直接复用当前设备属性

### Discovery Result Modeling

- 探测结果允许生成“当前设备库中不存在”的新设备候选
- 这些结果不直接写回真实数据库，只在前端 demo 状态中表现为：
  - `待入库`
  - `已入设备清单`
- 新设备应尽量继承源设备上下文，例如：
  - 区域
  - 机房
  - 平台
  - 型号族

## Plan Types

### 网段探测

字段：

- 计划名称
- 所属区域
- 目标网段
- 排除网段
- 协议组合
- SNMP 凭证组
- 执行周期

### 设备种子

字段：

- 计划名称
- 种子设备
- 种子 IP
- 邻居发现深度
- 发现依据
- SNMP 凭证组
- 执行周期

## Execution Experience

- 立即执行采用“两段式真实感”：
  - 短暂的执行状态切换
  - 自动生成结果
- 每次执行记录应展示：
  - 执行时间
  - 计划名称
  - 探测范围
  - 发现总数
  - 新设备数
  - 已识别数
  - 待人工确认数

## Pending Inventory Flow

- `待入库设备` 列展示：
  - 设备名称
  - IP
  - 识别类型
  - 发现来源
  - 命中凭证
  - 纳管动作
- 支持动作：
  - `入设备清单`
  - `忽略本次`
- `入设备清单` 后：
  - 抽屉中的记录状态更新为 `已入设备清单`
  - 主页面设备清单增加对应设备
  - 设备默认进入 `待采集 / 待监控`

## Copywriting Principles

- 不出现 `demo` 字样
- 用值班现场口径，不用方案宣传口径
- 结果数据要像真实运维输出，例如：
  - `发现 14 台，识别成功 11 台，待人工确认 3 台`
  - `来源：设备探测`
  - `纳管状态：待采集`

## Scope

本次只做前端闭环，不新增真实探测后端能力，不改现有采集/监控主流程，只把探测结果自然接入现有设备清单旅程。
