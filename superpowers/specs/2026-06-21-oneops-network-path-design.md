# OneOPS 网络路径前端设计

## 背景与目标

当前 OneOPS 已具备 NetPath 路径分析、真实拓扑投影、设备证据抽屉、Probe Plan、Probe Execution、Task/Ticket 承接等基础能力，但这些能力仍主要以研发工作台形态存在，缺少适合网络运维日常查询与业务访问异常定位的正式产品旅程。

本次设计目标是在不新增“演示专用世界观”的前提下，基于当前 OneOPS 前后端与真实拓扑数据，重构出一条完整、自然、可演示的产品旅程：

1. 网络运维可以从 `拓扑中心` 进入 `网络路径`。
2. 可以兼顾 `业务访问不通定位` 与 `日常查询` 两类使用场景。
3. 路径、节点证据、连通性验证、后续处理可以在同一诊断主线中完成。
4. 页面与数据不能出现 `demo`、`mvp`、`工作台` 等内部语义。
5. 所有关键拓扑节点、链路与路径摘要优先使用真实拓扑数据。

## 借鉴与设计原则

本设计借鉴以下成熟产品的共性：

- Datadog Network Path: 先列表筛选，再进入单条 Path View。
- ThousandEyes Path Visualization: 主路径优先、复杂度可控、节点/链路交互细节丰富。
- Azure Network Watcher Topology: 从拓扑直接 drill down 到诊断与处理动作。
- Cloudflare / Grafana traceroute 可视化: 把 ping / traceroute 作为路径证据的一部分，而不是孤立测试页。

设计原则如下：

1. 真实拓扑优先，不另造虚构网络。
2. 用户先完成查询，再进入单条详情，不在首页堆满技术细节。
3. 路径图是诊断主画布，节点证据与连通性验证是围绕主画布展开的证据面板。
4. 优先复用现有 OneOPS API 与组件，只做产品化重组和文案重构。
5. 文案先表达业务判断，再表达系统动作。

## 用户角色与核心场景

### 目标角色

网络运维。

### 核心场景

同一入口同时承接两类场景：

1. 业务访问不通定位。
2. 日常访问路径查询。

### 入口归属

`拓扑中心 / 网络路径`

此入口属于现有拓扑产品域，不归属监控、AIOps 或工单子域。

- 页面副标题与摘要说明

## 用户旅程

### 查询台旅程

1. 用户从 `拓扑中心` 进入 `网络路径`。
2. 在查询页输入源节点、目的节点、协议、端口、入口设备等条件。
3. 系统发起路径分析，返回主路径及候选路径结果。
4. 用户在结果列表中快速判断当前是否存在异常，以及应该点开哪条路径。
5. 用户点击 `查看详情` 进入单条路径详情页。

### 详情页旅程

1. 页面默认高亮主路径。
2. 顶部摘要先回答“当前结论是什么”。
3. 用户点击路径中的某个网络设备节点。
4. 右侧诊断面板立即切换到该节点的相关证据。
5. 用户根据路径摘要、节点证据、连通性验证完成判断。
6. 如需继续处理，进入验证任务、策略核查或工单处理。

## 信息架构

采用双层结构。

### 页面一：网络路径

职责：

- 接收查询
- 展示结果列表
- 承接最近查询与异常筛选

### 页面二：路径详情

职责：

- 展示当前主路径与候选路径
- 提供节点级证据查看
- 提供连通性验证查看
- 提供后续处理入口

## 风险与取舍

### 风险 1：真实 flow 字段暂不完整

如果当前后端尚未稳定返回真实源 IP、目的 IP、端口、接口等完整信息，首版前端应以真实节点路径为主，不预先写死假值。

### 风险 2：图投影与路径 hop 不完全匹配

如出现拓扑中存在链路但路径图无法完全投影，应保留诊断提示，不隐藏分析结果。

### 风险 3：现有页面能力过于集中在单文件

当前 `NetPathMvp.vue` 内容过重，重构时需要先拆职责，再整合风格，否则会继续维持“研发工作台感”。

### 明确取舍

首版优先做：

1. 正式入口与页面分层
2. 真实拓扑上的路径高亮
3. 节点证据产品化展示
4. 连通性验证整合

首版暂不追求：

1. 全量复杂度控制器
2. 全量趋势分析
3. 大规模跨页闭环编排

## 结论

本设计不是新造一套网络路径系统，而是基于真实拓扑、现有 NetPath 结果、现有拓扑组件与现有证据接口，把当前研发工作台升级为正式产品旅程。

最终对外呈现为：

- 一个正式的 `网络路径` 查询页
- 一个围绕真实拓扑展开的 `路径详情` 页
- 一条能够同时承接 `日常查询` 与 `业务访问异常定位` 的网络运维主线
