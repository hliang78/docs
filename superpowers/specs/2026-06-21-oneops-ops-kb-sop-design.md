# OneOPS 运维知识库与 SOP 转换设计

Date: 2026-06-21

## 目标

将以下两份外部运维资料转换为适合 OneOPS 仓库长期维护和后续 AI 检索的文档资产：

- `tmp/运维团队SOP文档-202510.docx`
- `tmp/服务故障事件记录表.xlsx`

转换后的结果不追求原文逐页搬运，而是沉淀为：

1. 面向知识检索的故障模式知识条目
2. 面向值班执行的标准 runbook
3. 面向后续扩展的源资料映射说明

## 现状判断

源 `docx` 是一套 GPU/算力服务器运维手册，覆盖服务器识别、检查项、故障信息收集、硬件故障、系统故障、配置操作、压测、重装、常用操作与历史复盘。文档体量大，包含大量截图，且文件结构不完全标准，不适合直接作为 OneOPS 知识库主格式。

源 `xlsx` 是一份故障/事件记录台账，包含 475 条有效记录，可用于提炼高频故障类型、处理路径和机型差异。它更适合作为知识抽取输入，而不是直接给值班同学使用。

仓库当前已有两个合适落点：

- `docs/knowledge`: 放经验条目、模式总结、 lessons learned
- `docs/runbooks`: 放可执行 SOP / runbook

## 设计原则

### 1. OneOPS-first

转换后的 SOP 应以 OneOPS 能力为主线组织：

- 告警管理
- 设备监控 / 日志中心 / 网络拓扑
- 统一凭证
- 任务模板 / 变量集 / 任务中心
- 配置管理
- 操作审计

原文中的 Zabbix、带外管理、厂商工具、手工命令保留为补充取证或兜底步骤，而不是主导航。

### 2. 去截图依赖

原文大量依赖图片说明。转换时不复制图片，而是抽取可操作的文字知识：

- 怎么判断
- 去哪里看
- 采集什么证据
- 先做什么，再做什么
- 什么场景必须升级到厂商或更换备件

### 3. 结构分层

不同资料转成不同文档，不混成一篇巨文：

- `source map`: 说明原始资料覆盖范围、转换边界、哪些章节被吸收
- `knowledge`: 沉淀故障模式、机型差异、高频原因与方案
- `runbook`: 值班执行流

### 4. 脱敏和去遗留化

不把原文中的明文口令、代理凭据、飞书路径等直接抄进 OneOPS 文档。内部地址与命令仅保留到确有运维价值的程度，并优先改写为“从 OneOPS 页面或统一凭证获取”。

## 文档产出

### A. 知识库源映射

文件：

- `docs/knowledge/gpu-ops-sop-source-map-2026-06-21.md`

职责：

- 记录原 SOP 的章节结构
- 说明转换成 OneOPS 后分别落到哪些知识条目 / runbook
- 标明未直接迁移的部分和原因

### B. 事件知识条目

文件：

- `docs/knowledge/gpu-server-fault-patterns-from-incident-log-2026-06-21.md`

职责：

- 从事件表提炼高频故障原因与处理路径
- 给出按机型的差异观察
- 提供初步故障分类与优先级建议

### C. 执行型 SOP

文件：

- `docs/runbooks/gpu-server-fault-response.md`

职责：

- 给值班/排障同学一个可直接执行的响应流程
- 将排障拆成：确认影响、定位对象、采集证据、分类处置、验证闭环、复盘入库
- 以 OneOPS 页面与能力为主流程

## 验证标准

文档完成后需满足以下检查：

1. 能从内容上对应回原 `docx` 的核心章节
2. 能从内容上反映 `xlsx` 的高频故障模式
3. 目录风格与现有 `docs/knowledge` / `docs/runbooks` 保持一致
4. 文档中不出现明显敏感信息抄录
5. 值班同学只读 runbook 即可完成一次标准排障闭环

## 不做的事情

本次不做：

- 不把原 Word 全量转成 Markdown
- 不把全部 300+ 图片落库
- 不建立结构化数据库或向量索引
- 不自动接入 OneOPS 产品内知识库上传流程

这些可以作为下一阶段工作，在当前文档稳定后再继续。
