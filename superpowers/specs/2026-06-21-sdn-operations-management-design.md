# SDN运维管理设计说明

## 背景

本设计面向 OneOPS 前端演示场景，目标是在当前 `设备管理 -> SDN控制器` 页面基础上，扩展出一套可信、连续、可操作的 SDN 运维工作台，覆盖以下四类能力：

- 主流品牌 SDN 控制器集群维护
- SDN 资源管理
- SDN 配置下发
- SDN 网络告警

本设计不将 SDN 运维管理并入 `统一监控运维`，而是保留在当前 `设备管理` 域下，对现有 `SDN控制器` 页面进行增强。

## 设计目标

### 产品目标

让用户在一个页面框架内完成以下事情：

1. 查看控制器集群健康与节点状态
2. 查看 SDN 资源总体规模与明细对象
3. 发起 SDN 配置下发
4. 查看和处理 SDN 网络告警

### 演示目标

- 保持和当前 OneOPS 页面结构连续
- 最大化复用已有 API 与页面能力
- 不做“从零新起一个平台”的假结构
- 页面数据和对象命名要像真实控制器运维界面

## 外部参考

本设计参考了主流 SDN 控制平面产品的官方运维表达方式，重点借鉴其工作台组织方式而非视觉样式：

- Cisco Catalyst Center：强调单一控制台和控制面集中运维
- Cisco Catalyst Center HA：明确三节点高可用集群的状态与运维流程
- VMware NSX Administration：强调 manager cluster、alarm、诊断与控制面维护
- Juniper Apstra Cluster Management：明确 cluster management 作为独立运维视图
- Arista CloudVision：强调配置流程、告警与控制面集中管理

设计落点是：

- 先看控制器集群
- 再看资源、配置、告警
- 保持一线运维台的节奏，而不是展示型门户

## 范围

### 本次纳入设计

- 当前 `设备管理 -> SDN控制器` 页增强
- 控制器集群总览与节点卡片
- SDN 资源标签页
- 配置下发标签页
- 网络告警标签页

### 本次不重做

- 控制器配置新增/编辑基础表单逻辑
- 后端控制器同步、诊断、告警、资源接口本身
- 独立拓扑或路径分析能力

## 现有 OneOPS 能力复用

当前页面已经具备较强基础能力，可直接复用：

- 控制器列表
- 测试连接
- 同步
- 最近快照
- 同步历史
- 诊断
- 告警
- 配置计划
- 资源列表

本轮设计重点不是重做这些能力，而是把它们组织成更像真实 SDN 运维台的结构。

## 页面定位

增强后的 `SDN控制器` 页面不是简单列表页，而是 `集群优先工作台`。

用户的默认认知顺序应为：

1. 当前控制器集群是否健康
2. 哪个节点有问题
3. 这套控制面当前管理了多少资源
4. 能否继续发起配置下发
5. 当前是否有 SDN 告警需要处理

## 顶层结构

采用：

- `顶部集群总览 + 下方标签页`

不采用纯标签页，也不新增左侧二级菜单，原因如下：

- 纯标签页会导致切走后丢失集群上下文
- 二级菜单会让页面层级变重，偏离当前页面体系
- 固定集群总览更符合“先 1 再 2/3/4”的顺序

## 顶部集群总览

默认显示四个摘要：

- `集群健康`
- `控制器节点数`
- `最近同步`
- `当前告警数`

设计原则：

- 只显示控制面最关键的运行摘要
- 不提前把资源统计混进总览首屏
- 让用户进页 3 秒内判断“控制面是否稳定”

## 标签结构

页面主体标签页顺序固定为：

1. `控制器集群`
2. `SDN资源`
3. `配置下发`
4. `网络告警`

默认打开：

- `控制器集群`

## 标签页 1：控制器集群

### 主视图

采用：

- `横向节点卡片`

不采用表格或关系图作为默认主视图，原因如下：

- 表格虽然密度高，但不够像集群页
- 关系图更强视觉化，但会过度复杂
- 横向节点卡片最适合表达 leader / follower / online / syncing

### 节点卡片字段

第一组：

- 节点名称
- 角色
- 在线状态

第二组：

- 最近同步时间
- 版本

第三组：

- 当前告警数
- 同步状态

### 节点卡片动作

每张卡片底部只保留三类核心动作：

- `测试连接`
- `同步`
- `诊断`

原因：

- 这是控制面运维最常用动作
- 不把资源、配置、告警入口塞入节点卡片
- 保持卡片聚焦在“节点维护”

### 最近操作结果区

在节点卡片区下方补一块：

- `最近操作结果`

默认只显示最近一次操作结果，不显示最近三次，也不按类型分组。

字段结构：

- 操作类型
- 结果状态
- 执行时间
- 结果摘要

用途：

- 给首页一个明确反馈闭环
- 避免用户执行完测试连接或同步后找不到结果

## 标签页 2：SDN资源

### 主视图

采用：

- `资源总览 + 统一资源表格`

不采用资源树或左树右表作为默认主视图，原因如下：

- 资源树更适合深入浏览，不适合首页
- 统一资源表格更利于演示不同资源对象统一纳管

### 资源总览

第一层摘要：

- `租户数`
- `VRF数`
- `网络数`
- `Endpoint数`

第二层摘要：

- `Segment数`
- `Switch数`
- `Port数`
- `Contract数`

选择这组摘要的原因：

- 能兼容主流 SDN 控制器的通用资源模型
- 既能表达逻辑资源，也能表达底层网络对象
- 不会过早偏向某个厂商的专有术语

### 统一资源表格

默认列顺序：

1. `资源名称`
2. `资源类型`
3. `所属租户`
4. `关键摘要`

`关键摘要` 的作用：

- 用一行信息概括资源当前状态或归属
- 不承载完整对象关系图
- 示例：
  - `VRF: prod-blue / Network: app-seg-a`
  - `Switch: leaf-03 / Port: Ethernet1/31`

设计原则：

- 表格是运维明细视图，不是资源树
- 重点在快速筛选和定位对象，而不是展示全关系

## 标签页 3：配置下发

### 默认主视图

采用：

- `新建配置计划入口`

而不是先展示计划列表或最近变更记录。

原因：

- 这一标签的核心是演示“SDN配置下发能力”
- 入口先给动作，后续再承接结果

### 主链路

配置下发主链路固定为：

1. `目标控制器`
2. `配置对象类型`
3. `目标资源`
4. `变更动作`
5. `变更参数表单`
6. `预检查`
7. `执行预览`
8. `直接执行`
9. `最近执行结果`

### 配置对象类型

默认支持：

- `租户`
- `VRF`
- `网络`
- `Segment`
- `Contract`

### 变更动作

默认动作：

- `新增`
- `修改`
- `删除`

### 主链路原则

- 先定控制器
- 再定对象类型
- 再定对象
- 再定动作
- 先预检查，再预览，再执行
- 执行后回落最近执行结果，而不是回抽象列表

### 为什么不先审批

当前设计更偏演示型执行台，而非复杂流程型变更平台，因此：

- 不把审批强行插入主链路
- 保留执行控制感
- 突出 SDN 配置下发闭环

## 标签页 4：网络告警

### 默认主视图

采用：

- `当前告警列表`

不采用告警概览看板作为首屏。

### 默认排序

- `严重性优先`

### 默认列

- `告警标题`
- `严重性`
- `状态`
- `发生时间`

### 点击告警后的右侧详情

默认先看：

- `告警对象详情`

而不是先给结论或原始内容。

原因：

- SDN 告警的关键在于它落在哪个租户、网络、Segment、控制器或资源对象上
- 一线运维更关心“影响对象”而不是抽象描述

## 页面数据真实性原则

### 控制器与节点命名

- 节点名称要像真实控制器集群成员
- 示例：
  - `nsx-mgr-01`
  - `nsx-mgr-02`
  - `nsx-mgr-03`
  - `aci-apic-01`

### 集群角色

- 必须存在明确角色区分
- 如：
  - `Leader`
  - `Follower`
  - `Standby`

### 状态值

状态不能只写“正常/异常”，要像真实控制面：

- `Online`
- `Syncing`
- `Degraded`
- `Disconnected`

### 同步与时间

- 最近同步时间必须和最近操作结果逻辑一致
- 同步失败时，要能对应到告警或诊断状态

### 资源对象

- 租户、VRF、网络、Segment、Contract 命名要有层次规律
- 不能写成泛泛的 `tenant-1`、`network-1`
- 要像真实环境：
  - `prod-shz`
  - `vrf-core-prod`
  - `seg-app-blue`
  - `contract-web-to-db`

### 配置计划

- 配置计划名称不能是空泛的“计划1”“变更A”
- 应该像真实操作：
  - `新增 prod-shz / vrf-core-prod / seg-app-blue`
  - `修改 contract-web-to-db 默认动作`

### 告警对象

- 告警必须能落到真实资源对象
- 不能只有抽象告警标题
- 例如：
  - `Segment seg-app-blue 同步失败`
  - `Contract contract-web-to-db 下发失败`
  - `Controller nsx-mgr-03 数据面连接异常`

## 文案原则

### 页面语言风格

- 偏运维控制台
- 不写长解释
- 多用状态、字段、动作

### 推荐动作语言

- `测试连接`
- `同步`
- `诊断`
- `新建配置计划`
- `执行预览`
- `直接执行`

### 避免词汇

- 不写 `demo`
- 不写 `样例`
- 不写 `智能赋能`
- 不写 `全景洞察`

## 视觉与交互原则

- 控制面优先，资源/配置/告警后置
- 集群总览始终保留在页顶
- 标签切换要轻量，不要像新开一个后台
- 节点卡片强调“谁是 leader，谁有问题”
- 配置下发强调“动作链路”
- 告警标签强调“当前问题”

## 设计结论

增强后的 `设备管理 -> SDN控制器` 页面，不再只是控制器配置表，而是一个按顺序组织好的 SDN 运维工作台：

1. 先看控制器集群
2. 再看资源
3. 再发配置
4. 再处理告警

这种结构既符合你要求的顺序，也与当前 OneOPS 已有能力贴合，适合作为前端演示页面进一步实现。
