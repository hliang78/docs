# OneOPS Device V2 导入设备清单边界测试设计

Date: 2026-06-22

## 背景

`导入设备清单` 页面已经完成流程产品化，但当前测试仍以零散 smoke 为主，主要覆盖：

- 表头映射与默认值补齐；
- 导入流程步骤与成功交接页跳转；
- 少量页面级 smoke 与合同验证。

这还不足以回答一个上线产品必须回答的问题：

- 哪些导入数据必须被拦截？
- 哪些导入数据允许先登记？
- 哪些导入数据允许继续采集？
- 只要页面允许提交且提交成功，设备是否一定能在 Device V2 管理页中看到？

本设计目标不是重做导入功能，而是为 `导入设备清单` 建立一套系统化、可重复执行、可定位问题层级的边界测试体系。

## 设计目标

### 产品目标

把 Device V2 导入页的边界行为固化成可执行标准，确保：

1. 不可登记的数据必须在导入页被拦截；
2. 可登记但不可采集的数据允许提交，并在管理页稳定可见；
3. 可登记且可采集的数据允许提交，并在管理页稳定可见；
4. 页面不能出现“提交成功但管理页查不到”的成功假象。

### 测试目标

测试体系必须同时回答三类问题：

1. 这条导入数据在导入页应该被判成什么？
2. 这批允许提交的数据，提交后后端返回是否与页面分类一致？
3. 真实用户提交成功后，是否真的能在 Device V2 管理页看到对应设备？

### 诊断目标

失败时必须能快速区分问题出在哪一层：

- 数据解析 / 字段归一化问题；
- 页面分类 / 提交放行问题；
- 提交合同 / handoff 结果问题；
- 管理页范围查询 / 可见性问题；
- 真实浏览器链路问题。

## 产品规则基线

本设计以以下产品规则为最高优先级：

### 分层处理规则

- `必须拦截`
  - 数据连“登记为设备”都做不到，导入页必须阻断，不能进入提交成功。
- `允许登记`
  - 数据足以形成设备记录，但不足以继续采集。
  - 这类数据允许提交；只要提交成功，就必须出现在管理页。
- `允许采集`
  - 在允许登记基础上，管理地址、正式接入条件或凭据链路已具备。
  - 这类数据允许提交；只要提交成功，就必须出现在管理页，并体现对应能力状态。

### 成功可见性合同

- “页面允许提交”不等于“设备已成功可见”；
- “提交接口返回成功”不等于“只有任务创建成功”；
- 只要页面显示提交成功，且该设备被计入成功设备集合，它就必须能在 Device V2 管理页中被本次范围查询到；
- 可登记但不可采集的设备，也必须可见，只是状态不同；
- 被拦截的数据绝不能混入成功交接结果或管理页成功范围。

## 范围

### 本次纳入设计

- Device V2 导入页的数据边界测试矩阵
- 导入提交结果与 handoff 返回的一致性校验
- “提交成功 -> 管理页可见” 的合同验证
- 少量高价值真实浏览器端到端回放

### 本次不纳入设计

- Device V2 入库底层执行逻辑重写
- 导入页 UI 再次大改版
- Device V2 管理页整体产品逻辑重做
- 导入链路的全量真实环境压测

## 总体方案

采用 `分层混合测试方案`，拆成三层：

1. `数据规则 smoke`
2. `提交合同 smoke`
3. `真实端到端 smoke`

不采用“只靠一个巨型浏览器脚本”的原因：

- 执行慢；
- 失败定位差；
- 很难清晰区分是页面分类、接口合同还是管理页可见性出了问题。

不采用“只做后端合同测试”的原因：

- 会漏掉 Excel / 中文旧表头 / 默认值补齐 / 页面拦截放行等前端规则层问题。

## 边界分类

### A. 必须拦截

典型样例：

- 全空行；
- 只有备注，没有任何设备识别字段；
- `in_band_ip=999.1.1.1`；
- `in_band_ip=中文IP`；
- `login_method=telnet-unknown`；
- 重复表头映射到同一字段且值冲突；
- 关键表头无法识别，导致无法形成设备记录；
- 关键身份字段全缺，且无法通过默认规则生成 `biz_code`。

预期：

- 行级状态为 `需修改` 或等价阻断态；
- 不应计入可提交或成功设备统计；
- 不应被允许进入成功交接结果；
- 不应在本次成功设备管理范围中出现。

### B. 允许登记但不可采集

典型样例：

- `biz_code + biz_name + site_name`；
- 只有 `in_band_ip`，无凭据；
- 有位置、平台、类别，但无管理地址；
- 中文旧表头导入成功，但只补齐到台账字段；
- 只有 SSH 用户名，没有密码；
- remark / 归属类别 / 虚拟区域等历史字段被合并入 remark。

预期：

- 行级状态为 `待入库` / `仅可先登记`；
- 允许进入提交；
- 提交成功后计入成功设备集合；
- handoff 中计入 `registry_only_count`；
- 在管理页中必须可见，但不应被标成可采集设备。

### C. 允许登记且可采集

典型样例：

- `in_band_ip + in_band_username + in_band_password`；
- `in_band_ip + snmp_community`；
- `platform + catalog + site + rack + credential_ref`；
- 依赖默认值补齐后成为可采集；
- 中文旧表头映射后达到可采集条件。

预期：

- 行级状态为 `待入库` / `可采集`；
- 允许进入提交；
- 提交成功后计入成功设备集合；
- handoff 中计入 `manageable_count`；
- 在管理页中必须可见，并体现其可采集能力状态。

### D. 混合批次

第一轮必须包含：

- `混合批次 A`
  - 2 条可采集 + 1 条仅登记
- `混合批次 B`
  - 1 条可采集 + 1 条仅登记 + 1 条应拦截

预期：

- 混合批次 A 应可提交成功，handoff 统计与分类一致；
- 混合批次 B 中的应拦截行必须留在草稿或问题集合中，不进入提交集合；
- 混合批次 B 的其余可提交行仍可提交成功；
- 问题行不能混进成功设备统计，不能伪装成整批成功；
- handoff 与管理页范围只统计真正进入提交并成功的设备。

## 测试资产设计

### 1. 数据规则 smoke

建议文件：

- `OneOPS-UI/scripts/d2-ingest-boundary-classification-smoke.ts`
- `OneOPS-UI/scripts/d2-ingest-boundary-fixtures.ts`

职责：

- 验证表头映射；
- 验证默认值补齐；
- 验证字段归一化；
- 验证行级分类：
  - `must_block`
  - `registry_only`
  - `manageable`
- 验证批次统计：
  - `draft`
  - `issues`
  - `registry_only`
  - `manageable`

这层不接浏览器，不依赖真实上传。

### 2. 提交合同 smoke

建议文件：

- `OneOPS-UI/scripts/d2-ingest-submit-visibility-contract-smoke.ts`

职责：

- 验证允许提交的批次返回有效 `task_id`；
- 验证 `success / partial_success / failed` 判定；
- 验证 `success_device_codes`、`manageable_count`、`registry_only_count` 一致性；
- 验证“仅登记”设备不能被错误算作失败；
- 验证含拦截行的批次不能伪装成整批成功；
- 验证成功设备集合可以形成后续管理页范围查询输入。

这层重点测合同，不要求真实浏览器上传文件。

### 3. 真实端到端 smoke

建议文件：

- `OneOPS-UI/scripts/d2-ingest-visibility-e2e-smoke.cjs`

职责：

- 上传样例；
- 检查页面统计；
- 检查 `确认提交` 按钮状态；
- 提交后进入 handoff；
- 验证 handoff 统计；
- 跳转 Device V2 管理页；
- 按 `task_id` 或 `codes` 验证设备可见性。

这层只保留高价值主线，避免脚本膨胀。

## 首批样例分配

### 数据规则 smoke

第一轮覆盖全部核心边界样例：

- 3 条必须拦截；
- 4 条允许登记；
- 3 条允许采集；
- 2 条混合批次样例。

### 提交合同 smoke

第一轮至少覆盖 4 组：

1. 纯可采集批次
2. 纯仅登记批次
3. 可采集 + 仅登记混合批次
4. 含拦截行的混合批次

其中第 4 组的合同口径固定为：

- 问题行不进入提交集合；
- 允许提交的行仍可形成有效 `task_id`；
- 成功设备统计只覆盖真正被提交的行。

### 真实端到端 smoke

第一轮至少覆盖 3 条：

1. 纯可采集批次
2. 纯仅登记批次
3. 可采集 + 仅登记混合批次

暂不把“含拦截行的混合批次”放进真实 E2E 的第一批主线，避免浏览器链路过早承担过多诊断职责。

## 每层断言

### 数据规则 smoke 断言

每条样例至少断言：

- 解析后字段值正确；
- 默认值按预期补齐；
- 行级分类正确；
- 行级提示正确；
- 批次统计正确。

### 提交合同 smoke 断言

每个允许提交的批次至少断言：

- 返回存在有效 `task_id`；
- `success` 或 `partial_success` 时，`success_device_codes` 非空；
- `manageable_count + registry_only_count` 与成功设备集合一致；
- “仅登记”设备不会被错误计入失败；
- 含拦截行的批次不会把问题行混入成功结果。

### 真实端到端 smoke 断言

每条浏览器主线至少断言：

- 上传后页面统计正确；
- `确认提交` 按钮状态正确；
- 提交后进入成功交接页；
- handoff 页的 `成功设备 / 可纳管 / 仅登记` 计数正确；
- 跳转管理页后，所有成功设备都能查到；
- “仅登记”设备可见但状态不同。

## 失败信号标准化

测试输出统一收敛成以下失败类型：

- `分类错误`
- `页面放行过宽`
- `页面拦截过严`
- `提交合同错误`
- `统计不一致`
- `成功但不可见`

这样可以让失败信息直接对应到应该排查的层级。

## 实施顺序

建议按以下顺序落地：

1. 先补 `fixtures + 数据规则 smoke`
2. 再补 `提交合同 smoke`
3. 最后补 `真实端到端 smoke`

原因：

- 这样能先把最便宜、最稳定、信息量最大的规则层兜住；
- 再用合同层证明“允许提交 -> 提交成功 -> 可形成管理页查询范围”；
- 最后再用少量真实 E2E 证明用户主线成立。

## 验收标准

这套边界测试落地后，团队应能够稳定证明：

1. 每条导入样例都会被稳定判成拦截、仅登记或可采集之一；
2. 允许提交的设备，提交成功后都会进入成功设备集合；
3. 所有成功设备都能在 Device V2 管理页被本次范围查询到；
4. 被拦截的数据不会被误算进成功交接或管理页成功范围；
5. 失败时可以快速定位是规则、合同、可见性还是浏览器主线问题。
