# OneOPS Device V2 导入设备清单产品化流程设计

Date: 2026-06-22

## 背景

当前 `导入设备清单` 页面已经具备较强的草稿解析、字段映射、历史任务回看和结果展示能力，但页面语义仍偏向内部工作台：

- 上传、检查、提交、结果交接混在同一页面语境中；
- 前端“提交成功”与后端真实入库成功语义不完全一致；
- 用户容易把 `task` 创建成功理解成“设备已经在 Device V2 管理页稳定可见”；
- 首页信息密度高，但主任务不够单一。

本设计目标不是重做一套新平台，而是在现有 Device V2 导入链路基础上，把页面收紧成正式上线产品可接受的流程页。

## 设计目标

### 产品目标

让用户可以按一个稳定、可解释、可交付的四步流程完成设备导入：

1. 上传清单
2. 修复阻断问题
3. 只确认真正可提交的设备
4. 在成功交接页进入 Device V2 管理页查看本次成功设备

### 语义目标

页面必须满足以下承诺：

- “允许提交”不等于“已经提交成功”；
- “提交成功”不等于“task 创建成功”；
- “提交成功”必须表示至少存在真实成功入库的设备；
- 只要页面提示提交成功，用户就必须能在设备管理页看到这批成功设备。

### 体验目标

- 首屏主任务唯一，不做运营大看板；
- 每一步都有清晰阶段边界；
- 用户在检查阶段先解决不能提交的问题；
- 结果交付阶段不再承担清单编辑职责。
- 页面保持紧凑、专业、少解释；
- 默认不使用大块解释性、描述型卡片；
- 优先使用步骤、按钮、状态、表格和行内反馈引导用户。

## 范围

### 本次纳入设计

- `导入设备清单` 页信息架构重组
- 四步流程表达重构
- 提交语义与结果语义收紧
- 成功交接页设计
- 与 Device V2 管理页的交接契约

### 本次不纳入设计

- Device V2 入库底层执行引擎大改
- Device V2 管理页整体重构
- 凭据平台、历史批次中心的独立产品化
- 跨模块权限模型重做

## 页面定位

增强后的 `导入设备清单` 页面不是：

- 运营控制台；
- 结果看板；
- 多功能入口首页。

它应当被定位为：

- `严格流程化的导入向导页`

默认顺序必须是：

1. 准备清单
2. 系统检查
3. 确认提交
4. 成功交接

## 总体方案

采用：

- `A1 严格流程页`
- `上传优先`
- `问题修复台`
- `只确认可提交设备`
- `成功交接页`

不采用：

- 首页看板优先
- 上传与手工新增双主入口
- 检查阶段先看全局总览
- 提交阶段确认整张清单
- 在同一页里同时承载准备、提交、结果、再次编辑的全流程

原因：

- 这些方案会继续放大当前“上传/提交成功/可见性”语义不一致的问题；
- 正式上线产品更需要阶段清晰和结果承诺，而不是单页能力堆叠。

## 顶层流程

### Step 1：准备清单

主目标：

- 上传一张设备清单，并启动系统检查

页面主动作：

- `上传清单`

页面辅动作：

- 下载模板
- 查看字段说明
- 加载历史清单
- 手工新增一台设备
- 去准备凭据

设计原则：

- 首屏默认主路径只能有一个；
- `上传清单` 作为批量导入默认入口；
- `新增一台设备` 保留，但不与上传入口争同等视觉权重。

首屏指标只保留三类：

- 清单设备数
- 需修改数
- 可提交数

不在首屏提前强调：

- 可采集
- 可同步 V1
- 结果看板
- 最近任务结果

原因：

- 这些信息要么属于检查阶段，要么属于提交后阶段，不应抢首屏主任务。
- 首屏必须更像正式产品操作面，而不是说明页。

### Step 2：系统检查

主目标：

- 告诉用户哪些设备当前不能提交、为什么不能提交、应该如何修复

页面形态：

- `问题修复台`

优先展示：

- 阻断问题设备列表
- 问题类型分组
- 可提交设备数
- 批量修复或返回编辑入口

不优先展示：

- 大量全局状态卡
- 提交后结果摘要
- 历史批次结果

设计原则：

- 检查阶段的任务不是“看”，而是“修”；
- 用户在这一页必须能回答两个问题：
  - 为什么还不能提交？
  - 我下一步该改什么？
- 页面默认不出现大段说明文字；
- 错误原因和修复动作优先以行内提示、状态标签、问题列表表达。

### Step 3：确认提交

主目标：

- 只确认真正会被提交的设备

页面形态：

- `可提交设备确认页`

只展示：

- 本次可提交设备数
- 其中可采集设备数
- 仅登记后补充设备数
- 提交后的系统动作说明

明确说明：

- 第 2 步未修复的问题设备不会被提交；
- 它们会继续保留在草稿中，等待后续修复。

不采用“整张清单一起确认”的原因：

- 用户很容易误解为整张清单都会进入清册；
- 这会直接破坏“提交成功”的可信语义。

### Step 4：成功交接页

主目标：

- 确认本次成功入库结果，并把用户交接到 Device V2 管理页

页面形态：

- `成功交接页`

核心内容：

- 成功入库设备数
- 其中可采集设备数
- 需要后续补充的设备数
- 进入 Device V2 管理页查看本批设备
- 返回修复剩余问题设备
- 开始一张新清单

设计原则：

- 结果页只处理“提交后的交接”；
- 不再继续承担草稿编辑职责；
- 用独立页面语境明确宣布：准备阶段已经结束，当前进入交付阶段。
- 成功页文案保持短促，只强调结果和下一步动作，不铺陈解释。

## 信息架构

### 流程页职责

流程页只负责：

- 设备清单准备
- 阻断问题修复
- 可提交设备确认

流程页不负责：

- 历史结果中心
- 提交后持续运营看板
- 长期批次管理

### 成功交接页职责

成功交接页只负责：

- 成功结果确认
- 跳转 Device V2 管理页
- 返回处理剩余问题设备

### 历史任务能力

历史任务仍可保留，但降级为辅助能力：

- 放在折叠区或二级入口；
- 不占首页主视觉；
- 不与本次清单主流程争夺注意力。

### 帮助信息策略

默认策略：

- 不展示大块帮助卡片；
- 不用说明型面板占据首屏主工作区。

帮助信息只在以下情况下出现：

- 行内错误提示
- 阻断问题修复提示
- 上传失败或提交失败反馈

次级帮助如果仍需保留，应采用：

- 抽屉
- 折叠入口
- 字段级 tooltip

而不是常驻首屏的大段说明区域。

## 真实成功语义

这是本设计最关键的契约。

### 前端允许提交

含义：

- 当前清单满足最小提交前置条件；
- 用户可以进入确认提交步骤。

不代表：

- 后端一定成功创建 Device V2 记录；
- 提交后一定有设备进入管理页。

### 提交成功

含义必须收紧为：

- 至少存在 `created + updated > 0` 的真实成功入库设备；
- 前端能够拿到这批成功设备的真实交接依据；
- 用户可以从成功交接页进入 Device V2 管理页并看到它们。

以下情况不能使用“提交成功”：

- 只创建了 ingest task，但 `created = 0` 且 `updated = 0`
- 只有失败或全部跳过
- 无法稳定定位成功设备到 Device V2 管理页

### 部分成功

如果存在：

- 部分设备成功入库
- 部分设备失败或被阻断

页面应表达为：

- `部分提交成功`

并且成功交接页只能围绕成功设备组织，问题设备回到草稿继续修复。

## Device V2 管理页交接契约

### 目标

只要前端显示“提交成功”或“部分提交成功”，本次成功设备就必须能在 Device V2 管理页看到。

### 交接依据

优先级如下：

1. 本次成功设备的真实 `device_code`
2. 与本次提交绑定的稳定 `task_id` / ingest 标识作为兜底锚点

### 管理页要求

管理页必须支持：

- 按真实成功 `codes` 过滤
- 在 `codes` 缺失或部分失准时，按本次 ingest 交接锚点兜底恢复成功设备集合

### 页面文案要求

成功交接页不得使用模糊表达：

- “请用业务编码、设备名称或 IP 找到设备”

必须改成更强承诺的表达：

- “进入设备管理页查看本次成功入库设备”

## 状态与错误处理

### 上传阶段

- 上传失败：停留在 Step 1，给出明确错误原因和重试入口
- 模板/旧表头兼容：显示自动转换说明，但不把“已转换”误表达成“已提交”

### 检查阶段

- 阻断问题：停留在 Step 2，优先展示问题设备和修复动作
- 没有阻断问题：允许进入 Step 3

### 提交阶段

- `created + updated = 0`：显示提交失败，不进入成功交接页
- 部分成功：进入成功交接页，但同时保留“返回修复剩余问题设备”入口
- 全部成功：进入成功交接页

### 交接阶段

- 如果管理页交接条件不完整：不能显示成功交接完成
- 如果管理页无法稳定命中本批成功设备：页面应退回结果错误态，而不是假装完成

## 测试与验证要求

### 流程级验证

必须覆盖以下用户旅程：

1. 空状态进入 -> 上传清单 -> 检查 -> 提交 -> 成功交接
2. 上传后出现阻断问题 -> 修复 -> 提交 -> 成功交接
3. 部分成功提交 -> 交接成功设备 -> 返回继续修剩余问题设备
4. 提交失败 -> 停留在结果错误态，不出现成功承诺

### 语义级验证

必须验证：

- 前端不会在 `created + updated = 0` 时提示成功
- 成功交接页只能承接真实成功设备
- 进入管理页后，用户能看到本次成功设备

### 页面级验证

必须验证：

- 首屏只有一个主入口
- 检查阶段首先呈现阻断问题
- 提交阶段只确认可提交设备
- 成功交接页与准备流程页职责清晰分离
- 首屏不存在大块解释性、描述型卡片
- 用户可以仅通过步骤、按钮、表格和行内反馈理解下一步动作

## 实施建议

实施上建议分三层推进：

1. 先收紧前端流程表达和文案语义
2. 再补强提交成功判定与管理页交接契约
3. 最后拆出成功交接页并补足结果闭环

这样可以先解决最严重的产品语义问题，再逐步把真实成功与管理页可见性绑定起来。

## 设计结论

最终采用的产品化方向是：

- `A1 严格流程页`
- `上传优先`
- `问题修复台`
- `只确认可提交设备`
- `成功交接页`

这套方案的核心价值在于：

- 让每一步都只承担一个清晰目标；
- 让“允许提交”“提交成功”“管理页可见”三个语义重新对齐；
- 把当前偏内部工作台的页面，收紧成正式上线产品可接受的流程产品。
