# EVE-NG 第一条真实场景运行手册：`bridge + boundary + pnet0`

更新时间：2026-06-27  
适用环境：`192.168.100.20` 上的 EVE-NG Community `6.2.0-4`

## 1. 这条场景为什么最先做

第一阶段的目标不是“把功能点都点一遍”，而是尽快建立一条可以反复复用的真实设备验证链。

`bridge + boundary + pnet0` 是最适合做第一条场景的原因：

1. 它足够小，容易重复执行
2. 它能覆盖内部网络、边界设备、外联网络三层关系
3. 它天然适合承载采集、监控、策略、拓扑、脚本等多类能力
4. 它最容易暴露薄弱环节，例如接口挂线、启动状态、外联可达性、配置解析和策略命中

## 2. 场景目标

这条场景的直接目标不是一次性验证所有功能，而是先建立第一条“真实设备能力链”：

1. 拓扑能被稳定创建和清理
2. 两端测试节点之间的内部二层可达
3. 边界设备 inside / outside 双口行为明确
4. `bridge` 到 `pnet0` 的边界路径可验证
5. OneOPS 可以围绕这个边界设备逐步挂接采集、监控、策略、拓扑和脚本测试

## 3. 最小拓扑

```text
pc-a ----\
          \
           [Net_inside: bridge] ---- [boundary] ---- [Net_outside: pnet0]
          /
pc-b ----/
```

角色说明：

1. `pc-a`
   内网源端，用于发起探测流量
2. `pc-b`
   内网对端，用于验证二层/三层和策略路径
3. `boundary`
   真实目标设备，建议优先替换成要测试的防火墙或路由器
4. `Net_inside`
   内部实验二层网络
5. `Net_outside`
   对外网络，映射 `pnet0`

## 4. 推荐节点选择

### 4.1 最小节点组合

第一轮建议只上四个元素：

1. `pc-a`: `vpcs`
2. `pc-b`: `vpcs`
3. `boundary`: 一台真实设备镜像
4. `Net_outside`: `pnet0`

### 4.2 `boundary` 优先顺序

根据当前测试重点选：

1. 防火墙优先
   如果当前更关注配置采集、策略生成、策略查询、ACL/NAT 边界
2. 路由器优先
   如果当前更关注接口/路由/邻居采集、监控和拓扑
3. 三层交换机次之
   如果当前更关注 VLAN、接口、LLDP、二三层拓扑

## 5. 地址规划

### 5.1 内网固定地址

建议固定用下面这组，便于文档、脚本和证据复用：

1. `boundary inside`: `10.10.20.1/24`
2. `pc-a`: `10.10.20.20/24`，网关 `10.10.20.1`
3. `pc-b`: `10.10.20.21/24`，网关 `10.10.20.1`

### 5.2 外联地址两种模式

`pnet0` 的外联地址建议按环境成熟度分两种模式：

1. 模式 A：先只验证 outside 接口接入
   适用：还没有固定外联地址规划时
   做法：outside 口使用 DHCP 或仅验证链路、ARP、邻居状态
2. 模式 B：验证真正出网链路
   适用：已经有固定的 `pnet0` 地址和网关时
   做法：outside 口配置静态地址、默认路由和必要 NAT

## 6. 执行前固定预检查

每次执行前都先做这几个动作：

1. 确认使用独立 Lab，禁止直接在生产性 Lab 上改线
2. 如果当前 EVE 不是空白环境，创建节点后优先选择高 ID 节点做数据面验证，规避控制台端口冲突
3. 新建 `bridge` 后立刻 `GET /api/labs/<lab>/networks` 回读，确认网络真的存在
4. 查看宿主机 `brctl show`
5. 查看宿主机 `bridge link`
6. 查看宿主机 `ip tuntap list`
7. 确认没有上一轮残留的孤儿 `vunl...` 口干扰排障

## 7. 推荐搭建顺序

1. 新建 Lab
2. 新建 `pc-a`
3. 新建 `pc-b`
4. 新建 `boundary`
5. 新建 `Net_inside`，类型 `bridge`
6. 新建 `Net_outside`，类型 `pnet0`
7. 连接 `pc-a -> Net_inside`
8. 连接 `pc-b -> Net_inside`
9. 连接 `boundary inside -> Net_inside`
10. 连接 `boundary outside -> Net_outside`
11. 回读所有接口挂线结果
12. 启动节点
13. 配置地址
14. 执行分层验证

## 8. 分层验证步骤

### 8.1 第 0 层：拓扑和接口存在性

目标：

1. 先证明拓扑对象本身正确存在

检查点：

1. Lab 可被 `GET /api/labs/<lab>` 正常读取
2. `Net_inside` 和 `Net_outside` 都能被网络列表回读
3. `pc-a`、`pc-b`、`boundary` 接口挂线结果与设计一致

### 8.2 第 1 层：内部二层通信

目标：

1. 先证明 `Net_inside` 本身可用

步骤：

1. 在 `pc-a` 配 `10.10.20.20/24`
2. 在 `pc-b` 配 `10.10.20.21/24`
3. `pc-a -> pc-b` 连续 `ping`
4. `pc-b -> pc-a` 连续 `ping`

通过标准：

1. 双向 `ping` 成功

### 8.3 第 2 层：边界设备 inside 口接入

目标：

1. 证明边界设备 inside 口已经真正加入内部网

步骤：

1. 给 `boundary inside` 配 `10.10.20.1/24`
2. 从 `pc-a` `ping 10.10.20.1`
3. 从 `pc-b` `ping 10.10.20.1`

通过标准：

1. 两台测试节点都能到 `10.10.20.1`

### 8.4 第 3 层：边界设备 outside 口接入 `pnet0`

目标：

1. 证明边界设备 outside 口已经接到外联网络

步骤：

1. 给 outside 口配置 DHCP 或静态地址
2. 检查接口 `up/down`
3. 检查 ARP、邻居、默认路由或下一跳

通过标准：

1. outside 口处于可用状态
2. 可以观察到外联方向的邻居或路由证据

### 8.5 第 4 层：内网经边界访问外联目标

目标：

1. 真正证明“`bridge` 通过边界接入 `pnet0`”

步骤：

1. `pc-a` 和 `pc-b` 默认网关指向 `10.10.20.1`
2. 在边界设备上配置静态路由或 NAT
3. 从 `pc-a` 访问外联目标
4. 必要时从 `pc-b` 重复一次

通过标准：

1. 内网主机可以经边界到达 `pnet0` 侧目标

## 9. 这条场景如何挂接 OneOPS 能力链

这一步才是它最有价值的地方。场景搭起来之后，不是停在连通性，而是把 OneOPS 各条能力链逐段挂上去。

### 9.1 设备采集

验证点：

1. 设备能否被纳管
2. 设备基础身份是否稳定
3. 接口、路由、对象或配置是否能被采上来

重点观察：

1. 首次采集是否成功
2. 重复采集结果是否稳定
3. 失败时边界是否清晰

### 9.2 监控推送

验证点：

1. 监控任务能否下发到目标设备
2. 指标是否按预期写入
3. 大量任务时行为是否退化

重点观察：

1. 任务创建成功不代表监控真正生效
2. 要结合设备侧流量、接口计数器和平台侧时序数据共同验证

### 9.3 防火墙配置解析

如果 `boundary` 是防火墙，至少要验证：

1. 接口和安全域解析
2. 地址对象、服务对象解析
3. 策略条目解析
4. NAT 或路由关键字段解析

重点观察：

1. 解析结果是否和控制台实际配置一致
2. 缺失项是否能明确归因到设备差异或解析薄弱点

### 9.4 策略生成和策略查询

如果 `boundary` 是防火墙，建议固定验证三种流：

1. `pc-a -> pc-b`
   验证内部区域流量
2. `pc-a -> 外部目标`
   验证出网策略和 NAT
3. 一条明确应拒绝的流
   验证策略查询和边界命中

### 9.5 拓扑生成

验证点：

1. 平台是否能识别 `pc-a / pc-b / boundary / pnet0` 这一最小结构
2. 设备接口关系是否和 EVE 设计一致
3. 路由或邻居信息是否支持补充链路解释

### 9.6 自动化脚本

验证点：

1. 先用只读脚本验证登录、命令执行和返回解析
2. 再做可回滚的小变更脚本

第一轮建议只做：

1. 查看接口状态
2. 查看路由表
3. 查看安全策略或配置摘要

不要第一轮就做大范围改配。

## 10. 这条场景最容易暴露的薄弱点

这条场景设计出来就是为了更早暴露问题。重点盯下面几类：

1. 拓扑对象创建成功，但网络实际不存在
2. 节点显示启动，但控制台或真实进程并未就绪
3. `bridge` 下存在残留孤儿 tap 口，导致排障结论失真
4. inside 通、outside 不通，暴露边界设备外联配置问题
5. 设备可登录但解析不完整，暴露适配链薄弱点
6. 监控任务创建成功但无有效数据，暴露下发与生效脱节
7. 策略查询能返回结果，但和真实流量命中不一致

## 11. 通过标准

第一条场景要算通过，至少满足下面六条：

1. 拓扑可重复创建
2. 拓扑可重复清理
3. `pc-a` 和 `pc-b` 内部互通
4. `boundary inside` 和 `boundary outside` 行为都可解释
5. 至少挂上一条 OneOPS 能力链并完成验证
6. 每个失败点都能给出清晰边界，而不是“看起来不对”

## 12. 证据清单

每次执行建议固定保留：

1. Lab 名称和创建时间
2. 节点、网络、接口 API 回读结果
3. `pc-a`、`pc-b`、`boundary` 控制台关键输出
4. 连通性测试结果
5. OneOPS 采集、监控、策略或脚本执行结果
6. 宿主机 `brctl show`
7. 宿主机 `bridge link`
8. 宿主机 `ip tuntap list`
9. 清理后的环境状态

## 13. 收尾和清理

执行结束后，固定做下面几步：

1. 停止场景节点
2. 删除测试 Lab
3. 回查 `/api/status`
4. 回查宿主机 bridge 成员
5. 确认没有残留 `vunl...` 口
6. 把“本次验证通过点”和“本次发现边界”写回对应测试记录

## 14. 下一步推荐

这条通用场景跑通后，建议马上分裂成三条设备家族变体：

1. `bridge + firewall + pnet0`
   优先打通配置采集、策略解析、策略查询和策略生成
2. `bridge + router + pnet0`
   优先打通接口、邻居、路由、监控和自动化脚本
3. `bridge + server + boundary + pnet0`
   优先打通服务器采集、监控、脚本和网络联动
