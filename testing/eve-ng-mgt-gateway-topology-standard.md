# EVE-NG 管理网关拓扑标准

更新时间：2026-06-27  
适用环境：`192.168.100.20` 上的 EVE-NG Community `6.2.0-4`

## 1. 目的

这份标准用于统一 OneOPS 第一阶段真实设备测试的实验拓扑底座。

它解决的不是“某一条测试怎么连”，而是：

1. 所有设备如何稳定接入外部网络
2. 所有设备如何获得统一的带外管理平面
3. 每次测试结束后如何彻底清理
4. 后续不同测试目标下，业务网络设备如何在同一套底座上自由组合

## 2. 角色定义

这套拓扑只分两类设备：

1. 网关设备
   一台思科交换机，只负责准备基础环境，不参与具体业务能力测试。
2. 业务网络设备
   每次测试真正要验证的设备，可以是交换机、路由器、防火墙、服务器或其他网络节点。

## 3. 网关设备标准

### 3.1 基本要求

网关设备固定为一台思科交换机，统一承担以下职责：

1. 第一个接口连接 `pnet0`
2. 提供外部网络连通
3. 提供整个实验拓扑的带外管理网关
4. 为所有业务网络设备提供统一的带外管理二层接入

网关设备的具体创建和配置步骤，统一参照：

1. [EVE-NG Cisco 网关交换机标准操作手册](/Users/huangliang/project/OneOPS-ALL/docs/testing/eve-ng-cisco-gateway-standard-operation.md)

### 3.2 接口规划

网关设备初始配置 `20` 个接口。

接口用途如下：

1. 第一个接口
   连接 `pnet0`
2. 其余接口
   全部属于 `vlan 1`
   用于连接所有业务网络设备的带外管理接口

补充说明：

1. 当前 EVE 的 Cisco IOL 模板定义里，`1` 个 `ethernet` 模块等于 `4` 个接口。
2. 因此如果采用 Cisco IOL 作为网关设备，接口数应按 `ethernet=5` 来创建，才能得到 `20` 个以太网接口。
3. 这一点已经通过本次临时探测的宿主机进程参数得到实证，不是仅凭模板说明推断。

### 3.3 MGT VRF 规划

网关设备需要配置一个 `MGT VRF`。

固定要求如下：

1. 第一个接口加入 `MGT VRF`
2. 第一个接口地址：`192.168.100.150/24`
3. 第一个接口网关：`192.168.100.254`
4. `interface vlan1` 也加入 `MGT VRF`
5. `interface vlan1` 地址：`172.32.2.254/24`

这里 `172.32.2.254/24` 是整个实验拓扑中所有网络设备的带外管理网关。

## 4. 业务网络设备标准

### 4.1 基本要求

所有业务网络设备都遵循同一套初始接口标准：

1. 每台设备初始准备 `20` 个网络接口
2. 最后一个接口固定用作带外管理接口
3. 带外管理接口接入网关设备的 `vlan 1`
4. 带外管理接口加入各自设备上的 `MGT VRF`

当前已确认的厂商例外：

1. `Ruijie Firewall (Ruijiefirewall-V1.03)` 不适用“最后一个接口 + MGT VRF”规则。
2. 锐捷防火墙当前标准管理口固定为 `Ge0/0`。
3. 锐捷防火墙当前管理平面固定留在 `main`，不纳入 `MGT VRF` 规划。

### 4.2 带外管理地址规则

所有业务网络设备的带外管理地址从 `172.32.2.0/24` 中分配。

固定约束如下：

1. 每台设备带外管理地址必须唯一
2. 所有业务设备的 `MGT VRF` 默认网关统一为 `172.32.2.254`
3. 所有 OneOPS 对设备的 SSH、采集、监控、自动化脚本优先走这个带外平面

对锐捷防火墙的补充约束：

1. 地址仍从 `172.32.2.0/24` 分配。
2. 但地址承载接口固定为 `Ge0/0`。
3. 该地址位于 `main`，不是 `MGT VRF`。

## 5. 业务连线规则

### 5.1 网关设备与业务设备的关系

网关设备不参与业务能力测试。

它只负责：

1. `pnet0` 外联
2. 带外管理网

业务网络设备之间的业务互联，根据每次测试目标单独决定。

### 5.2 业务设备之间的连线

每次测试中：

1. 参与测试的业务设备类型可以不同
2. 业务互联关系可以不同
3. 是否二层互联、是否三层互联，都由当前测试目标决定

这意味着：

1. 管理平面固定
2. 业务平面可变

## 6. 接口资料和登录基线

由于不同厂商、不同设备的接口命名不一样，所以每一类业务设备都必须预先准备两类资料。

### 6.1 接口名称列表

每种设备都要有一份明确的接口名称列表，例如：

1. 第 1 到第 20 个接口在设备上的真实命名
2. 哪个接口对应最终带外管理口
3. 哪些接口适合做三层互联
4. 哪些接口适合做二层互联

### 6.2 初始登录和配置模板

每种设备都要准备一份最小初始化配置，至少包含：

1. 登录方式
2. 用户名和密码初始化
3. `MGT VRF` 定义
4. 带外管理接口配置
5. 基础 SSH 或远程登录配置
6. 基础路由或动态路由前置配置

统一模板结构，参照：

1. [EVE-NG 设备初始化模板标准](/Users/huangliang/project/OneOPS-ALL/docs/testing/eve-ng-device-initialization-template-standard.md)

## 7. 三层互联地址规则

如果设备之间采用三层互联，则业务地址从 `172.32.64.0/24` 开始使用。

当前约定规则为：

1. 以节点 ID 作为地址最后一位
2. 例如 `SW10` 与 `SW50` 之间互联时，地址可以使用：
   `172.32.64.10/24`
   `172.32.64.50/24`

## 8. 动态路由规则

默认通过 `OSPF` 作为动态路由协议。

统一原则如下：

1. 除 `MGT VRF` 外，其余业务接口都加入 `OSPF`
2. 设备上采用 `network 0.0.0.0 0.0.0.0` 这类全量纳管方式时，必须明确限制在非 `MGT VRF`
3. 不同厂商命令语法可以不同，但语义要一致

补充说明：

1. 锐捷防火墙当前不纳入这条 `MGT VRF` 隔离规划。
2. 如果后续在锐捷上做动态路由或业务互联测试，必须单独记录其 `main` 管理面边界，不能默认套用“管理 VRF 已隔离”的假设。

目标是让所有业务互联接口自动进入同一套动态路由平面，而管理平面独立隔离。

## 9. 收尾和清理标准

每次测试结束后，固定执行下面的清理动作：

1. 关闭所有设备
2. 删除整个拓扑
3. 检查所有动态网卡
4. 清除残留动态网卡
5. 回查宿主机 bridge 状态

宿主机侧固定检查项包括：

1. `brctl show`
2. `bridge link`
3. `ip tuntap list`

## 10. 这套标准的价值

这套标准一旦固定下来，后续每条测试场景就都不需要重新定义管理平面。

后续变化只集中在：

1. 哪些业务设备加入测试
2. 业务设备之间怎么连
3. 要验证哪条能力链

这会让测试设计从“临时拼拓扑”变成“在统一底座上替换业务设备和业务连线”。

## 11. 当前需要明确记录的边界

这套规则里有一个很重要的边界必须写清楚：

如果一台业务设备在同一个拓扑里同时参与多条三层互联，那么只用“节点 ID 作为地址最后一位”会发生地址冲突。

例如同一台 `SW10`：

1. 与 `SW50` 互联时使用 `172.32.64.10/24`
2. 与 `FW30` 再互联时，如果仍然使用 `172.32.64.10/24`

那么同一台设备的两个不同三层接口就会出现重复地址，这是不成立的。

因此，当前规则更适合作为：

1. 单设备只参与一条三层互联的最简场景
2. 或者单拓扑里业务设备数量较少、三层链路较少的第一阶段场景

如果后续进入多链路、多设备、多对等的复杂场景，建议把三层地址规则升级为：

1. 每条链路一个独立子网
2. 节点 ID 只作为主机位编码参考

这样才能避免地址冲突。

另一个当前边界也要明确：

1. “Cisco IOL 可起成 20 口”已经验证
2. “`i86bi_LinuxL2-AdvEnterpriseK9-M_152_May_2018.bin` 满足 `MGT VRF + interface vlan1 + 上联口 + SSH`”也已经完成 CLI 级实证
3. 但如果后续改用其它 Cisco IOL 镜像，仍然不能自动假设它们具备完全相同的命令行为

因此，在正式把其它 Cisco IOL 镜像固定为网关设备之前，仍然必须补一条专项验证：

1. 启动目标镜像
2. 回看真实接口命名
3. 验证 `VRF` 命令面
4. 验证 `interface vlan1` 配置面

## 12. 第一阶段建议执行方式

第一阶段建议严格按下面顺序推进：

1. 先固定网关设备和管理平面
2. 再为每类业务设备补齐接口名称表和初始化配置
3. 再按测试目标替换业务设备和业务连线
4. 每次执行后彻底清理
5. 把通过点和边界回写到测试记录

这会让后续采集、监控、拓扑、策略、脚本等测试都建立在同一个稳定实验底座上。
