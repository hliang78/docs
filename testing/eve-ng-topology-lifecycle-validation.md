# EVE-NG 拓扑生命周期操作验证记录

更新时间：2026-06-27  
验证环境：`192.168.100.20` 上的 EVE-NG Community `6.2.0-4`

## 1. 目的

这份记录只回答一件事：在 OneOPS 第一阶段真实设备测试中，如何把 EVE-NG 作为“可重复拓扑底座”来安全使用。

本次重点验证了以下动作：

1. 新建拓扑（Lab）
2. 新建节点
3. 新建设备间网络
4. 建立接口连线
5. 停止设备
6. 断开接口
7. 删除网络
8. 删除节点
9. 删除整个拓扑

## 2. 已验证结论

### 2.1 可直接使用的动作

以下动作已经在隔离临时 Lab 中实测通过：

1. `POST /api/labs` 可以新建 Lab
2. `POST /api/labs/<lab>/nodes` 可以新建节点
3. `POST /api/labs/<lab>/networks` 可以新建 `bridge` 和 `pnet` 网络
4. `PUT /api/labs/<lab>/nodes/<id>/interfaces` 可以把节点接口挂到网络上
5. `GET /api/labs/<lab>/nodes/<id>/interfaces` 可以回读接口挂线结果
6. `GET /api/labs/<lab>/nodes/<id>/stop` 可以触发节点停止
7. `PUT /api/labs/<lab>/nodes/<id>/interfaces` 传空字符串可以断开接口
8. `DELETE /api/labs/<lab>/networks/<id>` 可以删除网络
9. `DELETE /api/labs/<lab>/nodes/<id>` 可以删除节点
10. `DELETE /api/labs/<lab>` 可以直接删除整个拓扑文件

### 2.2 已完成的数据面验证

2026-06-27 的补充验证里，已经完成“两台新建设备之间的真实数据面通信”验证。

验证链路如下：

1. 新建隔离 Lab
2. 新建 21 台 `vpcs`
3. 选择 `id=20` 和 `id=21` 作为目标测试节点
4. 新建 `bridge`
5. 把 `20/21` 两端接口挂到同一个 `bridge`
6. 显式启动 `20/21`
7. 通过控制台分别配置 IP
8. 执行双向 `ping`

实际验证结果：

1. `pc20` 配置 `10.10.20.20/24` 后可稳定回读
2. `pc21` 配置 `10.10.20.21/24` 后可稳定回读
3. `pc20 -> pc21` 连续 `3` 次 `ping` 全部成功
4. `pc21 -> pc20` 连续 `3` 次 `ping` 全部成功

这说明在当前 EVE 环境中：

1. API 建立的接口连线不仅能写入拓扑
2. 只要节点真实启动并拿到可用控制台
3. 两个新建设备之间的数据面通信是可以打通的

结论上现在可以明确区分：

1. **已验证**：拓扑层面的节点、网络、接口关系可以被 API 创建和回读
2. **已验证**：在满足运行态前置条件时，新建设备之间的真实数据面通信可以打通

### 2.3 已确认的关键坑点

#### A. `bridge` 网络不是“创建成功就一定真的存在”

如果只传：

```json
{
  "name": "Net_pc1_pc2",
  "left": 358,
  "top": 192,
  "type": "bridge"
}
```

接口返回会是 `201 success`，但 `GET /api/labs/<lab>/networks` 可能仍然读不到这条网络，随后接口挂网会报：

```text
设备不能连接，无效的线缆ID (20033)
```

本环境下，`bridge` 网络至少要带：

```json
{
  "name": "Net_pc1_pc2",
  "left": 358,
  "top": 192,
  "type": "bridge",
  "visibility": 1
}
```

建议同时显式带上 `icon`：

```json
{
  "name": "Net_pc1_pc2",
  "left": 358,
  "top": 192,
  "type": "bridge",
  "visibility": 1,
  "icon": "lan.png"
}
```

#### B. 节点列表里的 `status` 不能直接当“停机成功证据”

本次用 `vpcs` 节点验证时：

1. 节点创建后 `GET /nodes` 里就是 `status: 2`
2. 调用停止接口返回成功后，`GET /nodes` 里仍然可能继续显示 `status: 2`

这说明在当前环境下，至少对 `vpcs` 来说，`GET /nodes` 返回的 `status` 不足以单独证明节点已经真正停机。

因此，后续真实设备拓扑测试里：

1. `stop` 接口成功只能算“已发起停止动作”
2. 对防火墙、路由器、交换机等真实虚拟设备，停机确认应增加第二证据
3. 第二证据建议来自控制台、设备状态轮询或后续接口行为验证

#### C. 当 EVE 上存在常驻实验室时，低 ID 节点会带来控制台端口冲突风险

这是一条已经在本环境中实测过的历史风险：

1. 新建 Lab 中的低 ID 节点，例如 `id=1/2`
2. 会得到 `32769/32770` 这类低位控制台端口
3. 如果 EVE 上已经有其它常驻实验室或残留运行节点，这些端口就可能被占用
4. 结果就是：
   - API `start` 可能返回成功
   - 但新节点不一定真正启动
   - 控制台 URL 也可能连到其它实验节点

本次最终能完成真实互 ping，依赖的是下面这个规避条件：

1. 在临时 Lab 中把目标节点放到更高 ID
2. 例如 `20/21`
3. 让控制台端口变成 `32788/32789`
4. 避开可能存在的低位端口占用

因此，这条检查规则应该保留：

1. 如果 EVE 上有常驻实验室，必须做控制台端口冲突规避
2. 如果 EVE 已被清空为测试专用空白环境，这条风险会明显降低，但仍建议保留启动后控制台可达验证

#### D. 删除 Lab 后，宿主机可能残留孤儿 tap 口

补充排查中还确认了一个宿主机层面的风险：

1. Lab 文件删除后
2. 相关进程虽然可能已经退出
3. 但对应的 `vunl...` tap 口不一定会自动从宿主机上清掉
4. 这些孤儿口仍然可能继续挂在原来的 `vnet...` bridge 下面

这会造成两个后果：

1. 宿主机抓包时，看起来 bridge 下面还有很多“设备口”
2. 排障时容易误以为 bridge 正在给不存在的节点转发，或者误判 bridge 自身异常

本次实际清理结果里，`vnet0_1` 在删除测试 Lab 之后一度仍然挂着：

1. `vunl0_20_0`
2. `vunl0_21_0`
3. `vunl0_2_0`

而这些口已经：

1. 没有对应 Lab 文件
2. 没有对应 tmp 引用
3. 没有对应运行进程

最终需要在宿主机上手动删除这些孤儿 tap 口，`vnet0_1` 才恢复到只挂真实成员口的状态。

因此，后续如果宿主机侧继续使用抓包、`bridge link`、`brctl show` 做网络排障，必须把“bridge 下是否有孤儿 tap 口”列为固定检查项。

#### E. API 能连线，不代表平台层应该允许“在线改线”

本次 `vpcs` 验证中，节点即使显示为运行态，接口挂网也能成功。  
但前端 UI 逻辑明确要求“先停机，再连线”。

所以平台实现和测试标准都应采用更保守的约束：

1. 设计上先停机再改线
2. API 自动化也先停机再改线
3. 不要把 `vpcs` 这次的宽松行为当成所有设备通用规则

#### F. 节点删除后，`tmp` 覆盖目录不一定自动清理

这轮在继续攻关 Cisco ASAv 时，又确认了一条会直接影响“首次启动”判断的现场风险：

1. API 删除节点和网络成功
2. 不代表 `/opt/unetlab/tmp/<tenant>/<lab>/<node_id>` 一定已经被删掉
3. 如果后续在同一个 `node id` 上重新创建同类设备，旧的覆盖盘状态可能被继续复用

本次实测证据：

1. `ASA1` 的 `node 9` 通过 API 删除后
2. `/opt/unetlab/tmp/0/a2e9276e-5cf6-4ffc-97b5-4ef89ecd800a/9` 目录仍然存在
3. 需要在宿主机上手工 `rm -rf` 后才真正清空

因此，后续凡是要验证“全新设备首次启动行为”“默认账号口令”“首次自动初始化”这类场景时，必须增加这条检查：

1. 节点已删
2. 网络已删
3. `tmp` 覆盖目录也已删
4. 再重新创建设备

## 3. 推荐操作顺序

面向 OneOPS 后续“自动建拓扑”和“真实设备场景测试”，建议采用下面的安全顺序。

### 3.1 新建拓扑

1. 新建 Lab
2. 校验 Lab 能被 `GET /api/labs/<lab>` 正常读取
3. 再开始加节点和网络

示例请求：

```json
POST /api/labs
{
  "author": "Codex",
  "description": "Disposable link validation lab",
  "scripttimeout": 300,
  "version": "1",
  "name": "oneops-link-sandbox-<timestamp>",
  "body": "",
  "path": "/"
}
```

### 3.2 新建节点

本次已验证 `vpcs` 最小可用载荷如下：

```json
POST /api/labs/<lab>/nodes
{
  "template": "vpcs",
  "type": "vpcs",
  "name": "pc1",
  "left": 220,
  "top": 180,
  "icon": "Desktop.png",
  "config": 0,
  "delay": 0
}
```

后续做真实设备测试时，节点创建动作要改为“按厂商模板参数表生成”，不能只靠这份最小 `vpcs` 载荷。

如果当前 EVE 上已经有其它运行实验室，还要额外考虑控制台端口冲突：

1. 不要默认使用 `id=1/2` 这类低位节点做数据面验证
2. 要先确认目标控制台端口没有被现有实验室占用
3. 或者直接把测试节点放到高 ID，再做启动和控制台验证

### 3.3 新建网络

推荐固定使用下面这类载荷：

```json
POST /api/labs/<lab>/networks
{
  "name": "Net_pc1_pc2",
  "left": 358,
  "top": 192,
  "type": "bridge",
  "visibility": 1,
  "icon": "lan.png"
}
```

创建后必须立刻执行一次回读：

```text
GET /api/labs/<lab>/networks
```

只有回读里真的出现了这条网络，才允许继续挂接口。

### 3.4 建立设备间连线

本次验证的挂线方式如下：

```json
PUT /api/labs/<lab>/nodes/<node_id>/interfaces
{
  "0": "<network_id>"
}
```

说明：

1. 键名 `"0"` 表示接口索引
2. 值为网络 ID 字符串
3. 两端节点分别执行一次，即可把两端接口挂到同一个 `bridge`

挂线后必须用接口回读确认：

```text
GET /api/labs/<lab>/nodes/<node_id>/interfaces
```

期望结果：

```json
{
  "ethernet": [
    {
      "name": "eth0",
      "network_id": 1
    }
  ]
}
```

如果要把“拓扑已连好”进一步验证成“设备真的能通信”，建议继续做下面两步：

1. 通过控制台给两端设备配置同网段地址
2. 执行双向 `ping`

本次实测通过的最小示例是：

1. `pc20`: `10.10.20.20/24`
2. `pc21`: `10.10.20.21/24`
3. `pc20 -> pc21` 连续 `3` 次 ping 成功
4. `pc21 -> pc20` 连续 `3` 次 ping 成功

## 4. 推荐清理顺序

### 4.1 单个网络和节点的安全清理

推荐顺序：

1. 停止节点
2. 清空接口挂线
3. 删除网络
4. 删除节点

断开接口示例：

```json
PUT /api/labs/<lab>/nodes/<node_id>/interfaces
{
  "0": ""
}
```

断开后再回读接口，确认 `network_id: 0`。

### 4.2 删除整个拓扑

本次已验证：

1. Lab 内即使还有节点和网络
2. 直接 `DELETE /api/labs/<lab>`
3. 也可以成功删除整个拓扑文件

但补充验证还发现一个重要风险：

1. 如果 Lab 删除前节点仍然处于真实运行态
2. `DELETE /api/labs/<lab>` 删除拓扑文件后
3. 运行进程不一定会自动退出
4. 例如本次两台正在运行的 `vpcs` 在 Lab 删除后仍然残留在全局状态里
5. 需要再次通过控制台显式 `quit` 才能把 `vpcs` 计数清回 `0`

因此，平台级实现不要默认依赖“直接整删”的宽松行为。  
推荐把删除拓扑做成两层：

1. 标准安全删除
   先停机、断线、删网络、删节点、最后删 Lab
2. 强制删除
   仅在标准路径失败时使用，并记录 evidence

## 5. 建议固化为 OneOPS 的拓扑编排约束

后续如果 OneOPS 要做 EVE 驱动的真实设备测试编排，建议把下面这些规则直接写进测试框架和平台服务：

1. 所有写操作只允许落到“临时测试 Lab”或“测试前缀命名 Lab”
2. 创建 `bridge` 网络后必须立即回读校验，禁止信任 `201`
3. 改线前默认执行停机动作
4. 停机成功后必须再做第二证据确认，不能只看节点列表 `status`
5. 删除拓扑时优先走标准安全清理链路，强制删除作为兜底
6. 每次拓扑编排都要记录 evidence：Lab 名、节点 ID、网络 ID、接口回读结果、删除结果

## 6. 这份记录对后续测试设计的意义

这次验证说明，EVE 可以作为 OneOPS 第一阶段真实设备测试的拓扑底座，但不能把它当成“接口返回成功就等于状态真实”的黑盒。

后续围绕网络设备、服务器设备、防火墙设备做场景测试时，应该把测试拆成两层：

1. 拓扑生命周期层
   验证 Lab、节点、网络、连线、清理是否可靠
2. 设备能力层
   在可靠拓扑之上再验证采集、监控推送、配置解析、策略生成、策略查询、拓扑生成和自动化脚本

只有先把第一层打牢，第二层的真实设备结论才可信。
