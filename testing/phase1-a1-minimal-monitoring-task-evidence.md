# A1 最小监控任务补证标准

更新时间：2026-06-28  
适用环境：`192.168.100.20` 上的 EVE-NG Community `6.2.0-4`

## 1. 目的

这份文档用于把 `A1` 里目前普遍空白的“最小监控任务”收口成可执行标准。

当前目标不是一上来覆盖所有监控能力，而是先为每类主线设备拿到第一条真实监控闭环证据：

1. 任务真的被创建
2. 任务真的被分配到正确 Agent
3. Agent 侧真的存在运行中的任务
4. 目标侧真的产生了可解释的监控结果

## 2. 当前共享判断口径

`A1` 的监控任务不允许只凭“页面创建成功”就记为通过。

第一条最小监控任务只有在下面 4 层证据都成立时，才能记为“监控已落证”：

1. 平台计划层：
   编译或应用动作成功，且没有明显的参数、能力或目标解析错误
2. 平台任务层：
   在监控任务管理或统一监控视图中能看到对应任务、目标和状态
3. Agent 运行层：
   Agent 侧能看到任务存在，或能回读到监控状态
4. 目标结果层：
   目标设备或平台时序结果里出现至少一条可解释的数据或状态变化

如果只做到前 2 层，应标记为：

1. 任务已创建，但未证明生效

如果做到前 3 层，但没有目标结果层，应标记为：

1. 任务已下发，但未证明采到有效数据

## 3. 第一轮补证规则

为了尽快补齐 `A1`，第一轮监控补证统一遵循下面 6 条规则：

1. 只做单设备、单能力链、只读任务
2. 不在第一轮混合多种 `scope + kind`
3. 不用“大策略集一次下发多个插件”来证明监控闭环
4. 优先选当前初始化已稳定的管理面，不把“监控问题”和“接入问题”混在一起
5. 第一条任务只要求拿到一个最小有效结果，不追求全指标大盘
6. 每台设备第一轮都要留下“成功证据”或“失败边界”，不能只留口头判断

补充说明：

1. 现有监控设计里，任务能力选择依赖 `collection_type` 与 Agent 能力匹配
2. 当前多组 `scope + kind` 共用单一 `collection_type` 仍是已知设计风险
3. 因此 `A1` 第一轮应避免用“多组混合计划”来证明监控能力

## 4. 最小证据清单

每台设备的第一条监控任务，至少保留下面 8 项证据：

1. 目标设备编码、名称、管理地址
2. 使用的监控任务类型或模板类型
3. 平台侧创建动作结果
4. 平台侧任务状态截图或字段摘要
5. Agent 侧任务存在或运行状态摘要
6. 第一条有效数据或状态结果
7. 若失败，失败点落在哪一层
8. 最终判断是：
   已稳定支持 / 已发现薄弱点 / 已命中明确边界

## 5. 第一轮推荐任务类型

### 5.1 网络设备

第一条任务建议固定为：

1. 管理面存活 + 身份/接口摘要采集

推荐顺序：

1. 如果该设备已有稳定 `SSH` 管理面，优先走只读 `SSH` 模板
2. 如果后续补齐了稳定 `SNMP`，再把 `SNMP` 作为第二轮补证

本轮关注的不是“全接口监控做得多漂亮”，而是：

1. 平台能否把任务正确下发
2. 能否稳定回收一条只读结果
3. 失败时能否明确归因到模板、凭据、能力匹配或设备差异

### 5.2 防火墙设备

第一条任务建议固定为：

1. 管理面存活 + 接口/系统摘要采集

推荐顺序：

1. 优先走当前已打通的只读入口
2. 不把策略解析、NAT、日志监控放进第一条最小任务

这样做的目的是先证明：

1. 防火墙能进入监控任务链
2. 平台和 Agent 的任务映射正确
3. 设备返回的最小系统信息可被稳定接收

### 5.3 服务器设备

第一条任务建议固定为：

1. 主机基础存活 + CPU/内存/负载摘要

推荐顺序：

1. 若设备内本机 Agent 已就绪，优先走本机监控任务
2. 若本机 Agent 未就绪，先用 SSH 只读任务完成第一轮补证

本轮不要求：

1. 大量脚本编排
2. 全进程树监控
3. 日志模板联动

## 7. 推荐执行顺序

当前最顺的补证顺序应是：

1. `H3C VFW`
   当前在线、登录稳定、缺的正是最小监控任务
2. `Ruijie Firewall`
   当前在线，但模板漂移需要在监控回执里如实区分
3. `Huawei AR + H3C VSR + Cisco Gateway`
   适合作为 `A1` 监控补证与 `B1` 联调前置
4. `Huawei USG + Cisco ASAv`
   接入已较成熟，适合在防火墙监控链上补第一条证据
5. `Ubuntu + Debian + Kylin`
   最后补服务器最小监控任务，和后续 `D1/E1` 直接衔接

## 8. 回写规则

后续每补完一台设备的第一条最小监控任务，至少要回写 3 处：

1. [A1 单设备接入基线回执（第一版）](/Users/huangliang/project/OneOPS-ALL/docs/testing/phase1-a1-single-device-baseline-receipt-v1.md)
   把该设备“最小监控任务”从 `未执行` 改成真实状态
2. [A1 逐设备标准回执](/Users/huangliang/project/OneOPS-ALL/docs/testing/phase1-a1-per-device-standard-receipts.md)
   在对应设备条目里补“监控已落证 / 监控薄弱点 / 监控边界”
3. 当前设备自己的标准手册或专项回执
   把真实任务类型、有效结果和失败边界沉淀到设备标准文档

## 9. 当前结论

`A1` 现在已经不缺“怎么写文档”，缺的是按同一口径补齐第一条真实监控证据。

这份文档的作用就是把“第一条监控任务应该怎么验”固定下来，避免后续出现：

1. 有人把“任务创建成功”误记成“监控通过”
2. 有人拿复杂大策略集去掩盖最小闭环没打通
3. 不同设备线用完全不同标准，最后没法横向比较
