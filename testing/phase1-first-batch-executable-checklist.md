## 2. 本批执行原则

### 2.1 只用已就绪主线设备

本批不再把时间花在尚未收敛的镜像上。只用以下已就绪设备推进联合场景：

1. Cisco 网关交换机
2. Huawei AR 路由器
3. H3C VSR 路由器
4. Huawei USG 防火墙
5. Ruijie 防火墙
6. Cisco ASAv 防火墙
7. Ubuntu Server
8. Debian Server
9. Kylin Server

### 2.2 每条场景只打一个核心问题

每条场景都必须有一个明确的核心问题，例如：

1. 这台设备是否真的能稳定接入
2. 两台路由器之间的邻居和路由是否能被稳定采集
3. 防火墙的策略与 NAT 是否能被正确解析和验证
4. 服务器跨越防火墙后的任务与监控是否还稳定

### 2.3 每条场景都要产出边界结论

场景结束后，不能只写“通过”或“失败”。必须写出以下三类结论之一：

1. 已稳定支持
2. 已发现薄弱点
3. 已命中明确边界

### 2.4 登录凭据按设备标准判断

本批场景不再把“所有设备共用同一套账号口令”作为前置要求。

统一要求改为：

1. 每台设备都按其标准手册中的真实可用登录入口执行
2. 账号、口令、首次改密规则、SSH 兼容参数都属于测试证据的一部分
3. 如果设备凭据和旧模板不一致，应先判断“这是已记录例外”还是“这是未记录漂移”
4. 只有“未记录漂移”才应被标成风险点

## 4. 场景 A1：单设备接入基线

### 4.1 目标

1. 为每条主线设备补齐统一接入基线
2. 验证设备登录、管理地址、基础采集、基础监控是否稳定
3. 为后续联合场景建立共同起点

### 4.2 执行对象

1. Cisco 网关交换机
2. Huawei AR
3. H3C VSR
4. Huawei USG
5. Ruijie Firewall
6. Cisco ASAv
7. Ubuntu
8. Debian
9. Kylin

### 4.3 最小检查点

1. 节点按标准模板启动成功
2. 管理地址可达
3. SSH 或厂商可用管理入口可登录
4. 基础身份信息可采集
5. 接口信息可采集
6. 一条最小监控任务可下发并回收结果

### 4.4 要重点观察的薄弱点

1. 不同设备的管理接口命名差异
2. VRF 与非 VRF 管理模型差异
3. 首次启动、重启后、二次登录后的稳定性
4. 接口枚举结果是否与 EVE 挂线预期一致

### 4.5 退出标准

1. 九类设备都拿到统一格式的接入回执
2. 每类设备至少写出一条当前支持边界

## 5. 场景 B1：双路由 OSPF 互联

### 5.1 目标

1. 验证网络设备的接口、邻居、路由采集
2. 验证 OSPF 收敛后平台能否还原关键拓扑关系
3. 优先暴露接口命名、地址规划和路由采集薄弱点

### 5.2 设备组合

1. Huawei AR
2. H3C VSR
3. Cisco 网关交换机仅提供带外管理

### 5.3 地址建议

1. Huawei AR 带外管理：`172.32.2.10/24`
2. H3C VSR 带外管理：`172.32.2.15/24`
3. Huawei AR 业务互联口：`172.32.64.10/24`
4. H3C VSR 业务互联口：`172.32.64.15/24`

### 5.4 最小检查点

1. 两台设备都能从带外管理面稳定登录
2. 两端业务接口都能互通
3. OSPF 邻居建立成功
4. 双方路由表中可见对端学习结果
5. 平台采集结果中能看到接口、邻居、路由三类数据
6. 平台拓扑结果中能看到这条三层关系

### 5.5 要重点观察的薄弱点

1. 华为与 H3C 对 OSPF、VRF、接口命名的抽象差异
2. 平台是否把管理平面接口错误混入业务拓扑
3. 采集结果是否丢接口别名、邻接状态或下一跳信息

### 5.6 退出标准

1. 形成一条完整的双路由互联证据链
2. 能明确写出平台当前拓扑还原的可信范围

## 6. 场景 B2：路由器接网关管理平面复验

### 6.1 目标

1. 验证 Cisco 网关交换机作为共同底座时的稳定性
2. 验证业务设备带外管理接入模型是否一致可复用
3. 优先暴露 EVE 挂线与设备 CLI 接口映射偏差

### 6.2 设备组合

1. Cisco 网关交换机
2. Huawei AR
3. H3C VSR

### 6.3 最小检查点

1. 两台业务设备都从网关带外管理网络获取稳定可达性
2. Huawei AR 的 `MGT` 实例和 H3C 的 `MGT` 实例都可复用
3. SSH 登录在重启后仍然稳定
4. EVE 挂线接口编号与设备侧接口名形成对应表

### 6.4 要重点观察的薄弱点

1. 首口、末口、EVE 口号与 CLI 口号不一致
2. 设备重启后管理入口漂移
3. 管理 VRF 默认路由配置是否被遗漏

### 6.5 退出标准

1. 形成“网关设备 + 双路由器”标准接线回执
2. 形成第一版接口映射对照表

## 7. 场景 C1：Huawei USG 南北向策略链

### 7.1 目标

1. 验证 Huawei USG 的策略对象、策略规则与 NAT 规则
2. 验证平台对策略查询与策略生成的基本可信度
3. 优先暴露安全域模型与平台抽象之间的冲突

### 7.2 设备组合

1. Huawei USG
2. Ubuntu 或 Debian Server 一台
3. Cisco 网关交换机提供带外管理

### 7.3 最小检查点

1. 管理口登录、策略对象创建、策略下发都可执行
2. Server 到目标地址的访问符合 permit/deny 预期
3. 至少一条 `DNAT` 和一条 `SNAT` 规则能落证
4. 平台查询结果与设备当前配置一致

### 7.4 要重点观察的薄弱点

1. `destination-zone` 假设是否与设备真实模型冲突
2. `nat-policy` 命名模板是否稳定
3. 策略对象、地址对象、服务对象之间的引用是否可正确回收

### 7.5 退出标准

1. 形成 Huawei USG 首条联合策略链回执
2. 写出当前策略生成与查询链的支持边界

## 8. 场景 C2：Ruijie 防火墙采集与查询链

### 8.1 目标

1. 验证 Ruijie 防火墙的 `Ge0/0 + main` 管理模型在平台侧是否可稳定适配
2. 验证 SSH 与 Web 双入口下的基础采集与查询链
3. 优先暴露接口、路由与管理面模型例外

### 8.2 设备组合

1. Ruijie Firewall
2. Debian 或 Ubuntu Server 一台
3. Cisco 网关交换机提供带外管理

### 8.3 最小检查点

1. `172.32.2.11/24` 管理地址稳定可达
2. SSH 登录稳定
3. Web 登录与 session 获取稳定
4. 接口信息和路由信息都能按现有只读合同回收
5. 平台能区分“支持查询”和“暂不支持配置下发”的边界

### 8.4 要重点观察的薄弱点

1. `VRF` 不支持时平台如何建模
2. SSH 与 Web 接口结果是否一致
3. 前 8 口与扩展口命名差异是否会破坏接口解析

### 8.5 退出标准

1. 形成 Ruijie 第一条稳定采集与查询链回执
2. 明确写出它在第一阶段的可交付支持范围

## 9. 场景 C3：Cisco ASAv 管理与访问控制链

### 9.1 目标

1. 验证恢复启动后的 Cisco ASAv 是否适合纳入标准防火墙样本
2. 验证管理面、对象、访问控制和重启持久化

### 9.2 设备组合

1. Cisco ASAv
2. Ubuntu 或 Debian Server 一台

### 9.3 最小检查点

1. `Mgmt0/0` 登录稳定
2. 本地管理账号可复用
3. 至少一条访问控制规则能落证
4. 重启后配置仍然存在

### 9.4 要重点观察的薄弱点

1. 恢复启动依赖是否会影响标准化
2. 首次 SSH 改密是否影响自动化接入
3. ASA 密码策略是否阻碍统一实验口令

### 9.5 退出标准

1. 形成 Cisco ASAv 第一条联合场景回执
2. 明确写出恢复启动是否可以被平台标准化吸收

## 10. 场景 D1：服务器穿越防火墙联动

### 10.1 目标

1. 验证服务器资产、任务、脚本能力与防火墙策略链的联动
2. 验证问题定位是否能跨服务器与网络边界设备完成

### 10.2 设备组合

1. Ubuntu、Debian、Kylin 三选一
2. Huawei USG、Ruijie、Cisco ASAv 三选一

### 10.3 最小检查点

1. 服务器侧资产与基础信息可识别
2. 服务器侧脚本任务可执行
3. 防火墙侧策略调整前后，服务器访问行为发生预期变化
4. 平台能把失败现象定位到服务器侧还是防火墙侧

### 10.4 要重点观察的薄弱点

1. 任务成功但网络失败时，平台告警与诊断是否清晰
2. 平台是否能把配置问题和执行问题分层表达

### 10.5 退出标准

1. 形成第一条跨服务器与防火墙的联动证据链
2. 形成跨层问题定位的最小操作手册

## 11. 场景 E1：小规模批量监控与任务下发

### 11.1 目标

1. 在真实设备基础上做第一轮小规模批量验证
2. 先观察并发、失败隔离和接口响应，而不是一开始就追求极限压测

### 11.2 执行对象

1. 已就绪主线设备全量

### 11.3 最小检查点

1. 并发下发多条采集任务
2. 并发下发多条监控任务
3. 混合成功与失败任务
4. 统计任务耗时、失败率和接口响应时间

### 11.4 要重点观察的薄弱点

1. 哪类设备最容易拖慢批量任务
2. 同一设备多任务并发时是否互相影响
3. 失败任务是否会拖垮其他设备
4. 平台接口是否出现明显排队或超时

### 11.5 退出标准

1. 形成第一份真实设备小规模批量任务回执
2. 找出下一轮性能与稳定性优化的优先薄弱点

## 12. 推荐执行顺序

1. 先做 A1，统一所有主线设备接入基线
2. 再做 B1 和 B2，把网络设备与共同管理底座打实
3. 再做 C1 和 C2，把两条差异最大的防火墙线打实
4. 然后做 C3，补齐 Cisco 防火墙样本
5. 再做 D1，验证服务器与防火墙的跨层联动
6. 最后做 E1，形成第一轮小规模并发与稳定性结论

## 13. 每轮执行后的文档产出

每执行完一条场景，至少新增或更新以下内容：

1. 一份场景回执
2. 一份支持边界更新
3. 一份薄弱点记录
4. 如果暴露公共问题，再回写到测试矩阵或环境总览
