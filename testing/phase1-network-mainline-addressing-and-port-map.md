# 第一阶段网络设备主线地址与端口分配

更新时间：2026-06-28  
适用环境：`192.168.100.20` 上的 EVE-NG Community `6.2.0-4`

## 1. 目的

这份文档把第一阶段网络设备主线的设备 ID、管理地址、三层互联、业务网段和端口角色一次性冻结下来，避免后续每次搭场景都重新发明规则。

本文件服务于：

1. 拓扑搭建
2. 配置初始化
3. 采集核对
4. 监控核对
5. 拓扑真值回对

## 2. 统一分配规则

统一规则固定如下：

1. 设备 ID 驱动地址分配
2. 除 `GW` 外，管理地址最后一个八位固定等于设备 ID
3. 管理网固定为 `172.32.2.0/24`
4. 管理网关固定为 `172.32.2.254`
5. 路由平面三层互联从 `172.32.64.0/24` 连续分配到 `172.32.69.0/24`
6. 服务器业务网段固定为 `172.32.101.0/24` 与 `172.32.102.0/24`
7. 服务器业务网关统一使用 `.254`
8. `GW` 保留设备 ID `1`，但作为管理网关交换机，管理侧地址固定例外为 `172.32.2.254/24`

第一波网关归属冻结表：

| --- | --- | --- | --- |

| `172.32.102.0/24` | `172.32.102.254` | `R3` | `R4` 在 V1 不承担该网段网关角色，也不在该网段配置 IP |

补充规则：

3. `ACC1`、`ACC2` 只承接二层接入关系，不在本基线中承担服务器业务网关职责
4. 服务器业务口不直接加入路由 Full Mesh

## 7. 端口角色规则

所有节点统一按角色分口，不靠临时记忆：

1. `MGT`：管理口，只接管理平面
2. `ACC`：接入口，只接服务器或业务设备接入口
3. `L3MESH`：路由互联口，只接 `R1/R2/R3/R4` 之间的三层互联
4. `UPLINK`：`GW` 到 `pnet0` 的外联口

端口使用原则：

1. 一台设备的管理口只承担带外管理
2. 一台业务路由设备至少预留 `1` 个 `MGT` 口、`1` 个 `ACC` 口、`3` 个 `L3MESH` 口
3. 接入交换机必须同时承接 `MGT` 与业务接入口
4. 观测服务器 `OBS` 只需要 `MGT` 口

位序冻结规则：

1. 以 EVE 节点接口展开顺序，自前向后记为 `IF1`、`IF2`、`IF3` ... `IF-last`
2. 第一波所有业务网络设备都按位序冻结接口，不再临时换口
3. 每台业务网络设备的最后一个接口 `IF-last` 固定保留给管理平面
4. 业务网络设备的非管理接口按“先接入、后 Mesh”的顺序分配

这里的业务网络设备固定指：

1. `ACC1`
2. `ACC2`
3. `R1`
4. `R2`
5. `R3`
6. `R4`

## 8. 逐设备接口位序冻结表

厂商接口名可以不同，但位序不允许不同。两位操作人只要按下面的 `IF` 位序接线，就必须得到同一套第一波场景。

| 设备 | `IF1` | `IF2` | `IF3` | `IF4` | `IF5` | `IF-last` 规则 |
| --- | --- | --- | --- | --- | --- | --- |
| `GW` | `pnet0` 上联 | `ACC1` 管理 | `ACC2` 管理 | `R1` 管理 | `R2` 管理 | `GW` 继续顺序分配给 `R3`、`R4`、`S1`、`S2`、`OBS` 管理，不适用“最后一口保留管理”规则 |
| `ACC1` | `S1` 业务接入 | `R1` 接入侧 | `R2` 接入侧 | `GW` 管理上联 | - | `IF4` 为管理口 |
| `ACC2` | `S2` 业务接入 | `R3` 接入侧 | `R4` 接入侧 | `GW` 管理上联 | - | `IF4` 为管理口 |
| `R1` | `ACC1` 接入侧 | `R2` Mesh | `R3` Mesh | `R4` Mesh | `GW` 管理上联 | `IF5` 为管理口 |
| `R2` | `ACC1` 接入侧 | `R1` Mesh | `R3` Mesh | `R4` Mesh | `GW` 管理上联 | `IF5` 为管理口 |
| `R3` | `ACC2` 接入侧 | `R1` Mesh | `R2` Mesh | `R4` Mesh | `GW` 管理上联 | `IF5` 为管理口 |
| `R4` | `ACC2` 接入侧 | `R1` Mesh | `R2` Mesh | `R3` Mesh | `GW` 管理上联 | `IF5` 为管理口 |
| `S1` | `ACC1` 业务口 | `GW` 管理口 | - | - | - | `IF2` 为管理口 |
| `S2` | `ACC2` 业务口 | `GW` 管理口 | - | - | - | `IF2` 为管理口 |
| `OBS` | `GW` 管理口 | - | - | - | - | 只使用单管理口 |

补充说明：

1. 路由设备的 `IF2` 到 `IF4` 固定是三层互联口，不允许和接入口互换
2. 接入交换机的 `IF1` 固定接服务器，`IF2` 与 `IF3` 固定接业务路由设备
3. 如果某厂商模板默认接口名不同，仍然要把“最后一个接口 = 管理口”的位序关系保持不变
4. 若节点模板接口数量不足，不允许现场改拓扑，必须先补模板再执行

## 10. 核对要求

后续所有执行和回执都按这份冻结表核对：

1. 设备 ID 不再临时变更
2. 除 `GW` 外，管理地址最后一段必须等于设备 ID
3. 路由平面六条边必须齐全
4. 服务器必须分别挂在 `ACC1`、`ACC2`
5. `OBS` 必须只走管理平面
6. 每台业务网络设备最后一个接口必须保留给管理平面
7. `R1` 必须承接 `172.32.101.254`，`R3` 必须承接 `172.32.102.254`
8. `R2`、`R4` 在第一波不得承担服务器业务网关角色，也不得在对应服务器业务网段配置 IP
9. 平台拓扑输出必须能回对到本文件定义的子网与对端关系
