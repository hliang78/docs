## 2. 场景定义

本回执固定对应第一波主线采集场景：

1. 业务路由设备：`R1`、`R2`、`R3`、`R4`
2. 接入交换机：`ACC1`、`ACC2`
3. 业务服务器：`S1`、`S2`

本轮不纳入这份采集回执统计的对象：

1. `GW`
   只承担管理网关与 `pnet0` 外联，不参加业务采集判断
2. `OBS`
   只承担 `controller + agent` 执行入口，不作为被采集业务对象写入本回执

本回执只看采集阶段五类最小事实：

1. 身份信息
2. 接口
3. 接入事实
4. 邻居
5. 路由

同时必须显式写出：

1. 历史依据
2. 明确支持边界

## 3. 统一判断口径

采集阶段不允许只凭“任务创建成功”记为通过。

每台设备至少按下面 7 个字段记录：

1. 身份信息结果
2. 接口结果
3. 接入事实结果
4. 邻居结果
5. 路由结果
6. 历史依据
7. 明确支持边界

结果状态统一只用下面三类：

1. 已回收
   只表示本轮执行已经直接回收到该事实，必须能回对到本轮任务结果、字段摘要或命令回读
2. 待回填
   本轮范围已冻结，但当前回执尚未补到直接证据
3. 明确不支持 / 当前不适用
   不是“忘了测”，而是设备角色、平台模型或本轮范围决定它不应被误记为支持

补充规则：

1. `历史依据` 只能承接既有文档、既有回执或历史现场，不得替代本轮 `已回收`
2. 如果某事实只有既有文档，没有本轮直接结果，该事实状态仍必须写 `待回填`
3. `接入事实` 是独立字段，不允许再分散写进“接口”“邻居”或自由备注里
4. 冻结端口场景下，接口字段必须按 `IF1..IF5` 或 `IF-last` 回写，不接受只写“接入口”“Mesh 口”这类泛化描述

## 5. 接入事实回写规则

`接入事实` 必须作为独立槽位记录，供后续 Task 4 直接消费。

统一回写格式固定为：

1. `本端设备 本端IF <-> 对端设备 对端IF | 事实类型 | 本轮状态`

事实类型固定优先级：

1. 物理/逻辑接线事实
2. 二层接入事实
3. 业务挂载事实

本轮最小必填接入事实固定如下：

1. `ACC1 IF1 <-> S1 IF1`
2. `ACC1 IF2 <-> R1 IF1`
3. `ACC1 IF3 <-> R2 IF1`
4. `ACC2 IF1 <-> S2 IF1`
5. `ACC2 IF2 <-> R3 IF1`
6. `ACC2 IF3 <-> R4 IF1`

记录要求：

1. 这 6 条接入事实必须逐条出现，不允许只写“ACC1 已接 S1/R1/R2”
2. 若平台只能回收其中一部分，也要把未回收项逐条标成 `待回填`
3. 若平台回收到的对端口号与冻结表不一致，必须原样记录并在对应设备的“明确支持边界”或后续专项回执中说明

## 8. 薄弱点分析

### 8.1 设备适配问题

1. `ACC1/ACC2` 是否能稳定产出二层接入邻接事实，当前仍依赖真实执行补证
2. `R1/R2/R3/R4` 的管理口与业务口分离较强，若厂商适配把管理口错误吸入主线路由事实，现场很容易误判
3. `S1/S2` 采用主机路由表模型，不一定能直接复用网络设备采集插件

### 8.2 数据建模问题

1. 路由器、交换机、服务器三类对象共用一条主线，但事实模型并不相同
2. 若平台只按“网络设备统一四件套”建模，服务器的邻居事实会天然空缺
3. 若平台把“无三层路由”一律视为异常，会把 `ACC1/ACC2` 的角色边界误判成缺陷

### 8.3 任务执行问题

1. 当前文件先冻结回执骨架，直接执行证据仍待回填
2. 若现场执行时没有把“待回填”“已回收”“历史依据”三者区分开，后续统计会失真
3. 若接入事实仍散落在接口、邻居或备注里，Task 4 很难稳定消费
4. 若采集阶段没有按 `R1/R2/R3/R4/ACC1/ACC2/S1/S2` 八个对象逐一回写，主线范围会再次漂移

### 8.4 明确支持边界

1. `GW` 与 `OBS` 不进入这份采集回执统计
2. `ACC1/ACC2` 在 `V1` 不承担服务器业务网关，也不要求输出主线路由结果
3. `S1/S2` 不要求输出网络设备式邻居协议事实
4. `R2/R4` 在 `V1` 不承担服务器业务网关，不能按 `R1/R3` 的网关职责去要求同类直连路由事实
