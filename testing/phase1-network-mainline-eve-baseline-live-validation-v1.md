## 4. 本轮明确踩到并收敛的现场问题

### 4.1 低位节点 ID 会引出控制台端口冲突

本轮已经再次验证：

1. 如果主线节点使用低位 `node id`
2. 容易与 EVE 上既有常驻实验节点撞到低位控制台端口
3. 现象会表现为控制台连到错误设备，或启动结果与控制台对象不一致

本轮收敛动作：

1. 主线节点统一抬到 `101-110`
2. 后续这条主线不再使用低位节点 ID

### 4.2 华为 AR 的 Mesh 口需要显式改成 OSPF 点到点

本轮已经确认：

1. 仅配置接口地址与 `network` 声明时
2. 华为 AR 在以太网 Mesh 口上会先出现 `2-Way/ExStart`
3. 不利于把路由平面稳定固化成第一波真值

本轮收敛动作：

1. 在 `R1/R2/R3/R4` 的 `GigabitEthernet0/0/1-3` 上显式配置 `ospf network-type p2p`
2. 收敛后四台路由器全部达到 `3/3 Full`

### 4.3 Ubuntu 20.04 这版镜像的 `netplan` 路由语法不能写 `to: default`

本轮已经确认：

1. 该镜像对 `to: default` 写法报错
2. 正确可复用写法应为 `to: 0.0.0.0/0`

本轮收敛动作：

1. `S1/S2/OBS` 全部改用 `to: 0.0.0.0/0`
2. 静态地址和默认路由已稳定生效

### 4.4 Linux 初始化自动化必须先做一次显式 `sudo` 授权握手

本轮已经确认：

1. 不同实例的 Linux 控制台，`sudo` 提示行为并不完全一致
2. 如果直接把配置内容通过 `sudo` 管道下发
3. 某些实例会把配置内容误吞成密码输入，从而造成“看起来像命令失败、实际上是认证流程没走对”

本轮收敛动作：

1. Linux 侧初始化自动化先执行一次 `sudo -S -v`
2. 收到密码提示后显式送入口令
3. 再下发 `netplan / hostname / ssh / chpasswd` 等配置

### 4.5 Ubuntu 服务器 `snmpd` 安装后必须显式重启

本轮已经确认：

1. Ubuntu 20.04 服务器在安装 `snmpd` 时，会先按默认配置启动服务
2. 默认启动形态只监听 `localhost`
3. 如果后续只执行 `systemctl enable`，不会自动吸收新写入的 `snmpd.conf`
4. 结果会表现为“服务看起来是 active，但远端 `snmpwalk` 超时”

本轮收敛动作：

1. 写入统一的 `/etc/snmp/snmpd.conf`
2. 显式执行 `systemctl restart snmpd`
3. 再从管理面做一次远端 `snmpwalk` 复验
