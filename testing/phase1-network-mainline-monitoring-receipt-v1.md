## 2. 适用范围

本回执固定对应第一波主线监控最小通过范围：

1. 业务路由设备：`R1`、`R2`、`R3`、`R4`

本回执可记录但不纳入第一波通过判定的后续扩展对象：

1. 接入交换机：`ACC1`、`ACC2`
2. 业务服务器：`S1`、`S2`

本回执不把下面两个对象记入被监控目标统计：

1. `GW`
   只承担管理网关和 `pnet0` 边界，不作为主线监控通过判断对象
2. `OBS`
   只承担 `controller + agent` 执行入口，不作为被监控业务对象

## 3. 四层证据模型

主线监控不允许只凭“任务创建成功”记为通过。

统一四层证据如下：

1. 平台计划层
   能证明编译、应用或计划生成成功，没有明显参数、能力或目标解析错误
2. 平台任务层
   能在平台任务列表或统一监控视图中看到对应目标、任务和状态
3. Agent 运行层
   能在 Agent 侧看到任务存在、执行中或已回传状态
4. 目标结果层
   目标设备或平台结果视图里出现至少一条可解释结果，例如存活状态、身份摘要、接口摘要、CPU/内存摘要或等价有效值

固定判断表：

| 证据层级 | 判断 |
| --- | --- |
| 只有 `平台计划层 + 平台任务层` | 任务已创建，但未证明生效，不通过 |
| `平台计划层 + 平台任务层 + Agent 运行层` | 任务已下发，但未证明采到有效数据，仍不通过 |
| 四层全部成立，且 `目标结果层` 可解释 | 监控基线通过 |

补充规则：

1. `目标结果层` 不是“有一条日志就算过”，而是结果必须能对应到设备角色和任务类型
2. 结果不可解释、字段空白、只见成功码不见业务值，都不能记为通过
3. 若前三层成立但第四层失败，必须转入薄弱点登记，不允许口头带过

四层证据锚点固定如下：

| 证据层 | 固定锚点字段 |
| --- | --- |
| 平台计划层 | 计划截图/结果编号、时间、关键字段摘要 |
| 平台任务层 | 任务截图/结果编号、时间、关键字段摘要 |
| Agent 运行层 | Agent 截图/结果编号、时间、关键字段摘要 |
| 目标结果层 | 结果截图/结果编号、时间、关键字段摘要 |

## 4. 最小回写字段

每个目标至少回写下面 13 个字段：

1. 目标设备
2. 任务编号
3. 任务类型
4. 平台计划层
5. 平台计划层证据锚点
6. 平台任务层
7. 平台任务层证据锚点
8. Agent 运行层
9. Agent 运行层证据锚点
10. 目标结果层
11. 目标结果层证据锚点
12. 当前判断
13. 备注 / 边界

状态统一只用下面 4 类：

1. 已通过
2. 待补证
3. 已命中薄弱点 / 明确边界
4. 留空

`任务编号` 规则：

1. 同一目标多次重试、多条最小任务并行时，必须逐条保留具体 `任务编号`
2. 四层证据都要能回对到同一 `任务编号`，不接受跨任务拼接证明
3. 若平台任务号与 Agent 任务号不同，允许写成 `平台任务号 / Agent 任务号`

第二波扩展对象规则：

1. `ACC1/ACC2/S1/S2` 若本轮未执行，整行统一保留为 `留空`
2. `待补证` 只用于“本轮已经纳入执行，但证据尚未补齐”的对象
3. 不允许同一轮里把“未执行”和“已执行待补证”混用同一种写法

## 7. 放行结论槽位

本轮监控阶段只有在下面 3 条同时满足时才允许进入拓扑阶段：

1. `R1/R2/R3/R4` 作为第一波主线最小通过范围，已经按目标逐一绑定 `任务编号`
2. `R1/R2/R3/R4` 的每个通过判断都证明了四层证据闭环，不存在把 `平台计划层`、`平台任务层` 或 `Agent 运行层` 单独误记为通过的情况
3. `ACC1/ACC2/S1/S2` 若本轮未补，不影响第一波放行，且统一按 `留空` 处理；若本轮执行了，才改写为真实结果或 `待补证`
