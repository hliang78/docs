# 第一阶段网络设备主线标准场景

更新时间：2026-06-28  
适用环境：`192.168.100.20` 上的 EVE-NG Community `6.2.0-4`

## 1. 目的

这份文档把 OneOps 第一条网络设备主线场景固定下来，作为第一波联合测试的统一入口。

固定口径如下：

1. 主线类型固定为网络设备主线
2. 场景规模固定为 `4+2+1`，这里只统计 `4` 台业务路由设备、`2` 台业务服务器、`1` 台观测服务器，不把 `GW`、`ACC1`、`ACC2` 计入主体验证规模
3. 验证顺序固定为 `采集 -> 监控 -> 拓扑`
4. 目标不是做一次性演示，而是持续暴露 OneOps 薄弱点
5. 结论必须能回写到后续采集、监控、拓扑回执

## 2. 场景定位

这条主线不是单设备接入补证，也不是混合型大而全场景，而是 OneOps 视角下第一条可重复执行的联合网络场景。

它要优先回答 3 个问题：

1. 多台业务路由设备能否稳定进入采集链
2. 采集通过后，最小监控任务能否稳定进入生效链
3. 监控通过后，平台能否把真实三层关系还原成可核对的拓扑

## 3. 固定骨架

第一波标准骨架固定为 `4+2+1`，含义固定为“`4` 台业务路由设备 + `2` 台业务服务器 + `1` 台观测服务器”：

1. `4` 台业务路由设备：`R1`、`R2`、`R3`、`R4`
2. `2` 台业务服务器：`S1`、`S2`
3. `1` 台观测服务器：`OBS`
4. `1` 台管理网关交换机：`GW`
5. `2` 台接入交换机：`ACC1`、`ACC2`

角色约束如下：

1. `GW` 只承担管理网关和 `pnet0` 外联，不参加业务验证
2. `ACC1`、`ACC2` 负责承接服务器和业务设备接入
3. `R1`、`R2`、`R3`、`R4` 承担主线采集、监控、拓扑验证负载
4. `S1`、`S2` 必须接在接入交换机上，不直接挂到路由平面
5. `OBS` 部署 `controller + agent`，并复用管理平面，不单独拉观测网

## 4. 三个平面

### 4.1 管理平面

管理平面统一复用现有 `172.32.2.0/24` 管理网络，网关固定为 `172.32.2.254`。

地址规则说明：

1. 除 `GW` 外，其余节点都按“管理地址最后一个八位等于设备 ID”分配
2. `GW` 保留设备 ID `1`，但作为管理网关交换机，管理侧地址固定例外为 `172.32.2.254/24`

这一层用于：

1. 管理登录
2. OneOps 任务入口
3. `OBS` 到各测试节点的观测与执行链

### 4.2 接入平面

接入平面由 `ACC1` 与 `ACC2` 承担，负责把服务器侧关系和业务设备接入口固定下来。

这一层用于：

1. 服务器接入事实
2. 交换机接口事实
3. 服务器与网络设备之间的接入关系核对

边界约束：

1. `ACC1`、`ACC2` 在本主线中只承担接入关系，不承担服务器业务网关角色
2. `S1` 所在业务网段 `172.32.101.0/24` 的第一波网关职责固定由 `R1` 承担
3. `S2` 所在业务网段 `172.32.102.0/24` 的第一波网关职责固定由 `R3` 承担
4. `R2`、`R4` 在第一波只承担接入侧事实和路由平面参与角色，不承担服务器业务网关职责，也不在对应服务器业务网段配置网关地址

### 4.3 路由平面

路由平面只由 `R1`、`R2`、`R3`、`R4` 组成，固定为全互联三层 Full Mesh。

这一层用于：

1. 接口采集
2. 邻居采集
3. 路由采集
4. 基于真实互联的拓扑还原

## 5. 逻辑连线规则

第一波场景的逻辑关系固定如下：

1. `GW` 上联 `pnet0`
2. `GW` 下联 `ACC1`、`ACC2`、`R1`、`R2`、`R3`、`R4`、`S1`、`S2`、`OBS` 的管理口
3. `ACC1` 下联 `R1`、`R2`、`S1`
4. `ACC2` 下联 `R3`、`R4`、`S2`
5. `R1`、`R2`、`R3`、`R4` 两两之间全部建立三层互联
6. `R1` 固定承接 `S1` 所在业务网段 `172.32.101.0/24` 的默认网关
7. `R3` 固定承接 `S2` 所在业务网段 `172.32.102.0/24` 的默认网关
8. `R2`、`R4` 在接入侧只参与事实观察与拓扑还原，不承担第一波服务器网关
9. `OBS` 只走管理平面访问所有节点

## 6. 验证顺序

执行顺序不允许调换，固定为 `采集 -> 监控 -> 拓扑`。

阶段门槛如下：

1. 采集不过，不进入监控
2. 监控不过，不进入拓扑
3. 拓扑结论必须回对真实地址和真实连线

各阶段关注点：

1. 采集阶段重点看身份、接口、邻居、路由事实
2. 监控阶段重点看任务创建、任务下发、Agent 运行、目标结果四层证据
3. 拓扑阶段重点看平台输出是否和 Full Mesh 路由平面、服务器接入关系一致

## 7. 场景边界

第一波主线场景明确不做下面这些扩展：

1. 不混入防火墙策略链
2. 不混入非主线边界设备
3. 不在第一轮追求大规模压测
4. 不为 `OBS` 单独建设新观测网络
5. 不在一条场景里混合多种不稳定初始化模板

## 9. 当前冻结结论

从 OneOps 视角看，第一阶段第一条网络设备主线标准场景已经冻结为：

1. 网络设备主线
2. `4+2+1` 主体规模
3. 管理平面、接入平面、路由平面三层结构
4. 业务服务器经接入交换机接入
5. `OBS` 复用管理平面
6. 路由平面固定为 `R1/R2/R3/R4` 全互联
7. `R1` 固定承接 `172.32.101.0/24` 网关，`R3` 固定承接 `172.32.102.0/24` 网关
8. 验证顺序固定为 `采集 -> 监控 -> 拓扑`
