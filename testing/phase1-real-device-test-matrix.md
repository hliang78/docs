# 第一阶段真实设备测试矩阵

更新时间：2026-06-28  
适用环境：`192.168.100.20` 上的 EVE-NG Community `6.2.0-4`

## 1. 目的

这份矩阵不是为了列出很多测试名字，而是为了把第一阶段最重要的事固定下来：

围绕真实设备，用有针对性的场景去验证 OneOPS 的采集、监控、解析、策略、拓扑和自动化能力，优先识别薄弱环节，而不是停留在少量冒烟脚本。

测试最高依据统一参照：

1. [Testing Objective Basis](/Users/huangliang/project/OneOPS-ALL/docs/superpowers/testing/platform-testing-baseline/testing-objective-basis.md)
2. [EVE-NG 当前测试环境准备总览](/Users/huangliang/project/OneOPS-ALL/docs/testing/eve-ng-current-environment-preparation-summary.md)

## 2. 第一阶段共同测试口径

所有设备线都用同一套判断口径：

1. 是否真的能稳定接入
2. 是否真的能稳定采集
3. 是否真的能稳定监控
4. 是否真的能支撑策略或配置类能力
5. 是否真的能参与拓扑或链路关系还原
6. 是否真的能执行自动化脚本或任务
7. 如果不能，是否已经命中清晰边界

## 3. 能力链拆分

第一阶段统一按六条能力链组织测试：

1. 设备接入链
   登录、认证、协议、管理地址、基础连通性。
2. 采集链
   接口、邻居、路由、策略、对象、系统信息、资产信息。
3. 监控链
   指标采集、任务下发、告警推送、失败重试、批量任务并发。
4. 解析与生成链
   配置解析、策略生成、策略查询、配置比对。
5. 拓扑链
   邻接关系、三层关系、边界流量路径、跨设备关联。
6. 自动化链
   脚本执行、批量配置、回滚、失败隔离、执行证据。

## 5. 设备类型测试重点

### 5.1 防火墙设备

目标设备：

1. Huawei USG
2. Ruijie Firewall
3. Cisco ASAv

重点能力：

1. 配置采集
2. 对象解析
3. 策略解析
4. 策略生成
5. 策略查询
6. NAT/ACL 命中验证
7. 管理面与数据面联动验证

要重点打的薄弱点：

1. 厂商策略模型差异
2. NAT 规则生成与查询一致性
3. Zone 模型与 OneOPS 统一抽象之间的偏差
4. SSH 与 Web API 双入口并存时的数据一致性
5. 规则规模上升后的查询与渲染性能

### 5.2 网络设备

目标设备：

1. Huawei AR
2. H3C VSR
3. Cisco IOL 网关交换机

重点能力：

1. 接口采集
2. 邻居采集
3. 路由采集
4. 监控任务下发
5. 拓扑生成
6. 自动化脚本执行

要重点打的薄弱点：

1. 多厂商接口命名差异
2. 管理 VRF 与业务路由表并存时的采集路径
3. OSPF 邻居和路由收敛后的拓扑还原
4. 批量设备并发采集时的稳定性
5. 任务下发失败后的重试与隔离

### 5.3 服务器设备

目标设备：

1. Ubuntu
2. Debian
3. Kylin

重点能力：

1. SSH 接入
2. 资产识别
3. 监控任务
4. 自动化脚本执行
5. 与网络边界设备联动验证

要重点打的薄弱点：

1. 基线镜像差异导致的账号、DNS、软件源问题
2. 多次重启后的管理稳定性
3. 大批量脚本任务下发时的执行可靠性
4. 与防火墙、路由器联动时的路径和权限问题

## 6. 第一批推荐场景

### 6.1 场景 A：单设备接入基线

目标：

1. 验证设备能否稳定接入
2. 验证基础采集与监控是否可工作
3. 产出该设备的第一条支持边界

适用：

1. 所有新设备线

### 6.2 场景 B：网络设备双机互联

目标：

1. 验证接口、邻居、路由采集
2. 验证 OSPF 动态路由收敛
3. 验证拓扑关系能否正确还原

适用：

1. Huawei AR + H3C VSR
2. Huawei AR + Cisco
3. H3C VSR + Cisco

### 6.3 场景 C：防火墙南北向策略链

目标：

1. 验证策略对象解析
2. 验证策略生成与下发
3. 验证 NAT 命中与日志证据
4. 验证策略查询结果与设备真实配置是否一致

适用：

1. Huawei USG
2. Ruijie Firewall
3. Cisco ASAv

### 6.4 场景 D：服务器穿越防火墙访问

目标：

1. 验证服务器资产与任务接入
2. 验证防火墙策略链与服务器行为链联动
3. 验证跨设备问题定位能力

适用：

1. Ubuntu 或 Debian 或 Kylin + 任一主线防火墙

### 6.5 场景 E：批量监控与任务下发

目标：

1. 验证大量 agent 或任务并发
2. 验证批量监控目标支持能力
3. 验证失败重试、超时和隔离
4. 验证接口响应与队列堆积情况

适用：

1. 第一批所有已就绪设备

## 7. 每类场景都必须保留的证据

1. 拓扑定义
2. 节点与连线回读结果
3. 初始配置与关键业务配置
4. 连通性结果
5. OneOPS 采集结果
6. OneOPS 监控或任务执行结果
7. 失败报错与边界说明
8. 必要时的宿主机抓包、bridge、tap 证据

## 8. 第一阶段通过标准

第一阶段不是要求所有能力一步到位，而是要求每条设备线都达到以下状态之一：

1. 已稳定支持，并有真实证据
2. 已发现薄弱点，并有可复现路径
3. 已命中明确边界，并有结论文档

只有做到这一步，平台面向客户时，所谓稳定性、鲁棒性、可靠性、可用性、健壮性，才不是口头判断，而是有设备证据支撑的工程结论。

## 9. 建议的执行顺序

1. 先完成每条主线设备的单设备接入基线
2. 再做网络设备双机互联和拓扑还原
3. 再做防火墙策略链与 NAT 场景
4. 再做服务器与网络边界联动
5. 最后集中做批量监控、批量任务与性能压力

## 10. 后续扩展方式

后面新增任何设备家族时，都不要单独发散设计。统一按下面方式并入：

1. 先补设备标准初始化文档
2. 再补该设备在测试矩阵中的主打能力链
3. 再补至少一条高风险薄弱点场景
4. 最后再决定它属于主线还是边界专项
